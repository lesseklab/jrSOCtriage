jrSOCtriage - Running Instructions
====================================

============================================================
PHILOSOPHY
============================================================

The core philosophy of jrSOCtriage is that noise should be
explained, not suppressed.

Traditional SOC workflows reduce alert volume by suppressing
rules or applying conditional filters. While effective at
lowering noise, this creates blind spots by hiding events
instead of understanding them. jrSOCtriage takes a different
approach. Known noisy behavior is documented and explained in
context, and the LLM evaluates each alert against that context.
If the alert matches expected behavior, it is suppressed. If it
falls outside that explanation, it is surfaced automatically.

The LLM operates in a stateless "goldfish" mode. It has no
memory of prior alerts or past decisions. Every alert is
evaluated independently using only the context provided in the
prompt. Because of this, accuracy depends on how well the
context describes what is normal for the environment.

This approach preserves visibility while still reducing noise.
Known patterns are ignored for the right reasons, and unknown
or unexpected variations are not missed simply because a rule
was suppressed. The goal is to minimize blind spots without
sacrificing signal.

This philosophy is supported through configurable context that
shapes LLM reasoning:
  - Sensor context defines how data is collected and its limitations.
  - Network context defines topology and expected traffic patterns.
  - Host notes capture expected behavior for specific systems.
  - Rule notes define why a rule is noisy and under what
    conditions it is normal.
  - General triage guidance provides overall reasoning constraints.

OPERATIONAL EXAMPLE
-------------------

In the test environment used to develop the system, there are
approximately 29,000 raw alerts per day (with observed bursts up
to ~43,000 on storm days). Deduplication reduces this to about
2,400 unique alerts per day reaching the LLM in escalate-everything
mode, or roughly 240 alerts per day in the default
level-6-plus-filters mode.

Two configurations have been exercised in production:

CONFIGURATION A — Send everything to the LLM
  With min_rule_level lowered to 3 (or effectively no filtering),
  all ~2,400 deduplicated alerts per day are evaluated by the LLM.
  Sustained in continuous operation on a single Ollama node
  (gemma4:26b on one RTX 3090) without backlog or saturation. This
  produces roughly 50-80 NOTE events and 0-2 NOTIFY events per day
  at the steady state reached after sustained tuning.

  This is the recommended configuration if you have local LLM
  capacity and want maximum visibility. NOTE events surface
  low-severity but useful signals — CVEs on internal hosts,
  configuration drift, stealth-grade activity that wouldn't trip
  a high-severity rule. With local Ollama, marginal cost per alert
  is effectively zero, so there is no reason to throw this
  visibility away.

  Caveat: alert storms have been tested twice — once on an early
  version (which surfaced fixable issues that motivated the current
  rules.json engine and dedup_silence_seconds=240) and once post-fix
  (which the system absorbed without backlog). Loads materially
  above the tested envelope have not been formally probed. If you
  experience an alert storm beyond what your rules.json tuning
  absorbs, the queue may grow faster than the LLM can drain it. See
  SCALE EXPECTATIONS below.

CONFIGURATION B — Filter to medium-and-above
  With min_rule_level set to 6 and rules.json populated with site-
  specific suppressions for known-noisy rules, the number of LLM
  evaluations drops to about 240 per day. This still surfaces the
  same 0-2 NOTIFY events per day as Configuration A (the LLM
  applies the same severity logic regardless of mode) but produces
  fewer NOTE events (around 10-15), because most of the low-severity
  signals never reach the LLM.

  This configuration is appropriate if you are using a cloud LLM
  with per-token cost, if your local LLM is shared with other
  workloads, or if you simply do not need the additional NOTE
  visibility.

WHICH TO USE

Configuration A trades local LLM capacity for visibility.
Configuration B trades visibility for lower resource use.
Both produce the same NOTIFY signal — the difference is in how
much low-severity context you keep around. Pick based on what
your local LLM can sustain and how much NOTE visibility you want.


DEDUPLICATION VERSUS WAZUH FREQUENCY THRESHOLDS
------------------------------------------------

If you already run Wazuh frequency-threshold rules — rules that
fire only after a signature occurs N times in a time window — you
will notice that jrSOCtriage's deduplication appears to collapse
fewer alerts than the figures in this document suggest. That is
expected, and it does not mean dedup is underperforming. A
frequency threshold is a form of pre-deduplication: it collapses
repeats upstream, at the Wazuh manager, before the alert ever
reaches jrSOCtriage. So jrSOCtriage's dedup sees fewer duplicates
to remove, and its measured collapse ratio looks lower. The total
reduction is unchanged — the same repeats are collapsed either
way; the work simply moved upstream. (The figures in this document
are measured on a configuration with no Wazuh-side frequency
thresholds, so the collapse ratio here is higher than you will see
if you filter upstream.)

The two mechanisms are not equivalent, and the difference matters:

  - Deduplication fires the first occurrence immediately. The first
    time a signature appears, it goes straight to triage with no
    delay; dedup suppresses only the repeats that follow within the
    window. A frequency threshold does the opposite: it stays
    silent until the Nth occurrence, so you are blind to the
    condition until it crosses the count. A threshold of "fire
    after 5" means the first four occurrences are invisible. For a
    genuine low-and-slow signal that occurs a few times and stops,
    a frequency threshold can miss it entirely; deduplication
    cannot, because the first occurrence always reaches triage.

  - Deduplication preserves the frequency information. The
    suppressed repeats are counted and aggregated, so the count is
    retained and available to the LLM and the operator — the alert
    that reaches triage carries "this fired 47 times in the window."
    A frequency threshold discards the first N-1 occurrences
    outright; that information is gone.

  - Deduplication runs on every alert. Because it has no
    first-detection cost and preserves the count, there is no
    downside to applying it universally — a high-value
    must-see-immediately alert and a noisy low-value alert get the
    same safe treatment. A frequency threshold, because it delays
    first detection and discards occurrences, can only be applied
    selectively, to known-noisy rules where missing the first few
    is acceptable. That means per-rule classification and tuning;
    deduplication needs none.

The practical consequence: deduplication and frequency thresholds
can coexist without conflict, but once jrSOCtriage's dedup is in
place the upstream frequency thresholds are largely redundant —
they are not reducing volume that dedup would not already reduce,
and they are costing you earlier alerting and retained frequency
data. If you have frequency-threshold rules at the Wazuh manager,
you can relax or remove them and let jrSOCtriage's dedup do the
same job, gaining earlier first-alerting and retained occurrence
counts at the same alert volume.


jrSOCtriage is designed to be shaped by expanding and refining
the context provided to the LLM rather than relying on heavy
filtering. Rule notes are the most powerful mechanism for this.
They allow you to describe why a rule is noisy and what
conditions make it expected behavior. In effect, they act as
guidance such as "this is normal because X; if not X,
investigate." Host notes allow behavior to be defined at the
system level. Roles let you define that behavior once for a KIND
of host and attach it to many hosts: write a "sales_laptop" role
with its expected traffic and behavior a single time, tag 200
sales laptops with it, and every one of them carries that context
into triage — maintained in one place instead of copied into 200
host notes. A host can hold several roles (e.g. a box that is both
a hypervisor and a workstation), and the LLM sees the context of
all of them. Host notes still cover whatever is unique to one
specific host; roles cover what is shared across a class of hosts.
Per-host rule notes allow the same rule to be
interpreted differently depending on which host it fired on —
for example, rule 504 (Wazuh agent disconnected) is routine on
a laptop but NOTIFY-worthy on a domain controller, and per-host
notes let both interpretations coexist without contradicting
each other. Additional context can be configured globally
through sensor, network, and general triage sections in
config.json.

These inputs act as structured reasoning hints rather than hard
filters. They guide the LLM toward correct conclusions without
removing visibility into the underlying events.


============================================================
SCOPE AND DEPLOYMENT CONSIDERATIONS
============================================================

jrSOCtriage is designed for home labs, small labs, personal
networks, and businesses up to medium-sized. In testing, the
pipeline handled substantially higher volumes than that —
how much higher hasn't been mapped, since the goal was to
establish what works rather than find the breaking point. See
the WORKED EXAMPLE further down in this document for the
specific load-test numbers.

The operator is expected to treat the pipeline as a guided
triage assistant rather than a full SIEM replacement. The
project is not positioned as enterprise-grade — but the actual
headroom in the current code is larger than that positioning
suggests, and this section is honest about both.

If you deploy it somewhere it was not designed for and it
struggles, that is a signal that you have outgrown it, not that
it is broken. Conversely, if you deploy it within scope and
size your LLM tier appropriately, it has more capacity than the
"home lab" framing implies.

The intended scope may expand over time if the project is well
received. For now, scope is deliberately limited to keep the
design simple, auditable, and approachable for a small team
running a single deployment.

ENTERPRISE / MULTI-DOMAIN DEPLOYMENT NOTE
-----------------------------------------

v1.0 does not provide native multi-tenant operation or
horizontally-scaled jrSOCtriage clustering. A workable enterprise
deployment pattern is to run separate jrSOCtriage instances per
security domain (for example: DMZ, workstations, servers, cloud,
a subsidiary, a lab), each with its own config, rules, host notes,
database, credentials, and LLM endpoint policy, each feeding
enriched records into its own Graylog node or stream. This
preserves isolation between domains and avoids pretending v1.0 has
tenant-aware internals.

When multiple instances ship into one shared Graylog, set
deployment.org and deployment.security_domain in each instance's
config (top of the Config tab). They stamp every record with
jrsoc_org and jrsoc_security_domain so each org or domain can filter
and search only its own alerts on the shared instance. See
graylog_searches.txt for the search syntax.

This is an architectural workaround, not a supported v1.0
enterprise mode. The tooling that turns this manual per-instance
pattern into a managed one is a deliberate path on the roadmap.
Roles (roles.json with shared context, so context is written once
per role instead of once per host) shipped in v1.0. The remaining
steps: the config-to-database migration plus periodic config sync
between instances (v1.3 — hosts, rules, and the rest move from
per-instance JSON files into a local database with a tenant_id
column on every table from day one; a candidate follow-on then
syncs the shareable config tables — hosts, roles, rules — between
instances on a schedule, hourly being plenty since config changes
are rare, so a host that moves between domains or a rule common to
several is edited once on the source and propagates rather than
being hand-copied; this reuses the same periodic-copy pattern as the
RAM-disk rsync, with no live shared access and therefore no
concurrent-write locking to manage); and true multi-tenancy (v2.0 —
one cluster, one interface, tenant context per session, where the
per-instance isolation collapses into a single tenant-aware
deployment). Each step builds on the last, and multi-tenancy is the
v2.0 through-line. See roadmap.md. The per-instance pattern works
today because each instance is exactly the single-domain
deployment v1.0 is built for — a team of trusted administrators
sharing one instance; you are running N independent deployments,
not one tenant-aware one.

RUNNING jrSOCtriage ON A WAZUH-MONITORED HOST
---------------------------------------------

In the normal topology, jrSOCtriage runs on a host that the same
Wazuh deployment also monitors — the host feeding the pipeline is
itself a monitored endpoint. That host's own agent reports its
system activity to the Wazuh manager like any other endpoint. This
self-monitoring is the root of the issue: the host is in-band with
the monitoring it serves, not isolated from it, so under heavy
sustained alert volume the high rate of journal and file activity
that jrSOCtriage's own operation generates is itself observed by
the local agent, and at high enough volume the agent's outbound
event buffer begins to fill. When that buffer fills, the agent
emits its own buffer-status events (Wazuh rule IDs in the 202-205
range, the agent queue / "events per second" status rules). Those
status events are themselves more activity for the agent to
report, which can push the buffer further, and the result is a
positive feedback loop: the host alerting about its own alerting,
climbing until it dominates the stream.

This only matters on a host running at high sustained volume that
Wazuh also monitors. A normal endpoint never approaches the rate
where its agent buffer saturates. It is specifically the
self-monitored, high-throughput host (the jrSOCtriage host
under heavy load) that can drive its own agent into this loop.

THE FIX: DISABLE THE LOCAL AGENT'S client_buffer

The clean fix is to disable the outbound event buffer on the
local agent — the agent on the jrSOCtriage host itself, not the
manager. Edit that host's /var/ossec/etc/ossec.conf and set the
client_buffer to disabled:

  <client_buffer>
    <disabled>yes</disabled>
  </client_buffer>

Then restart the agent:

  sudo systemctl restart wazuh-agent

With the buffer disabled there is no throttle to saturate, so the
buffer-status (202-205) events are never generated in the first
place. The feedback loop has no source. This is preferable to
suppressing those events after the fact (see the alternative
below) for three reasons:

  - It fixes the problem at its source. The buffer throttle firing
    is what generates the 202-205 events; with no throttle, the
    events never exist. Manager-side suppression instead lets them
    fire and filters them out downstream, which means the host is
    still generating the noise — you are just hiding it.

  - It is faster and simpler. One setting in one file on the one
    host that needs it, versus writing rule overrides on the
    manager, making sure the manager is configured to load them,
    scoping them to the right agent, and re-validating.

  - It has no rule-maintenance surface. A suppression rule is one
    more piece of manager ruleset config to maintain, keep correct
    across Wazuh upgrades, and scope carefully so it does not
    over-match. The agent-side toggle has nothing to maintain —
    there is no rule to drift or break because there is no rule.

This change is scoped to the one host that needs it. It affects
only that host's agent. The client_buffer stays enabled on every
other monitored host, where it does its normal job of pacing the
agent's output and protecting the manager from a flood. You are
not turning off a safety feature fleet-wide; you are turning it
off on the single high-volume self-monitoring host where the
buffer's pacing is being tripped by the host watching its own
work rather than by a real outbound-rate problem.

One thing to confirm after making the change: with the buffer
disabled, that host's agent ships its events to the manager
without the buffer's pacing, so the manager must be able to take
that host's full event rate. On the tested deployment the manager
absorbed it cleanly with no drops. If your manager is already near
its ingest capacity, watch manager-side health after the change.

ALTERNATIVE: MANAGER-SIDE SUPPRESSION (for ruleset operators)

If you already run custom rules on the manager and prefer to
handle this there, you can instead suppress the 202-205 rules for
the jrSOCtriage host on the manager side. This requires that your
manager is configured to load custom rule files (a <ruleset>
section in the manager's ossec.conf, present in a stock Wazuh
install) and that you scope the suppression to the single
self-monitored agent — you do not want to suppress buffer-status
rules for all agents, since on a normal endpoint those events are
a genuine signal that something is wrong. This route is offered
for operators who are already managing a custom ruleset; for
everyone else the agent-side client_buffer disable above is
simpler, more robust, and works regardless of how the manager's
rules are arranged.

VERIFYING THE FIX

Run the host at the sustained volume that previously triggered the
loop and watch the alert rate (per-minute) hold steady rather than
climbing. The feedback loop, when present, compounds — the rate
roughly doubles and keeps rising rather than drifting. A rate that
holds flat through and past the point where it previously began to
climb confirms the loop is gone. If you see the rate climb, read
the logs before concluding the fix failed: a genuine alert storm
(a real burst of downstream alert volume) produces the same
rising rate as the feedback loop, and only the log content tells
them apart — a feedback loop shows the 202-205 buffer-status
events stacking up from the local host, while a real storm shows
actual alert content. Diagnose by what is in the increase, not by
the rate alone.

WHAT'S MEASURED
---------------

The development environment generates approximately 29,000 raw
Wazuh alerts per day (with observed bursts up to ~43,000 on storm
days). After deduplication (currently at
dedup_silence_seconds=240; widened from 30s earlier in development),
about 2,400 alerts per day reach the LLM stage when the pipeline
is configured to escalate everything (Configuration A). In
level-6-plus-filters mode with rules.json populated, this drops
to roughly 240 alerts per day. The escalate-everything load was
sustained in continuous operation on
a single Ollama node (gemma4:26b on one RTX 3090) without backlog,
queue saturation, or LLM endpoint timeouts.

That demonstrates the pipeline can sustain 2,400 LLM-triaged
alerts per day on a single local Ollama worker. It does not
establish an upper bound — higher loads were not tested on
local-LLM-only deployments.

Single-endpoint reliability under sustained load varies by local
engine.

Ollama (gemma4:26b, Q4_K_M default): across approximately one month
of continuous "escalate everything" operation since April 14, 2026,
the pipeline produced 1-2 VERDICT: FAILED records per day (the
synthetic verdict emitted when all configured LLM endpoints fail
for a single alert). Against ~2,400 LLM-triaged alerts/day that is
a 0.04-0.08% per-call failure rate. Ollama's scheduler provides
idle recovery windows between calls that keep GPU memory and
compute stress moderate.

llama.cpp (same model, same quantization): same verdict quality
and ~25-30% faster per-call inference (mean ~11s vs ~15s), but
tighter scheduling — no idle recovery windows between calls. This
produces noticeably higher sustained GPU memory bandwidth and
thermal load than Ollama under continuous operation. On this
project's reference hardware (consumer-grade RTX 3090), FAILED-to-
converge rates have been observed from approximately 1 to 10 per
day depending on ambient temperature and active operational load.
Overnight at cool ambient: near zero. Daytime at warmer ambient
under active use: substantially higher. Mechanism not fully
isolated — potentially thermal margin, load-correlated GPU
resource contention, model convergence variability, or some
combination. Other operators with newer GPUs, better cooling,
lower ambient temperatures, or different workload patterns should
expect different rates.

Failures on either engine concentrate on genuinely-hard alerts
where the local model's reasoning budget is insufficient to reach
a confident verdict within the request timeout — by definition the
alerts that most benefit from a smarter, faster cloud model.
Enabling a cloud fallback endpoint (Claude Haiku, GPT-4.1 mini, or
GPT-4o mini in testing — GPT-4o mini being the cheapest at
comparable triage quality, see the cost section) eliminates them
operationally: the cloud model handles
these complex alerts in 4-6 seconds with a clean verdict. The
architecture isn't local-plus-redundancy; it's specialist
escalation — local handles the bulk efficiently, cloud handles the
genuinely-hard cases at pennies per call. Operators running with
cloud fallback configured should expect very few FAILED verdicts
surfaced operationally; operators running single-endpoint should
expect the per-day rate visible in Graylog (gl2_llm_verdict:FAILED)
and alert if it climbs substantially above their established
baseline.

Specialist escalation in production: across continuous operation
since April 14, 2026, the cloud fallback has handled the
genuinely-hard tail consistently. On llama.cpp with Bartowski's
Q4_K_M quantization, the escalation rate sustained approximately
12 FAILED/day out of ~2,400 alerts triaged daily (~0.5% per-call
rate) — local handled the bulk, cloud handled the cases the local
model couldn't converge on within the timeout budget. On Q4_K_L
from the same source (currently under observation) the rate is
substantially lower. Each escalation follows the same path: local
Ollama or llama.cpp call hits the request timeout, dispatcher
escalates to the configured cloud endpoint, anonymization runs,
the cloud model produces a clean verdict in 3-6 seconds, and the
verdict ships to Graylog correctly. That's the
specialist-escalation pattern doing its job: local handles the
bulk efficiently, cloud handles the genuinely-hard tail at
pennies per call. Sustained-pressure behavior was exercised in the pre-1.0
load-harness campaigns (raw alert volume tested well into the
millions per day — see the WORKED EXAMPLE later in this document).
The escalation code path has been exercised hundreds of times in
production and is operationally validated.

WHAT'S ESTIMATED — CLOUD-LLM CEILING
-------------------------------------

Local-LLM inference is the slowest stage in a local-primary
deployment, dominating end-to-end latency at 10-20 seconds per
call. With cloud LLM endpoints (Haiku-tier, ~4-6 sec/call as
measured against jrSOCtriage prompts of 13-21k chars) and
corresponding API rate quotas in the hundreds of RPM, the LLM
stage stops being the bottleneck and the pipeline can sustain
substantially more throughput.

The throughput estimates below come from reading the code and
reasoning about the serial path per alert (enrichment fetches,
DB writes, LLM call, GELF ship). The throughput numbers
themselves have NOT been verified by load testing at the specific
worker counts shown, though two end-to-end synthetic test runs
(May 2026) provide measured anchor points — see the worked
example at the end of this section. The per-call cost basis used
in this section HAS been re-confirmed by those runs. Treat the
table as ballpark guidance, not measured guarantees. A v1.3
synthetic load harness will replace these estimates with measured
numbers.

These are LLM-calls-per-minute, NOT alerts-per-minute. Most
alerts are dropped by dedup or rule-level/rules.json filtering
before the LLM stage; only the alerts that actually reach the
LLM count toward these rates. The figures below are RAW sustained
per-worker capacity against GPT-4.1-mini (~2-3 second average
response on this project's prompts). They do NOT include headroom
— the worker-sizing math elsewhere in this document applies a
40% headroom multiplier (×1.40) on top of these raw figures, and
you should always provision with that headroom rather than sizing
to the raw rate (a worker running flat-out has no margin for
latency wobble or burst):

  - 1 worker:     ~20 LLM calls/min raw sustained
  - 10 workers:   ~200 LLM calls/min raw sustained
  - 50 workers:   ~1000 LLM calls/min raw sustained

The 1- and 10-worker rows are backed by measurement (the May
2026 load test ran 24 workers and sustained ~154 LLM verdicts/min
against GPT-4.1-mini). The 50-worker row is a linear
extrapolation from that, not directly measured. Beyond 50 workers
is untested — this project has not load-tested higher worker
counts, so no numbers are quoted for them. At some point
per-worker enrichment overhead and upstream API stress would
become the binding constraint, but where that knee is hasn't been
measured.

These rates assume GPT-4.1-mini's speed. A slower endpoint
(Claude Haiku at ~4-6 sec/call, roughly 2× GPT-4.1-mini's time)
lowers the per-worker rate proportionally — see the
conservative-floor table in the SCALING CEILING section for
slow-endpoint numbers. Size to your actual endpoint's measured
response time, and always apply the 40% headroom from the
sizing math.

For most home-lab and small-team deployments, these ceilings
are not the binding constraint. Cost is — see "DON'T OVER-
PROVISION" elsewhere in this document for the cost/quality
tradeoff. A small deployment running cloud-primary with normal
level-6-plus-filters mode (~240 LLM calls/day at this lab's
volume) will spend roughly $200/year on GPT-4.1-mini. A
deployment running cloud-primary in "escalate everything" mode
at this lab's volume (~2,400 LLM calls/day, no rule filtering)
spends roughly $2,000/year on GPT-4.1-mini — about 10× more,
because the LLM sees 10× more alerts. These figures are
derived from a measured per-call rate of $0.0023 against
this project's full-fat prompts (13-21k characters with full
enrichment) billed on GPT-4.1-mini. GPT-4o mini is the cheaper option at comparable triage quality
AND the same per-call speed: measured ~$0.0007/call at
the same prompt sizes (its token pricing, $0.15/1M in and
$0.60/1M out, is roughly 0.375× GPT-4.1-mini's — well under
half), which brings the same volumes to roughly $60/year
(filtered) and $600/year (escalate-everything). The same-speed
claim is now measured head-to-head in clean current samples: on
this project's full-fat prompts GPT-4.1-mini runs ~2.1s median
(15 calls) and GPT-4o mini ~2.0-2.2s median (two 19-call samples)
— effectively identical. Use the median, not the mean, at these
sample sizes — a single outlier drags the mean (one 4o-mini
sample caught an 11.6s outlier while its median held at ~2.2s).
(An earlier ~3.1s 4.1-mini figure was from an OpenAI service-
degradation period — superseded by the ~2.1s clean median.) There
is no latency penalty for the cheaper model. Cloud latency varies
with provider conditions, so measure your own. Since GPT-4o
mini matches GPT-4.1-mini on both speed and triage quality at
under half the cost, it is the sensible default cloud endpoint
for essentially all deployments. Other validated cloud
endpoints (Claude Haiku 4.5, Gemini 3 Flash Preview) have
different per-token pricing and have not been measured at
the same prompt sizes — verify your specific provider's
pricing before sizing. Premium tiers (Opus, GPT-5, Gemini
3 Pro) multiply both figures by ~10-30×.

WORKED EXAMPLE — SUSTAINED CAPACITY AND COST TEST
--------------------------------------------------

The capacity and cost figures in this document are anchored on an
end-to-end synthetic load test: the full pipeline run for sustained
hours at rates far above any realistic deployment.

How the load was generated (this matters for interpreting the
numbers): the harness does not invent arbitrary alerts. It
multiplies real alerts captured from the development environment
across synthetic host identities (loac_host001, loac_host002, and
so on), preserving the underlying alert and rule shape — real rule
IDs, real field structure, real enrichment-triggering content. A
configurable umbrella window offsets the timestamps across the
synthetic hosts so the load arrives spread over time rather than as
one impossible same-timestamp burst. The result exercises the whole
pipeline — dedup, enrichment, Graylog round-trips, the LLM call, GELF
shipping — the way real traffic would, at a multiplied volume. It is
not random or malformed JSON, and it is not a microbenchmark of one
stage in isolation.

  Capacity test: level-6-plus-filters mode, full pipeline including
  LLM triage, 24 workers. Run across roughly three hours of sustained
  load at this level. The input rate is not a dialed-in target — it is
  the project's real, organic alert volume multiplied across synthetic
  hosts, so each hour's total reflects however much real traffic the
  live environment happened to produce that hour. The three test hours
  came out at approximately 26M, 28M, and 29.8M alerts/day. The
  pipeline processed all of it cleanly at every one of those rates —
  zero ingest loss, zero abandoned work, throughput and resource use
  stable throughout.
    Highest hour (~29.8M/day): ~11,100 LLM calls completed
    API cost, that hour: ~$8 on gpt-4o-mini
    Per-call cost: ~$8 / ~11,100 ≈ $0.0007 — consistent with the
    measured gpt-4o-mini rate cited above

An important point about what these numbers mean: the pipeline did
not "top out" at 29.8M/day. It cleanly handled everything it was
handed, and 29.8M/day is simply the most load the multiplied organic
volume ever produced in a test hour — not a limit the pipeline
reached. The true ceiling is higher; the test just never fed it
more. Even the smallest of the three hours (~26M/day) is roughly 29×
the ~900k/day the pipeline was originally designed for, so the whole
test band sits far above any realistic deployment.

The peak ~29.8M/day rate is on the order of 1,000× this project's
real production volume, on a single modest server. The escalation
threshold ran at the normal level 6 — a standard, buyer-typical
configuration, NOT an escalate-everything stress setting. What made
the test harder than a real deployment was the absence of any
Wazuh-side filtering upstream (no suppression rules, no fire-after-N
frequency rules), so the pipeline ingested the full unfiltered Wazuh
output that a real SOC trims heavily at the manager before triage.

A note on the cost figure and why it is a worst case: on this run,
roughly half of the alerts above level 6 (before dedup) were the
Wazuh agent-buffer self-monitoring rules (202, the 90%-full warning,
and 203, the 100%-full alert) — a feedback loop where the
high-volume test host reports on its own buffer pressure. Because
that feedback is highly repetitive, it made dedup MORE effective,
not less: dedup collapsed at ~96% on this run versus the ~90-92%
typical of normal traffic. So most of that feedback died at the
dedup stage before ever becoming an LLM call — the ~50% figure is a
share of input VOLUME, not of cost. A normal deployment without that
feedback loop has less input to begin with, but also less of the
dedup lift the repetition provided, so it lands somewhat cheaper
than $8/hour — not proportionally to the 50%. The takeaway: ~$8/hour
is a defensible worst-case ceiling for LLM-triaging a ~29.8M/day
firehose on the low-cost cloud model, and a real deployment costs
less.

The two escalation modes have different ceilings, both established
on clean runs. In level-6 mode — the normal, buyer-typical setting —
the pipeline is proven to the ~29.8M/day rate above. In
escalate-everything mode, where every alert above the floor is sent
to the LLM regardless of rules, the pipeline has run clean to a
~6.8–7.2M/day rate. (Escalate-everything's ceiling is lower because
it sends far more traffic to the LLM per unit of input, so it
saturates the triage stage at a lower ingest rate — which is exactly
why level-6-plus-filters is the cost-sensible default.)

For historical comparison, two earlier and smaller synthetic runs
(May 2026) were billed against GPT-4.1-mini: an escalate-everything
run at a ~2.7M/day pace (9,529 LLM calls in the hour, ~$18) and a
level-6 run at a ~6.3M/day pace (770 LLM calls, ~$2). Those remain
real measured figures on the pricier GPT-4.1-mini model. The newer
runs above — billed on the cheaper gpt-4o-mini — supersede them as
the current ceilings for each mode.

Separately, the database was stress-tested on its own — the same
pipeline with the LLM stage disabled — to about ~30.9M alerts/day
on a single consumer NVMe with on-disk SQLite, zero database-lock
contention, and zero write retries. This confirms the database is
nowhere near a bottleneck at any realistic scale and needs no
special storage; on-disk is the recommended default. (The small
difference between the ~29.8M with-LLM figure and this ~30.9M
LLM-off figure is just where the two separate runs happened to land,
not an architectural gap.)

Notes on worker count and cost:
  - Both the capacity test and the historical runs used 24 workers,
    this project's standard load-test configuration.
  - Worker count does NOT drive cost. Cost is determined entirely
    by how many alerts reach the LLM. Workers are throughput
    configuration; the same N LLM calls cost the same whether
    served by 1 worker or 100.
  - For comparison: production at this lab runs 1 worker at
    ~30,000 raw alerts/day and ~2,400 LLM calls/day, well within a
    single worker's sustained throughput.

These measured numbers do NOT validate the 1-worker / 10-worker /
50-worker throughput estimates in the table above — those remain
ballpark guidance pending a future synthetic load-harness campaign
across many worker counts. What they DO validate is the whole-
pipeline capacity and the per-call cost basis the section relies on.

WHERE THE ARCHITECTURE WOULD ACTUALLY BREAK
--------------------------------------------

Beyond the cloud-LLM estimates above, the substrate begins to
matter. In rough order of which limit hits first:

  1. AlertJSON ingest serialization — single ingester reads the
     Wazuh file sequentially. No fan-out at this layer, by design.
     At very high inbound rates this becomes the binding constraint
     before any of the limits below.

  2. Per-worker enrichment overhead — Python GIL and per-alert
     enrichment cost (Graylog fetch, Zeek lookup, ntopng query,
     AbuseIPDB lookup) limit how much each worker can do. Adding
     workers helps until upstream APIs become the bottleneck.

  3. Upstream API stress — Graylog and ntopng fetches scale
     linearly with worker count. Both are configurable but have
     their own limits. For self-hosted Graylog, that's a tuning
     question on your side.

  4. Graylog round-trip throughput. Each alert that reaches
     Stream 2 generates Graylog traffic in BOTH directions:

     - Read (enrichment): graylog_fetch queries Graylog for the
       context window around the alert, pulling up to ~100
       messages per query. At even 1-2 KB per message that's
       ~100-200 KB inbound to jrSOCtriage per alert.
     - Write (record): the enriched Stream 2 record ships back to
       Graylog via GELF, typically 10-30 KB.

     The read dominates — the context-window query pulls several
     times more data than the enriched record sends back. Any
     network/throughput budget has to account for the round trip,
     not just the GELF write.

     The binding constraint depends on whether Graylog and
     jrSOCtriage are colocated:

     - Colocated on the same host (a common SMB pattern, and this
       project's own deployment): both the enrichment read and
       the GELF write go through the loopback interface, not the
       physical NIC. Loopback isn't bound by line rate; it's
       limited by CPU/memory bandwidth (tens of GB/sec on modern
       hardware) — orders of magnitude above what the round trip
       needs. The binding constraint here is Graylog's own
       query-serving and ingestion throughput (indexer CPU,
       OpenSearch read/write capacity), not the network.

     - Separate hosts on a 1 Gbps link: the round-trip data volume
       (read-dominated) is what fills the pipe. Accounting for
       both directions, a single Graylog node on 1 Gbps supports
       roughly a few hundred thousand to ~1 million enriched
       alerts/day before the link saturates — which at ~90% dedup
       corresponds to single-digit-millions of RAW alerts/day.
       (This is lower than a write-only estimate would suggest,
       because the enrichment read is the larger flow.)

     - Separate hosts on a 10 Gbps server-plane network: multiply
       by ~10.

     Multi-node Graylog clusters with load-balanced query and
     ingest scale this further regardless of network speed. At
     very high volumes on separate hosts, this round trip is
     jrSOCtriage's most likely real-world bottleneck, ahead of
     any limit in jrSOCtriage itself. Colocating Graylog with
     jrSOCtriage sidesteps the network dimension entirely.

     These round-trip numbers are order-of-magnitude estimates,
     not firm limits. They have NOT been measured by load testing.
     The single biggest lever on the read-side traffic is the
     context-window setting — how far on either side of the alert
     graylog_fetch queries. The default is 30 seconds each side;
     this project's own deployment uses 1 minute each side for
     richer context, and an operator can set whatever window they
     want. A wider window pulls more messages per query and
     inflates the per-alert read volume proportionally — set it to
     something large like 15 minutes each side and the per-query
     payload climbs well past the couple-of-kilobytes range the
     rough estimate assumes. The numbers also depend on your
     actual Graylog message sizes, your enrichment payload size,
     and your specific hardware. Treat them as a starting point
     and measure your own deployment before committing to a sizing
     decision — the real ceiling on your setup could be
     meaningfully higher or lower, and your context-window choice
     is the main thing that moves it.

  5. AbuseIPDB rate limits — the free tier is 1000 lookups/day.
     The enrich.py per-IP cache mitigates repeat lookups, but
     burst traffic from many distinct IPs can exhaust quota fast.
     Paid tiers raise the ceiling.

  6. Cloud LLM API rate limits — provider-specific, typically in
     the hundreds of RPM for paid tiers. Haiku is generous; Opus
     and other premium tiers are tighter.

SQLite is not on this list as a throughput limit, and on-disk
storage is no longer a contention concern either. The disk-bound
theoretical ceiling on reference hardware (consumer NVMe measured
at ~5,200 fdatasync ops/sec, ~1 fsync per dedup-passing alert in
v1.0) is on the order of hundreds of millions of alerts per day.
More importantly, a direct stress test settled the practical
question: on-disk single-disk SQLite was tested clean to
**~30.9M raw alerts/day sustained (~1000× this project's normal
alert rate)** on a fully shared disk — the database sharing one
drive with Graylog, Wazuh, Zeek, ntopng, and the OS — with
**zero lock contention: zero retries and zero `database is
locked` errors**, verified against every lock-handling path in
the code. No contention ceiling was reached. The test drive was a
high-end Gen4 consumer NVMe (a Western Digital Black) — good
consumer hardware, not enterprise storage — so the result reflects
what a normal well-specced deployment will see.

**On-disk single-disk SQLite is therefore the recommended default
for any realistic deployment.** You do not need special storage
for the database.

The RAM-disk (tmpfs) option documented earlier in this document
(see "HOSTING jrsoc.db IN RAM (tmpfs) WITH PERIODIC rsync TO LOCAL
SSD") is still available, but its purpose has changed. It is no
longer a contention fix — it is an **optional way to reduce drive
write activity** for operators who want to minimize SSD wear or
who run the database on a drive already under heavy combined load
from other writers. Note the honest math: at the database's own
write rate this is not a real wear concern — the deduplicated
write path peaks around ~400 writes/sec at realistic volume, well
below (~10×) the sustained-gapless rate at which consumer-SSD
wear-leveling and garbage collection start to matter, and the
real-traffic write stream has idle gaps that a wear-stressing
stream would not. So the RAM-disk is offered for operators who
want that margin or peace of mind, or whose drive is heavily
loaded by other work — not because the database alone requires it.
(These write-rate and saturation figures were measured on the
high-end consumer NVMe above; a lower-endurance or QLC drive has
less headroom, though the database's low write rate keeps it
comfortable there too.)

v1.0's write profile is already minimal: dedup is in-memory,
baselines are computed from read-only queries, frequency writes
are aggregated per dedup window (one row per window with an
occurrence count, not one per raw alert), and each commit batches
its INSERT plus retention DELETE. v2.0 swaps cold history for
PostgreSQL.

STORM TEST HISTORY
------------------

Alert storms — days with roughly double the normal alert volume,
or bursts of hundreds-to-thousands of alerts in a short window —
have occurred many times over this project's operational life,
both as real production events and as deliberate lab tests. The
short version: only the very first storm caused trouble, and that
was on a one-week-old version of the program with none of the
current defenses in place. Every storm since has been absorbed by
the configuration without operator-visible degradation.

The first storm (early, fragile). A sustained one-hour storm on a
roughly week-old version of the system surfaced specific
weaknesses in the original deduplication design. The program was
fragile at that point — the dedup window was short, there was no
rules.json engine, and known-noisy rules had no per-rule caps.
That storm directly motivated the current design: the rules.json
engine (per-rule hourly caps for known-noisy rule IDs) and the
longer dedup_silence_seconds=240 baseline window both came out of
what that storm exposed.

Every storm since (handled by configuration). The same storm
conditions were later reproduced in lab against the post-fix
system and did not produce storm behavior — the rules engine and
240-second dedup window absorbed the load without backlog. More
importantly, production operation has demonstrated the same
absorption pattern across many real-world storm events since. One
observed example: a 10,866-alert burst from a single misconfigured
agent compressed cleanly into a single dedup window (under 4
minutes), producing one LLM call and one operator notification
rather than 10,866. Over 20 separate storm events of 5,000+ alerts
have been observed during continuous operation, plus numerous
double-normal-volume days; all have been absorbed similarly. Storm
handling by the dedup-and-rules engine is not edge-case behavior
in this deployment — it is daily operating reality, and has been
since the week-one fixes landed.

Two things explain why storms are absorbed so easily in practice:

  - Storms are usually repeat offenders. In this deployment, a
    storm is typically the same rule firing on the same host (or
    small set of hosts) over and over — a scanner, a misconfigured
    agent, a noisy signature. That is exactly the pattern dedup is
    built to collapse. The storm gets eaten by the dedup window
    before it ever reaches the LLM, so a 10,000-alert storm
    becomes a handful of LLM calls, not 10,000. A storm of
    genuinely distinct, never-before-seen alerts would be a
    different and much rarer event.

  - The LLM stage tends to run with substantial headroom. This
    isn't a deliberate tuning choice — it falls out of running a
    single worker against this deployment's actual escalation
    volume — but in practice the LLM stage sits at roughly half
    its sustainable call rate during normal operation. That means
    alert volume can roughly double before the LLM stage even
    approaches saturation. Combined with dedup eating the
    repeat-offender storms, a doubling of inbound volume is barely
    noticeable operationally. Your own headroom depends on your
    worker count, endpoint speed, and escalation volume — measure
    it rather than assuming the same margin.

A storm here means hundreds of alerts arriving in seconds,
sustained across a multi-minute window, or simply a day running at
multiples of baseline volume. Real-world causes:

  - A network scanner hitting many DMZ hosts simultaneously
  - Wazuh rule misconfiguration suddenly generating thousands of
    matches
  - A new Suricata signature firing on every packet of a busy
    interface

Beyond the tested envelope, the queue may grow faster than the
LLM can process it, and alerts will back up. The pipeline does
not currently implement adaptive burst capacity (e.g., temporarily
activating cloud endpoints alongside local Ollama when queue
depth exceeds a threshold). Adaptive surge handling — dynamic
worker provisioning, queue-aware backpressure, and surge-triggered
dedup widening — is the lead v1.2 deliverable (see roadmap.md).
If your environment is storm-prone today, plan accordingly: tune
rules.json aggressively for known-noisy rules, raise max_concurrent
on your endpoints, configure a cloud fallback (which absorbs
alerts that fail the local call but not alerts that succeed slowly),
or filter more aggressively at the rule layer to keep the LLM from
being asked to evaluate scanner traffic.

WHAT WAS NOT EXERCISED
----------------------

Not measured at any scale:

  - Multi-tenant or multi-site deployments
  - Sustained sub-second alert bursts of genuinely distinct
    (non-deduplicating) alerts
  - Adaptive cloud burst capacity (not yet implemented;
    planned as the lead v1.2 deliverable — see roadmap.md)
  - Hardware-token or keyring-based credential storage (planned
    for v1.2 — see roadmap.md)

(Raw alert volume itself has been load-tested well into the
millions per day — see the WORKED EXAMPLE earlier in this
document — so high alert volume is no longer on this list.)

Most of the symptoms that look like "I've outgrown this pipeline"
have direct fixes available today:

  - SQLite lock timeouts → not expected on local disk at any
    realistic volume. On-disk single-disk SQLite tested clean to
    ~30.9M raw alerts/day with zero lock contention (see the
    SQLite section under the WORKED EXAMPLE). If you somehow see
    lock errors, first confirm the database is on LOCAL disk (not
    a network filesystem, which is unsupported) and not on a drive
    saturated by other writers. The RAM-disk (tmpfs) option exists
    but is for reducing drive wear / offloading a heavily-shared
    drive, not as a contention fix — see "HOSTING jrsoc.db IN RAM"
    earlier in this document.
  - LLM endpoint queue backups → raise max_concurrent / add
    workers, and if you are cloud-primary, move to a higher
    OpenAI (or other provider) rate-limit tier. Endpoint queue
    depth is usually a provider RPM-tier question, not a pipeline
    limit — higher tiers cost more but lift the ceiling
    immediately.

  - Baseline query slowdowns → this is the one symptom that is a
    genuine signal you are approaching the current architecture's
    practical limit, since baseline queries run against the SQLite
    history table. If you are seeing this routinely after the
    RAM-disk migration, that is a real "approaching the edge of
    what v1.0 targets" indicator.

The v1.2+ roadmap addresses the remaining architectural
constraints directly (adaptive surge handling and keyring
credentials in v1.2; config migration with tenant abstraction in
v1.3; cold-history PostgreSQL at v2.0).
But for the two most common symptoms — lock timeouts and endpoint
backups — you do not need to wait for a future version; the fixes
are available now.

DATABASE PLACEMENT
-------------------

SQLite is used for alert history, baseline calculations, and
rate limiting. SQLite does not work reliably on network
filesystems. Symptoms of this mistake include database lock
errors, corrupted WAL files after a network blip, and baseline
queries that return stale or inconsistent data.

Place the database file on LOCAL disk. For most deployments,
the pipeline working directory is a fine choice. Recommended
options in order of preference:

  1. A local SSD or NVMe volume on the jrSOCtriage host.
  2. A RAM-disk (tmpfs) with periodic rsync to a local SSD —
     optional, for operators who want to reduce SSD write activity
     (wear) or who run the database on a drive already heavily
     loaded by other writers. NOT a contention requirement:
     on-disk SQLite tested clean to ~30.9M/day with zero lock
     contention, so this is a wear/offload convenience, not a
     fix for a contention limit. More involved to set up;
     documented separately below under "HOSTING jrsoc.db IN RAM
     (tmpfs) WITH PERIODIC rsync TO LOCAL SSD." Not needed at
     typical SMB or home-lab volumes.
  3. A local HDD on the jrSOCtriage host.
  4. A docker volume mounted from local storage.

NOT recommended:

  - NFS shares
  - SMB / CIFS shares
  - Anything mounted over a network filesystem
  - Sync folders that actively replicate the file to another
    host while jrSOCtriage is writing to it

The jrsoc.db file is small (usually under 100 MB even with
months of alert history) and local storage is almost always
the right answer.

SCHEMA UPGRADES
----------------

v1 ships without an automatic schema migration path. If a
future release adds columns or tables, the upgrade procedure
for v1 -> vN will be:

  1. Stop the jrsoctriage service.
  2. Optional: export your rules.json, hosts.json, anonymization
     data, and any custom configuration. Your config and rules
     DO NOT need to be recreated.
  3. Delete jrsoc.db (or rename it as a backup).
  4. Deploy the new code and restart the service.
  5. The pipeline will create a fresh database using the new
     schema. Alert history begins accumulating from scratch
     and the baseline will be marked "Baseline not yet
     established" until min_baseline_days of history exists.

Alert history is inherently transient (14-day retention) so
losing it on upgrade is not a significant operational cost.

If a future release adds a migration path, these instructions
will be updated.


HOSTING jrsoc.db IN RAM (tmpfs) WITH PERIODIC rsync TO LOCAL SSD
-----------------------------------------------------------------

WHEN THIS IS WORTH DOING

The default jrsoc.db placement (local SSD or NVMe) is the right
answer for essentially every deployment. On-disk single-disk
SQLite was tested clean to ~30.9M raw alerts/day with zero lock
contention (zero retries, zero "database is locked" errors) on a
high-end Gen4 consumer NVMe shared with the full stack — so a
RAM-backed (tmpfs) hot copy is NOT needed to avoid contention, and
you should not expect lock warnings on a healthy local disk.

Switching to a RAM-backed (tmpfs) hot copy with periodic rsync to
a separate local SSD is therefore an OPTIONAL convenience, useful
in narrower cases:

  1. You want to minimize SSD write activity (wear) on the drive
     holding jrsoc.db. Be aware the math says this is not a real
     concern at the database's own write rate: the deduplicated
     write path peaks around ~400 writes/sec at realistic volume,
     roughly 10× below the sustained-gapless rate at which
     consumer-SSD wear-leveling and garbage collection begin to
     matter, and the real-traffic write stream has idle gaps a
     wear-stressing stream would not. So this is for peace of mind
     or for very long-lived deployments on lower-endurance drives,
     not a measured necessity.

  2. The drive holding jrsoc.db is already heavily loaded by other
     writers (Graylog/OpenSearch, Wazuh, Zeek) and you want to take
     the database's writes off that contended spindle entirely.
     Even though the database alone does not hit a contention
     ceiling, moving its hot copy to RAM removes its small share of
     IOPS from an already-busy drive, which can help the OTHER
     writers more than the database itself.

If neither applies, leave jrsoc.db on local disk — it is simpler,
fully durable, and tested clean well past any realistic volume.

For context on the contention question: on-disk single-disk
SQLite has been tested clean to ~30.9M raw alerts/day with zero
lock contention on a fully shared disk, so on a healthy local
drive you should not expect database-lock warnings at any
realistic volume, and the RAM-disk is not needed to prevent them.
Its value here is the two benefits below — taking the database's
writes off a contended or wear-sensitive drive — not fixing a
contention ceiling the database does not actually hit.

THE TWO BENEFITS

Hosting the hot copy of jrsoc.db on tmpfs and snapshotting it
to a separate local SSD every 30 seconds gives you two things:

  - Integrity. The pipeline's reads and writes hit RAM, which
    is fast and contention-free. The snapshot drive holds a
    durable copy you can recover from if the host crashes. The
    snapshot lag is bounded by the rsync interval (30 seconds
    by default) so the worst-case loss on an unclean shutdown
    is up to 30 seconds of frequency-counter increments. Those
    counters are statistical context, not audit data; losing
    30 seconds of them does not affect correctness, only the
    LLM's baseline accuracy briefly.

  - Reduced SSD wear. The sustained random-small-write workload
    of jrsoc.db (per-alert counter updates with fsync) is
    replaced by sequential rsync writes to the snapshot drive
    every 30 seconds. The hot path no longer wears any SSD.

WORKED EXAMPLE: STARTING FROM A TYPICAL DEPLOYMENT

Most operators reaching this section already have a running
jrSOCtriage install with jrsoc.db sitting on a local SSD inside
the install directory. You don't need to pick a new drive or
rearrange storage. The existing location of jrsoc.db becomes
the snapshot target. What changes is that the hot copy moves
into RAM, and the SSD only sees the periodic rsync write rather
than every per-alert update.

Throughout the steps below, <install-dir> refers to wherever
you have jrSOCtriage installed — the path set as
WorkingDirectory in jrsoctriage.service. Substitute your actual
install path everywhere <install-dir> appears.

The plan:
  - /run/jrsoctriage_tmpfs becomes the tmpfs mount holding the
    hot copy of jrsoc.db (256 MB is comfortable headroom; the
    database is normally under 100 MB).
  - <install-dir>/db-snapshot/ becomes the snapshot target. It
    lives on the same SSD as the install but in a subdirectory
    isolated from the rest of the install files.
  - On boot, a oneshot systemd unit copies the snapshot from
    that location into the tmpfs before jrsoctriage.service
    starts.
  - While jrSOCtriage runs, a systemd timer fires every 30
    seconds and rsyncs the tmpfs copy back to disk.
  - The pipeline's config.json points database.path at the
    tmpfs copy.

This removes the per-alert random-small-write workload from the
SSD — the disk only sees a sequential snapshot every 30 seconds.

INSTALLING rsync IF NOT ALREADY PRESENT

rsync is usually pre-installed but verify:

  rsync --version

If it is missing:

  sudo apt install rsync         # Debian / Ubuntu
  sudo dnf install rsync         # Fedora / RHEL / Rocky / Alma

No other packages are needed. The systemd machinery used below
is part of any modern Linux distribution.

STEP 1: STOP THE PIPELINE AND PREPARE THE SNAPSHOT DIRECTORY

The jrsoc.db file is being written to right now. Stop the
pipeline cleanly before moving it:

  sudo systemctl stop jrsoctriage

Verify it stopped:

  systemctl is-active jrsoctriage    # should print "inactive"

(If the pipeline was processing alerts when you stopped it, up
to 5 minutes of graceful drain time may elapse before it fully
exits. That's normal; wait for `systemctl is-active` to report
inactive before continuing.)

Next, create a dedicated subdirectory inside the install for
the snapshot. The subdirectory keeps the database files
visually separate from the rest of the install and makes
backup/inspection easier.

  sudo mkdir -p <install-dir>/db-snapshot
  sudo chown root:root <install-dir>/db-snapshot
  sudo chmod 755 <install-dir>/db-snapshot

Seed the snapshot directory by moving your existing jrsoc.db
files into it:

  sudo mv <install-dir>/jrsoc.db <install-dir>/db-snapshot/jrsoc.db
  sudo mv <install-dir>/jrsoc.db-wal <install-dir>/db-snapshot/ 2>/dev/null || true
  sudo mv <install-dir>/jrsoc.db-shm <install-dir>/db-snapshot/ 2>/dev/null || true

(The -wal and -shm files are SQLite's write-ahead log and
shared-memory companions. They may not exist if the database
was cleanly closed; the `|| true` keeps the command working
either way.)

STEP 2: DEFINE THE TMPFS MOUNT

Add this line to /etc/fstab so the tmpfs mounts on every boot:

  tmpfs  /run/jrsoctriage_tmpfs  tmpfs  size=256M,uid=0,gid=0,mode=0755  0  0

Create the mount point:

  sudo mkdir -p /run/jrsoctriage_tmpfs

Tell systemd to re-read /etc/fstab so it generates the
corresponding mount unit. Without this step, the auto-generated
mount unit name that Step 3 depends on may not exist yet:

  sudo systemctl daemon-reload

Then mount it:

  sudo mount /run/jrsoctriage_tmpfs

Verify with:

  mount | grep jrsoctriage_tmpfs

You should see a tmpfs entry with size=256M. The tmpfs is empty
at this point.

SEED THE TMPFS NOW (DO NOT SKIP — DATA-LOSS RISK)

The tmpfs is empty. The snapshot directory you populated in
Step 1 has your real jrsoc.db. Before going any further, copy
the database into the tmpfs:

  sudo cp <install-dir>/db-snapshot/jrsoc.db /run/jrsoctriage_tmpfs/jrsoc.db
  sudo cp <install-dir>/db-snapshot/jrsoc.db-wal /run/jrsoctriage_tmpfs/ 2>/dev/null || true
  sudo cp <install-dir>/db-snapshot/jrsoc.db-shm /run/jrsoctriage_tmpfs/ 2>/dev/null || true

Verify the copy:

  ls -la /run/jrsoctriage_tmpfs/

You should see jrsoc.db at the same size as in db-snapshot/.

Why this matters: in Step 4 you will enable a timer that
rsyncs the tmpfs to the snapshot directory every 30 seconds.
If you start that timer with an empty tmpfs, the first rsync
will sync "nothing" to the snapshot directory, and if you ever
used --delete on that rsync (earlier versions of these
instructions did, which was a mistake) your snapshot would be
destroyed. We now use plain `rsync -a` (no --delete) so
unexpected files in the snapshot dir are preserved — but you
still want the tmpfs seeded before the timer runs, because
otherwise the snapshot mtime gets stuck at "30 seconds ago"
with no fresh data flowing in.

STEP 3: CREATE THE BOOT-TIME RESTORE UNIT

Without this step, every boot would start the pipeline with an
empty jrsoc.db, because the tmpfs is freshly empty on every
boot (that's the nature of RAM). The restore unit copies the
disk snapshot back into the tmpfs before jrsoctriage.service
starts.

HOW THE BOOT SEQUENCE FITS TOGETHER

This step plus Steps 4 and 6 wire up the boot ordering. Once
they're all in place, every boot goes through:

  1. Kernel boots; systemd starts (T+0).
  2. systemd reads /etc/fstab and generates a mount unit named
     run-jrsoctriage_tmpfs.mount.
  3. The mount unit runs and mounts the tmpfs. The tmpfs is
     empty at this moment (RAM has no contents at boot).
  4. jrsoctriage-db-restore.service runs:
       - Requires= and After= the mount unit, so it can't run
         until the tmpfs exists.
       - Before= jrsoctriage.service, so it must finish before
         the pipeline starts.
       - Its rsync copies the snapshot into the tmpfs. For a
         <100MB database this completes in well under a second.
  5. jrsoctriage.service starts (the dependency edit in Step 6
     adds Requires= jrsoctriage-db-restore.service, so this
     waits for the restore to complete).
  6. Pipeline opens jrsoc.db at the tmpfs path (per the config
     change in Step 5) and resumes normally.
  7. At T+2min, the snapshot timer (Step 4) fires for the
     first time and rsyncs the tmpfs back to the snapshot
     directory. The 2-minute delay is deliberate — it ensures
     the first snapshot doesn't race with the restore and
     captures real post-boot state. After that, the timer
     fires every 30 seconds.

If any step fails, systemd refuses to start jrsoctriage.service
and logs the failure. That's the safety property: a partial
boot doesn't quietly start the pipeline on an empty database.

THE UNIT FILE

systemd auto-generates a mount unit from the fstab entry in
Step 2. The unit name is derived from the mount path: each
slash becomes a hyphen, and the leading slash is dropped. For
/run/jrsoctriage_tmpfs, the mount unit is:

  run-jrsoctriage_tmpfs.mount

You can confirm this on your system:

  systemctl list-units --type=mount | grep jrsoctriage

We use an underscore in the directory name rather than a hyphen
specifically to avoid systemd's escape-encoding rules — hyphens
inside path components have to be encoded as `\x2d` in unit
names, which is fragile across systemd versions. Underscores
pass through cleanly.

Create /etc/systemd/system/jrsoctriage-db-restore.service:

  [Unit]
  Description=Restore jrSOCtriage database from snapshot to tmpfs
  Requires=run-jrsoctriage_tmpfs.mount
  After=run-jrsoctriage_tmpfs.mount
  Before=jrsoctriage.service

  [Service]
  Type=oneshot
  RemainAfterExit=yes
  ExecStart=/usr/bin/rsync -a <install-dir>/db-snapshot/ /run/jrsoctriage_tmpfs/

  [Install]
  WantedBy=multi-user.target

Why Requires= AND After= for the same unit:
  - Requires= establishes a hard dependency. If the mount
    unit fails to start (or stops), the restore unit is also
    stopped/failed. This prevents the restore from trying to
    rsync into a non-existent mount point.
  - After= establishes ordering. Without it, Requires= alone
    only guarantees that systemd will start the mount, but
    doesn't guarantee the mount has completed before the
    restore unit starts. With both, the mount must be
    finished before the restore begins.

Why Before= jrsoctriage.service:
  - This ordering directive ensures the restore finishes
    before the pipeline starts, so the pipeline never opens
    an empty database. Step 6 will add Requires= on the
    jrsoctriage.service side to make this a hard dependency
    (mount-fails → restore-fails → pipeline doesn't start).

Enable it:

  sudo systemctl daemon-reload
  sudo systemctl enable jrsoctriage-db-restore.service

STEP 4: CREATE THE PERIODIC SNAPSHOT UNITS

A oneshot service does the rsync, and a timer triggers it every
30 seconds.

Create /etc/systemd/system/jrsoctriage-db-snapshot.service:

  [Unit]
  Description=Snapshot jrSOCtriage database from tmpfs to local SSD

  [Service]
  Type=oneshot
  ExecStart=/usr/bin/rsync -a /run/jrsoctriage_tmpfs/ <install-dir>/db-snapshot/

Create /etc/systemd/system/jrsoctriage-db-snapshot.timer:

  [Unit]
  Description=Run jrSOCtriage database snapshot every 30 seconds

  [Timer]
  OnBootSec=2min
  OnUnitActiveSec=30s
  AccuracySec=1s
  Unit=jrsoctriage-db-snapshot.service

  [Install]
  WantedBy=timers.target

(Why OnBootSec=2min rather than 30s: the first snapshot should
not race with the boot-time restore. 30 seconds after boot the
restore may still be running, or the pipeline may not yet have
written anything worth snapshotting. Two minutes is enough head
start that the restore is done, the pipeline is processing
alerts, and the first snapshot captures real post-boot state.
OnUnitActiveSec=30s then runs every 30 seconds from that point
on, which is the cadence we actually care about.)

Enable and start the timer:

  sudo systemctl daemon-reload
  sudo systemctl enable jrsoctriage-db-snapshot.timer
  sudo systemctl start jrsoctriage-db-snapshot.timer

Verify the timer is running:

  systemctl list-timers | grep jrsoctriage

You should see the next-fire time within the next 2 minutes
(the OnBootSec delay before the first snapshot). After the
first fire, subsequent fires will be 30 seconds apart.

STEP 5: UPDATE THE jrSOCtriage CONFIG (DO NOT SKIP THIS)

The pipeline reads the database path from config.json at
startup. Until you update this value, jrSOCtriage will keep
opening the old jrsoc.db at its original location and the
tmpfs + rsync infrastructure you just built will do nothing.
The snapshot timer will dutifully rsync an empty tmpfs every
30 seconds, the pipeline will continue hammering the old SSD,
and nothing about IOPS contention or SSD wear will change.

You have two ways to change it. Pick one — both edit the same
underlying config.json file.

OPTION A: through the interface (recommended)

  1. Open the jrsoctriage-interface web UI.
  2. Find the database path field. (As of v1.0 this lives near
     the database section of the config form.)
  3. Change the value from its current path (typically
     <install-dir>/jrsoc.db) to:
       /run/jrsoctriage_tmpfs/jrsoc.db
  4. Save.

OPTION B: directly in config.json

Edit <install-dir>/config.json and change the database.path
value:

  "database": {
    "path": "/run/jrsoctriage_tmpfs/jrsoc.db",
    ...
  }

Whichever path you choose, after saving, confirm the change by
reading config.json back:

  grep -A1 '"database"' <install-dir>/config.json

You should see the new tmpfs path. If you still see the old
path, the save did not take effect; do not proceed.

STEP 6: UPDATE THE jrsoctriage.service UNIT

The service must not start until the tmpfs is mounted and the
snapshot has been restored. Edit jrsoctriage.service and add
the restore unit as a dependency. The [Unit] section should
read:

  [Unit]
  Description=jrSOCtriage AI Security Triage Pipeline
  After=network-online.target docker.service jrsoctriage-db-restore.service
  Wants=network-online.target
  Requires=jrsoctriage-db-restore.service

Reload and restart:

  sudo systemctl daemon-reload
  sudo systemctl restart jrsoctriage

Watch the journal during the restart to confirm the database
opens at the new path:

  sudo journalctl -u jrsoctriage -f

The "Database ready" log line should appear and the pipeline
should resume normally.

VERIFYING THE SETUP IS WORKING

After the first 2 minutes (the timer's OnBootSec delay), the
first snapshot should have fired:

  ls -la <install-dir>/db-snapshot/

The mtime on jrsoc.db should be recent (within the last 30
seconds — once the first snapshot has fired, subsequent ones
run every 30 seconds). Wait another minute and check again —
the mtime should advance, confirming the timer continues to
fire on its 30-second cadence.

If you can't wait 2 minutes for the first snapshot, you can
trigger it manually:

  sudo systemctl start jrsoctriage-db-snapshot.service

Then check the snapshot directory mtime; it should be current.

You can also list the timer status to see when the next fire
is scheduled:

  systemctl list-timers | grep jrsoctriage

Watch I/O on the snapshot drive briefly to confirm the writes
are short bursts and not constant:

  sudo iotop -o          # or use `dstat` or `iostat` to taste

You should see jrsoctriage doing almost no disk I/O during
normal operation (the writes are going to RAM). Every 30
seconds you should see a brief burst as the rsync runs.

VERIFYING BOOT RECOVERY ACTUALLY WORKS

The riskiest part of this setup is what happens at reboot: if
the boot ordering is wrong, the pipeline could start before the
restore has populated the tmpfs, and you'd silently get a fresh
empty database. It's worth testing this deliberately rather
than discovering it after a power event.

To test without rebooting the host, simulate the boot sequence:

  # Stop the pipeline.
  sudo systemctl stop jrsoctriage.service

  # Stop and reset the restore unit so it'll run again
  # (oneshot units with RemainAfterExit=yes stay "active" once
  # they've run; this clears that state).
  sudo systemctl stop jrsoctriage-db-restore.service

  # Empty the tmpfs to simulate post-boot state.
  sudo rm -f /run/jrsoctriage_tmpfs/*

  # Start the pipeline. Because jrsoctriage.service Requires
  # jrsoctriage-db-restore.service, systemd will run the
  # restore first.
  sudo systemctl start jrsoctriage.service

  # Verify both units are active and the tmpfs is populated.
  systemctl is-active jrsoctriage-db-restore.service   # should print "active"
  systemctl is-active jrsoctriage.service              # should print "active"
  ls -la /run/jrsoctriage_tmpfs/                       # should show jrsoc.db

  # Check the journal for the pipeline opening the database.
  sudo journalctl -u jrsoctriage.service --since "1 minute ago" | grep -i "database"

If the tmpfs is populated and "Database ready" appears in the
journal, the boot ordering is working. If the tmpfs is empty or
the pipeline logs a fresh-database initialization, the boot
ordering is wrong — re-check Steps 3 and 6.

When you're ready to test against an actual reboot, schedule it
during a quiet window and watch journalctl as the host comes
back up to confirm the same expected sequence.

CRASH RECOVERY EXPECTATIONS

On unclean shutdown (kernel panic, power loss, SIGKILL of the
pipeline), the worst-case loss is whatever was written to the
tmpfs after the last successful rsync — up to 30 seconds of
frequency-counter increments. These counters feed the LLM's
baseline context (how often this rule fires for this host on
average) and are statistical, not audit. A 30-second gap in
their accumulation does not affect correctness; it slightly
reduces baseline accuracy for the affected (host, rule) pairs
until enough subsequent traffic refills them.

The persistent snapshot in db-snapshot/ always reflects state
as of the most recent rsync. If the tmpfs is empty at boot
(which it always is, before the restore unit runs), the
boot-time restore unit populates it before jrSOCtriage starts.

If both the tmpfs and the snapshot drive fail simultaneously,
you have larger problems than jrsoc.db.

UNDOING THIS CHANGE

If you decide to revert:

  1. Stop the pipeline and the snapshot timer.
       sudo systemctl stop jrsoctriage.service
       sudo systemctl stop jrsoctriage-db-snapshot.timer
       sudo systemctl disable jrsoctriage-db-snapshot.timer
  2. Copy the database back to its original location.
       sudo cp /run/jrsoctriage_tmpfs/jrsoc.db <install-dir>/jrsoc.db
  3. Edit config.json to restore database.path to its original
     value.
  4. Remove the boot-time restore dependency from
     jrsoctriage.service.
  5. Reload systemd and restart the pipeline.

The tmpfs mount can stay in /etc/fstab harmlessly, or be
removed.


============================================================
SECURITY: FILE PROTECTION
============================================================

jrSOCtriage stores configuration and operational state in JSON
files on disk. These files require careful permission management
for two reasons: protecting CREDENTIALS that live inside them, and
protecting OPERATIONAL INTEGRITY against silent tampering.

The threat model assumes the host has a single trusted OS user
(or only trusted OS users) — this is about local operating-system
accounts on the machine, not about how many jrSOCtriage
administrators use the application. If multiple non-root users
share the host, treat any default permissions as insufficient and
follow the lockdown procedure below.

------------------------------------------------------------
Two threat categories
------------------------------------------------------------

CONFIDENTIALITY: config.json holds plaintext credentials.
Reading it gives an attacker:
  - LLM API keys (Anthropic, OpenAI, Gemini)
  - ntopng web username and password
  - Graylog API username and password
  - SMTP username and password (for email notifications)
  - AbuseIPDB API key
  - GreyNoise, VirusTotal, and AlienVault OTX API keys (v1.1
    threat-intelligence sources)
There is no keyring integration, no environment-variable
indirection, and no encryption at rest in v1.0. This is
appropriate for a host with only trusted OS users. It is NOT
appropriate for shared hosts where untrusted local users have
accounts.

v1.2 will add keyring-based credential storage with a one-shot
migration from existing config.json. This is the largest single
security concern in this release and closing it is a v1.2
priority — see roadmap.md for the full v1.2 feature list. Until
v1.2 ships,
operators in environments where filesystem credentials are a
real concern should use file-system permissions (mode 600,
ownership locked to the service account) and consider running
the pipeline on a host where only the operator has shell access.

INTEGRITY: hosts.json, rules.json, and the anonymization files
(users.json, domain.json, ip_aliases.json, anonymization.json)
shape the LLM prompt and the rules engine. An attacker who can
modify or delete these files could:
  - Strip host context from prompts so all alerts look generic
  - Remove rule-engine entries to bypass the decision engine
  - Corrupt anonymization mappings to leak real identifiers
  - Disrupt triage quality without triggering visible errors
The pipeline keeps running. Verdicts get worse silently. That
is the worst class of failure mode for a triage tool.

------------------------------------------------------------
Lock down sensitive files
------------------------------------------------------------
It is generally safe to run sudo setup.sh over again, which
will automatically change the permissions of sensitive files.
However, if you manually whish to lock them down, follow these
instructions.

Mode 600 (owner-only read AND write) on every state file:

  sudo chown root:root /path/to/jrsoctriage/config.json
  sudo chmod 600 /path/to/jrsoctriage/config.json

  sudo chown root:root /path/to/jrsoctriage/interface_auth.json
  sudo chmod 600 /path/to/jrsoctriage/interface_auth.json

  sudo chown root:root /path/to/jrsoctriage/hosts.json
  sudo chmod 600 /path/to/jrsoctriage/hosts.json

  sudo chown root:root /path/to/jrsoctriage/rules.json
  sudo chmod 600 /path/to/jrsoctriage/rules.json

  sudo chown root:root /path/to/jrsoctriage/users.json
  sudo chmod 600 /path/to/jrsoctriage/users.json

  sudo chown root:root /path/to/jrsoctriage/domain.json
  sudo chmod 600 /path/to/jrsoctriage/domain.json

  sudo chown root:root /path/to/jrsoctriage/ip_aliases.json
  sudo chmod 600 /path/to/jrsoctriage/ip_aliases.json

  sudo chown root:root /path/to/jrsoctriage/anonymization.json
  sudo chmod 600 /path/to/jrsoctriage/anonymization.json

The pipeline runs as root, so 600 does not restrict it. The
restriction is on every other user on the host.

The interface enforces this automatically going forward:

  - Every save through the interface (config edits, hosts/rules
    edits, user adds, etc.) sets mode 600 on the written file.
  - Auth file writes (save_auth) set mode 600.
  - On startup, interface.py scans the configured state files
    and warns to stderr about any with permissive modes,
    showing the exact chmod commands to fix them.

The startup warning is a one-shot reminder. Manual chmod is
still required for files that already exist with wrong modes —
the interface does NOT automatically chmod files it didn't write,
to avoid surprising operators who deliberately chose looser
permissions for their environment.

The provided setup.sh file is safe to re-run. This file will
automatically set permissions on any file not present when it
first ran.

------------------------------------------------------------
Backups
------------------------------------------------------------

If you tar the install directory for backup, the archive
contains plaintext credentials. Treat the archive with the same
care as config.json itself: root-owned, restricted permissions,
encrypted at rest if shipped off-host.

If you sync the install directory between hosts (rsync, NFS,
git, etc.), be aware that the destination MAY change file
ownership and permissions. After any sync, verify with:

  ls -la /path/to/jrsoctriage/*.json

and re-run the chmod commands above if needed. Consider
excluding config.json from broad sync rules, or running the
sync as a process that preserves ownership.

------------------------------------------------------------
Credential rotation
------------------------------------------------------------

  - LLM API keys: revoke at the vendor (Anthropic, OpenAI, etc),
    update config.json, restart the pipeline.
  - ntopng / Graylog passwords: change at the source, update
    config.json, restart the pipeline.
  - SMTP password: change at your provider, update config.json,
    restart the pipeline.
  - Web interface user password / TOTP: see the USERS (web
    interface logins) section in interface_guide.txt.

v1.2 adds keyring-based credential storage with a one-shot
migration from existing config.json — see roadmap.md for the
full v1.2 feature list. Until then, plaintext in mode-600
root-owned files is the security model. If your threat model
requires keyring-grade credential isolation today, wait for v1.2
or do not deploy in environments where filesystem credentials are
a concern.


============================================================
BUILDING THE HOST INVENTORY FROM WAZUH (AGENT IMPORT)
============================================================

You do not have to type your host inventory by hand. If your hosts
are Wazuh agents, jrSOCtriage can pull the agent list straight from
the Wazuh management API and turn it into host entries for you. On a
real network this is the difference between an afternoon of data
entry and a few clicks - and it stays useful afterward as a
reconciliation and maintenance tool, not just a one-time onboarding
step.

The full click-by-click walkthrough is in interface_guide.txt under
"WAZUH API - AGENT HOST IMPORT." This section covers the setup
prerequisites and the operational picture.

WHAT IT NEEDS (one-time setup)

  1. The Wazuh MANAGEMENT API reachable from the jrSOCtriage host.
     This is the manager API, default port 55000 - NOT the dashboard.
     On a Docker deployment, confirm the manager container PUBLISHES
     55000:

       docker ps | grep wazuh

     The manager line should show 0.0.0.0:55000->55000/tcp. If it
     only shows 55000/tcp (exposed but not published), add a ports
     mapping for 55000 to the manager service and recreate it.

  2. A Wazuh API USER - NOT your dashboard login. On the standard
     Wazuh Docker stack the API user is "wazuh-wui". The two are
     different accounts; the dashboard login will return "Invalid
     credentials" against the API. If you have lost track of the API
     password, it is stored where the dashboard keeps its own copy
     (the dashboard uses the same API user to list agents).

  3. The API is HTTPS with a SELF-SIGNED certificate by default, so
     "Verify SSL" is normally turned OFF on the connection card. Turn
     it on only if you have installed a properly-signed certificate.

  4. DNS that can resolve your hosts - forward AND reverse (PTR). The
     import verifies every agent against DNS (see below), and the
     verification is anchored on the REVERSE (PTR) record for each
     agent's IP. jrSOCtriage assumes hosts are registered in DNS with
     both a forward record and a PTR record. In Active Directory the
     forward records are usually automatic (machines self-register on
     domain join), but REVERSE zones and PTR records are NOT automatic -
     an admin must create the reverse lookup zones and enable PTR
     registration. Without PTR records, agents may fail verification
     (landing in IP/name mismatch, or skipped by renumber), and
     canonical-hostname resolution at alert time may fall back to the
     raw agent name.
     If your agents resolve on a different DNS server than the
     jrSOCtriage host uses by default - common on networks with an
     internal/AD DNS separate from a general resolver - set the "Custom
     DNS server" field on the card to the resolver that knows your hosts.
     Without it, internal hosts can fail verification simply because the
     default resolver doesn't know them.

     WORKING WITHOUT COMPLETE DNS: you can still run jrSOCtriage in a
     DNS-sparse environment by building hosts.json MANUALLY. A host with
     a manual entry gets its context regardless of DNS - the one
     requirement is that the entry's NAME match the Wazuh agent name (the
     pipeline's fallback when DNS can't resolve identity), and pin the IP
     with the Static IP field. The Wazuh import/renumber convenience
     features lean on DNS; the underlying pipeline does not, so manual
     host entries are a complete workaround.

     ADDING PTR RECORDS: AD does not auto-create reverse (PTR) records
     the way it does forward records - a common oversight worth checking.
     On a small network, add PTR records by hand in DNS Manager or enable
     PTR registration in the DHCP scope; at larger scale, search for "PTR
     record management" / "bulk create PTR records" utilities (most IPAM
     products automate it). This is standard DNS administration, outside
     jrSOCtriage's scope.

You enter all of this once on the "Wazuh API for Agent Host names"
card at the top of the Hosts tab, and save it.

WHY IT IS SAFE TO TRUST

The import does not blindly copy Wazuh's records. A Wazuh agent can
report a stale IP or a name that doesn't match reality. So every
agent is checked against DNS - the authoritative source for host
identity - and sorted automatically:

  - Cleanly verified, new        -> offered for one-click add
  - Name OK but IP disagrees      -> offered with a safe default
                                     (no fixed IP; resolved live) and
                                     the option to pin DNS's or the
                                     agent's IP
  - Name doesn't match DNS        -> two options: add it under its DNS
                                     name with one click (works now,
                                     because the pipeline identifies
                                     hosts by DNS not by the agent's
                                     reported name), or re-enroll the
                                     agent under the correct name for a
                                     clean fix at the source. See the
                                     re-enrollment runbook in
                                     interface_guide.txt ("FIXING A
                                     MISNAMED AGENT") - note that
                                     renaming the agent's ossec.conf
                                     alone is NOT enough; the manager
                                     keeps the old name until the agent
                                     re-enrolls.

You only spend attention where a real decision exists. Everything
that verifies cleanly is a single bulk action.

Nothing is written until you choose "Close & Save"; "Close" discards
a session you don't like. Adding a host that already exists UPDATES
its IP without touching the notes, roles, tags, or alias you wrote -
your hand-authored context is never overwritten by an import.

ONGOING MAINTENANCE

Re-running the import later reconciles your inventory against current
reality: newly-enrolled agents show up as addable, and agents whose
identity has drifted surface in the mismatch column. It is not just
an onboarding tool - it is how you keep hosts.json honest as the
network changes.

A note on IP addresses and network changes: hosts with NO fixed IP
(left blank, resolved live at alert time) are unaffected by a network
renumber - the pipeline simply resolves their new address. Only hosts
with a PINNED static IP need attention after a renumber. Preferring
blank IPs for anything that uses DNS keeps maintenance near zero.


============================================================
BRINGING HOSTS OVER FROM ANOTHER SECURITY DOMAIN (MERGE)
============================================================

When you run jrSOCtriage in more than one security domain - for
example a separate management domain and a sales domain, each with
its own Wazuh manager and its own hosts.json - a host that travels
between domains, or that everyone talks to across domains, loses its
context in whichever domain does not have it in its inventory.

Concretely: a sales laptop's Wazuh agent stays attached to its home
sales-domain manager. When that laptop relocates to the management
network, the management domain's Suricata sees its traffic and alerts
on it - but the management hosts.json has no entry for it, so the LLM
triages those alerts with no idea what the host is or what is normal
for it. The same gap appears for shared servers: if everyone connects
to a server that lives in one domain, every other domain's sensors
alert on that server with no context.

A third case is deterministic rather than incidental - Wazuh's
manager failover feature. Wazuh lets you configure an agent with a
LIST of managers or workers (which may belong to different domains);
per Wazuh's documentation, if the agent loses connection to its
current manager it cycles through the configured servers until one
responds. If those servers span domains, a failover can therefore
route a host's agent into a DIFFERENT domain's manager - and that
domain then needs the host's context to triage its alerts sensibly.
Unlike a laptop that happens to wander, this is a configured behavior:
if you use (or plan to use) Wazuh's manager failover across domains,
you have effectively decided your hosts can report into more than one
domain, so merging their context ahead of time is how you make sure
triage still has that context when a failover routes them cross-domain.
Configuring the failover manager list is a Wazuh matter, not a
jrSOCtriage one - see Wazuh's own documentation on agent enrollment /
agent configuration for how it works and how to set it up. (This note
only covers why it matters for jrSOCtriage's cross-domain context.)

The fix is to fold the other domain's inventory into yours so the
traveling, shared, or failover-routed hosts carry their context
wherever they are seen.
A standalone utility, merge_hosts.py, does this. It is a separate tool
- it does not require a running pipeline and imports nothing from
jrSOCtriage, so you can run it anywhere Python 3 is available.

WHAT IT DOES

merge_hosts.py takes two inputs - a SOURCE hosts.json (the other
domain's inventory, or a hand-built file of just the hosts you want
to bring over) and YOUR hosts.json - and folds the source hosts into
yours. Your file is updated in place, after a timestamped backup of
your original is written alongside it.

  sudo python3 merge_hosts.py management_hosts.json

That merges management_hosts.json into ./hosts.json. sudo is needed
because hosts.json is owner-only (mode 600); the tool preserves that
mode on both the backup and the rewritten file so your inventory is
never left world-readable.

If your hosts.json is not in the current directory, point at it:

  sudo python3 merge_hosts.py management_hosts.json -d /path/to/hosts.json

HOW COLLISIONS ARE HANDLED

Your file always wins. The source can only ADD hosts you do not
already have; it never overwrites a host you have authored. Every
collision is reported so you can see what was kept and what was
skipped:

  - Host name collisions (case-insensitive): your entry is kept, the
    source entry is skipped. The notes and roles you wrote are never
    touched.
  - Network collisions (matched on CIDR): your entry is kept. If the
    source uses a different NAME for the same CIDR, that is reported
    so you know the two domains disagree on what to call it.

IP ADDRESSES ARE BLANKED BY DEFAULT

Hosts brought in from the source have their identifiers.ip removed by
default, so they resolve live (by DNS or agent name) in your domain
rather than carrying a stale address from the other domain. This is
the correct behavior for the hosts this tool is meant for: a host
that appears cross-domain is either mobile (its address in your
domain is not its address in the source domain) or a server reached
by DNS (its address resolves fine without a pin). Blanking also means
there are no IPs to mis-attribute across domains.

Your OWN hosts are never touched - only the hosts brought in from the
source are blanked. If you have a specific reason to preserve the
source IPs (for example the two domains share addressing and the pins
are correct), use --keep-ips:

  sudo python3 merge_hosts.py management_hosts.json --keep-ips

SAFETY

Both files are validated as well-formed hosts.json BEFORE anything is
written. If either file is malformed, the tool reports the problem and
changes nothing - no backup is made and your file is left untouched.
On a successful run your original is always preserved as
hosts.json.bak.<timestamp>, so a merge is never destructive.

Running the same merge twice is safe: the second run finds every host
already present (your file wins the collisions) and adds nothing.

A NOTE ON WHEN TO USE THIS

This assumes the two domains share a routable network - which is what
makes cross-domain traffic, and therefore cross-domain alerts, happen
in the first place. That shared routing also guarantees the two
domains do not reuse the same private IP ranges. Do not use this to
merge inventories from disconnected sites that independently reuse the
same private ranges (two separate 192.168.1.0/24 networks, say) -
the hosts would appear to collide when they are actually different
machines. The tool is for domains that can see each other.


============================================================
RECOMMENDED: RUNNING AS A SYSTEMD SERVICE
============================================================

Running as a systemd service means jrSOCtriage starts automatically on boot,
restarts automatically if it crashes, and logs to the system journal. To get
started, make the script executable and run it:

  sudo chmod +x setup.sh
  sudo ./setup.sh   

This will update the system units that ship with the project to include your
install directory.

Two units ship with the project:
  jrsoctriage.service           - the pipeline (main.py, system Python)
  jrsoctriage-interface.service - the web interface (interface.py, venv)

You can run either or both. Most installations want both: the interface
for configuration and observation, the pipeline for actual triage work.

BEFORE INSTALLING
------------------
Both unit files have hardcoded paths assuming an install at
/mnt/appdata/jrsoctriage. If your install is elsewhere, edit BOTH files
to update WorkingDirectory.  These hardcoded paths are replaced by setup.sh
with the current install directory.

The pipeline unit uses /usr/bin/python3 (system Python). This is correct
because main.py only needs `requests`, which is typically installed
system-wide. If your `requests` is in a different location (rare), edit
the ExecStart line.

The interface unit uses venv-relative paths (./venv/bin/python3). This
assumes you ran setup.sh or set up the interface venv manually as
described in the WEB INTERFACE section above. If you installed interface
deps system-wide instead (Option C), change the ExecStart in
jrsoctriage-interface.service to /usr/bin/python3 instead.

FIRST-RUN SETUP FOR INTERFACE
------------------------------
The first time you start the interface, it drops into a terminal flow to
create the first admin user (prompts for username, password, scans QR code
for TOTP). Systemd has no terminal, so this WILL NOT work under systemd.

Run interface.py manually ONCE before installing the systemd unit:

  cd /path/to/jrsoctriage
  sudo ./venv/bin/python3 interface.py

Complete the first-user prompts, then Ctrl+C to stop. Subsequent additional
users can be added via `interface.py --add-user`.

Note: The pipeline (main.py) runs using the system Python environment. The 
web interface uses a separate virtual environment to isolate its 
dependencies.

STEP 1: Copy the service unit files
------------------------------------
sudo cp jrsoctriage.service /etc/systemd/system/
sudo cp jrsoctriage-interface.service /etc/systemd/system/

STEP 2: Reload systemd
-----------------------
sudo systemctl daemon-reload

STEP 3: Enable on boot
-----------------------
sudo systemctl enable jrsoctriage jrsoctriage-interface

STEP 4: Start the services
---------------------------
Start the interface first so you can configure jrSOCtriage:

  sudo systemctl start jrsoctriage-interface

Then connect to it (see ACCESSING FROM ANOTHER MACHINE below), complete
configuration, and start the pipeline:

  sudo systemctl start jrsoctriage

STEP 5: Verify both are running
--------------------------------
sudo systemctl status jrsoctriage
sudo systemctl status jrsoctriage-interface

You should see: Active: active (running)


DAILY OPERATIONS:
-----------------
Stop:     sudo systemctl stop jrsoctriage
Start:    sudo systemctl start jrsoctriage
Restart:  sudo systemctl restart jrsoctriage
Logs:     sudo journalctl -u jrsoctriage -f
Status:   sudo systemctl status jrsoctriage

NOTE: Stop and restart can take 15-30 seconds to complete. The pipeline
finishes processing the current alert before exiting cleanly. This is
normal — do not force kill. Wait for the prompt to return.

If you load an update to main.py or any other pipeline module,
sudo systemctl restart jrsoctriage.service must be run for it to
take effect. If you load an update to interface.py, restart the
interface unit instead:

  sudo systemctl restart jrsoctriage-interface.service

NOTE: Restart is required after changes to:
  - config.json
  - rules.json
  - Any .py module file (pipeline → jrsoctriage.service;
    interface.py → jrsoctriage-interface.service)

hosts.json and network-related context are reloaded dynamically during normal
operation and do not require a restart. The service may also be restarted
directly in the interface.

ALTERNATIVE: RUNNING MANUALLY (no systemd)

For testing, development, or systems without systemd:

cd /path/to/jrsoctriage
sudo python3 main.py

Logs will print directly to the terminal.
Press Ctrl+C to stop cleanly — the pipeline handles SIGINT gracefully

============================================================
ZEEK SERVICE optional (required for network flow context)
============================================================

Zeek must be running for network flow data to appear in alerts.
It is managed separately from jrSOCtriage.

REQUIRED: Zeek must log in TSV format (the default). jrSOCtriage's Zeek
reader parses the TSV "#fields" header and tab-separated rows. If Zeek is
configured for JSON logs (the json-streaming / json-logs policy), the reader
finds no "#fields" header and silently returns EMPTY Zeek context with no
error — alerts will simply show no network flow data. If you are pointing
jrSOCtriage at an existing Zeek sensor, confirm it logs TSV first. (JSON /
dual-format support is planned for a later release; until then TSV is
required.)

Check status:   sudo systemctl status zeek
Start:          sudo systemctl start zeek
Stop:           sudo systemctl stop zeek
Enable on boot: sudo systemctl enable zeek

If zeek.service is missing, create it at:
/etc/systemd/system/zeek.service

---------- BEGIN zeek.service ----------
[Unit]
Description=Zeek Network Security Monitor
After=network-online.target
Wants=network-online.target

[Service]
Type=forking
ExecStart=/opt/zeek/bin/zeekctl start
ExecStop=/opt/zeek/bin/zeekctl stop
ExecReload=/opt/zeek/bin/zeekctl restart
Restart=on-failure
RestartSec=60

[Install]
WantedBy=multi-user.target
---------- END zeek.service ----------


ZEEK LOG PATH MODEL
--------------------

jrSOCtriage reads two kinds of Zeek output:

  1. CURRENT logs - where Zeek writes live data. Files are named
     conn.log, dns.log, ntlm.log (no date/time suffix).
  2. ARCHIVE logs - dated subdirectories containing rolled-over logs.
     Files are named conn.HH:MM:SS-HH:MM:SS.log.gz, etc. Archive
     directory contains subdirs like 2026-04-22/, 2026-04-23/.

Config exposes these as two independent paths so they can live on
separate mounts or filesystems:

  sources.zeek.current_log_dir   (e.g. /opt/zeek/logs/current)
  sources.zeek.archive_log_dir   (e.g. /var/log/zeek-archive)

Three common layouts are supported:

LAYOUT 1 - Stock Zeek, everything under one parent:
  /opt/zeek/logs/
    current/         ← current_log_dir
    2026-04-22/      ← archive is parent of these
    2026-04-23/

  config:
    current_log_dir: "/opt/zeek/logs/current"
    archive_log_dir: "/opt/zeek/logs"      (or leave blank - defaults to parent)

LAYOUT 2 - Current and archive on different mounts:
  /opt/zeek/logs/current/                  ← current_log_dir
  /mnt/bulk-storage/zeek-archive/
    2026-04-22/
    2026-04-23/

  config:
    current_log_dir: "/opt/zeek/logs/current"
    archive_log_dir: "/mnt/bulk-storage/zeek-archive"

LAYOUT 3 - Symlink trick (common when archive moved mid-project):
  /mnt/logstore/zeek/current/              ← actual current
  /mnt/logstore/zeek/2026-04-22/           ← actual archive dirs

  config:
    current_log_dir: "/mnt/logstore/zeek/current"
    archive_log_dir: "/mnt/logstore/zeek"

BACKWARD COMPATIBILITY
  Older configs used a single 'sources.zeek.log_dir' pointing at the
  current directory, with archives assumed to be siblings of current/
  under the shared parent. This still works - jrSOCtriage detects
  the legacy form, treats log_dir as current_log_dir, defaults
  archive_log_dir to the parent, and logs a deprecation warning on
  startup. Edit your config via the interface and click Save to
  migrate to the new fields.

VERIFYING THE PATHS
  Start the pipeline and watch for INFO lines like:
    zeek_fetch: Zeek returned N conn, M dns, ... row(s)
  Non-zero row counts (when the host had network activity in the
  alert window) confirm both paths are configured correctly. If all
  alerts show 0 rows despite active hosts, check that current_log_dir
  contains conn.log and archive_log_dir contains dated subdirs.



============================================================
WEB INTERFACE (interface.py)
============================================================

jrSOCtriage includes a web-based configuration interface for editing
all config files, viewing live logs, and restarting the service without
needing to edit JSON files manually.

INSTALLATION
-------------
The PIPELINE (main.py) and the INTERFACE (interface.py) have different
dependency profiles, and only the interface needs a venv.

PIPELINE DEPENDENCIES (main.py)
  Two third-party Python packages: `requests` (HTTP client used for
  Graylog/Ollama/AbuseIPDB/GeoIP) and `dnspython` (thread-safe reverse
  DNS for hostname resolution). Both are listed in `requirements.txt`
  and installed system-wide by `setup.sh` automatically.

  If you prefer to install manually:

    sudo pip install -r requirements.txt --break-system-packages

  Or via your distro's package manager where versions are recent enough:

    sudo apt install python3-requests python3-dnspython    # Debian/Ubuntu
    sudo dnf install python3-requests python3-dnspython    # Fedora/RHEL

  Note that `dnspython` packaged by some distros lags behind PyPI. If
  the distro version is too old to import correctly, fall back to pip.

  Pipeline dependencies are installed into the system Python (not a
  venv) because the systemd service runs main.py with /usr/bin/python3.

  In addition to Python packages, the pipeline shells out to the
  `whois` command-line tool for external IP organization lookups.
  Install it via your distro:

    sudo apt install whois        # Debian/Ubuntu
    sudo dnf install whois        # Fedora/RHEL

  If `whois` is not installed, external IP whois enrichment is silently
  skipped — the pipeline keeps running but org/netname fields will be
  blank in alerts that would otherwise have them populated.

  Everything else main.py uses is Python standard library.

INTERFACE DEPENDENCIES (interface.py)
  The web interface needs Flask, bcrypt, pyotp, and qrcode. These are
  not standard-library and not commonly pre-installed. Use a venv to
  avoid polluting system Python.

  OPTION A — Recommended: setup.sh (one command, idempotent)
    From the jrSOCtriage source directory:

      bash setup.sh

    The script:
      - Requires Python 3.12 or newer (3.13 and 3.14 are formally
        tested; 3.12 is allowed with a notice that it is untested,
        and versions newer than 3.14 are allowed with a forward-
        compat warning)
      - Creates a venv at ./venv
      - Installs interface dependencies into the venv
      - Sets mode 600 on existing config/state files
      - Edits systemd files to point to correct install directory
      - Prints next steps

    Re-running setup.sh is safe — it skips work that's already done.
    Override Python with PYTHON_BIN=python3.14 bash setup.sh if needed.

    

  OPTION B — Manual venv (if you prefer to do it by hand)
    The venv is created by your normal user in the jrsoctriage directory.
    The interface then runs with sudo using the venv's python directly,
    so root can read /mnt/appdata and restart the service while still
    using the venv's package set.

      cd /path/to/jrsoctriage
      python3 -m venv venv
      source venv/bin/activate
      pip install -r interface_requirements.txt
      deactivate

  OPTION C — System-wide (least clean, but quickest for one-host labs)
    If you don't want to manage a venv, install interface deps system-wide:

      sudo pip install -r interface_requirements.txt --break-system-packages

    Pros: no venv to manage.
    Cons: packages go into the system Python; they may conflict with
    distro-managed packages over time; harder to remove cleanly.

interface_requirements.txt lists the direct interface dependencies:

    Flask>=3.0,<4
    bcrypt>=4.0
    pyotp>=2.9
    qrcode>=7.4

pip resolves transitive dependencies (Werkzeug, Jinja2, MarkupSafe,
itsdangerous, click, blinker) automatically.

RUNNING THE INTERFACE
----------------------
Run from the jrsoctriage directory.

If you used setup.sh or manual venv (Options A or B), use the venv's
python with sudo. Do NOT activate the venv and then sudo — sudo resets
PATH and will pick up system python instead, which won't have the
packages:

  cd /path/to/jrsoctriage
  sudo ./venv/bin/python3 interface.py

If you installed interface deps system-wide (Option C):

  cd /path/to/jrsoctriage
  sudo python3 interface.py

The interface starts on http://127.0.0.1:9090 by default.
To use a different port:

  sudo ./venv/bin/python3 interface.py --port 9191

To specify a config file explicitly:

  sudo ./venv/bin/python3 interface.py --config /path/to/config.json

To add a new authentication user from the terminal (GUI Add User
in the Users tab is also supported; see interface_guide.txt → USERS
(web interface logins) for full details on both paths):

  sudo ./venv/bin/python3 interface.py --add-user

This launches the same prompted flow as first-run setup but
appends to the existing auth file rather than creating a new one.
Useful for adding additional admins from a script or remote shell
where the GUI is not convenient (the command exits after the user
is added; restart the interface as a separate step if needed).

Stop with Ctrl+C.

ACCESSING FROM ANOTHER MACHINE (SSH TUNNEL)
---------------------------------------------
The interface binds to 127.0.0.1 only and is not accessible from
the network directly. To access it from another machine:

  ssh -L 9090:127.0.0.1:9090 user@YOUR_JRSOCTRIAGE_HOST

Then open http://127.0.0.1:9090 in your browser on the remote machine.
The tunnel stays open as long as the SSH session is active.

TABS
-----
Config         Core pipeline settings — processing, filtering, LLM
               endpoints, email, and file paths.
Hosts          Host inventory — names, OS, roles, VLANs, notes,
               and anonymization aliases.
Rules          Per-rule escalation control and rate limiting.
Anonymization  Master switches for identity masking, plus user,
               domain, and IP alias management.
Networks       CIDR ranges shown as network context in every LLM prompt.
Journal        Live log stream from jrsoctriage.service with filter
               and color coding.
Users          Authentication users for the web interface itself.
               Add and remove are GUI-based; password change is
               terminal-only in v1.
Lookup         Reverse lookup for anonymized aliases — search box
               plus four browsable tables (hosts, users, domains,
               IPs).
Maintenance    Put a host into maintenance mode (suppress non-external
               LLM escalations) for a fixed duration. Active hosts are
               listed with a countdown and a Clear button.
Restart        Restart jrsoctriage and capture startup output.
               Includes a Check Status button to view service
               status without restarting.

IMPORTANT: CHANGES REQUIRE RESTART
-------------------------------------
Changes saved through the interface take effect only after restarting
jrsoctriage. Use the Restart tab in the interface, or from the command line:

  sudo systemctl restart jrsoctriage

The Restart tab captures the first 60 lines of startup output so
you can confirm the pipeline started cleanly after your changes.

PATHS SECTION
--------------
The Paths card at the bottom of the Config tab controls where jrSOCtriage

============================================================
NTOPNG (optional — active flow enrichment)
============================================================

ntopng provides L7 application protocol detection for active flows.
Run it as a systemd service so it survives reboots and restarts
automatically.

IMPORTANT: The ntopng config file must be in a persistent location.
Do NOT use /tmp — it is cleared on reboot and will cause ntopng to
crash-loop on startup with "Config file not found".
Use /etc/ntopng/ instead.

STEP 1: Create the config file
--------------------------------
sudo nano /etc/ntopng/ntopng.conf

Example contents (adjust for your environment):
  --interface=YOUR_CAPTURE_INTERFACE
  --http-port=3001
  --http-address=YOUR_MANAGEMENT_IP
  --data-dir=/path/to/ntopng/data
  --local-networks=192.168.10.0/24,192.168.20.0/24
  --disable-ipv6

Notes:
  --interface       The interface Zeek/Suricata is monitoring (SPAN port)
  --http-address    Bind to your management IP, not 0.0.0.0
  --data-dir        Must be a persistent path — not /tmp
  --local-networks  Your internal network ranges for host classification

STEP 2: Create the systemd service unit
-----------------------------------------
sudo nano /etc/systemd/system/ntopng.service

---------- BEGIN ntopng.service ----------
[Unit]
Description=ntopng Network Traffic Monitor
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
ExecStart=/usr/sbin/ntopng /etc/ntopng/ntopng.conf
Restart=always
RestartSec=5
User=root
LimitNOFILE=65536

[Install]
WantedBy=multi-user.target
---------- END ntopng.service ----------

STEP 3: Enable and start
--------------------------
sudo systemctl daemon-reload
sudo systemctl enable ntopng
sudo systemctl start ntopng

STEP 4: Verify
----------------
sudo systemctl status ntopng
curl -s "http://YOUR_MANAGEMENT_IP:3001/lua/rest/v2/get/ntopng/interfaces.lua" -u admin:PASSWORD

Web interface: http://YOUR_MANAGEMENT_IP:3001

If ntopng is not running, jrSOCtriage continues normally —
ntopng enrichment is silently skipped. If ntopng was down when
jrSOCtriage started, restart jrSOCtriage after ntopng is back up.

A NOTE ON ntopng "no data" ERRORS IN THE LOG
--------------------------------------------
Newer ntopng versions signal "I have no data for this host" by
closing the connection instead of returning an empty HTTP 200. In
the journal this surfaces as:
  ntopng_fetch: ntopng request error: ('Connection aborted.',
  RemoteDisconnected('Remote end closed connection without response'))
This is expected and harmless. It occurs on lookups for hosts ntopng
has no active-flow data for — typically external IPs. The alert
completes normally; only the ntopng portion of that one host's
enrichment is empty (which is correct — there is no data). You can
confirm it is not a network problem: Graylog and Zeek query the same
ntopng host in the same cycle and are unaffected. The error rate
tracks how many external/unknown hosts are being looked up, so it
rises during load tests and busy periods.


============================================================
VERIFYING THE PIPELINE IS WORKING
============================================================

1. Check service is running:
   sudo systemctl status jrsoctriage  (Or navigate to the Restart tab in
   the interface and click Check Status.)

2. Watch live log output:
   sudo journalctl -u jrsoctriage -f  (Or navigate to the Journal tab in
   the interface and click Start Stream.)

3. Confirm alerts are being read (look for):
   [INFO] ingest: Read N alerts at or above level 0  (Navigate to the Journal
   tab in the interface and type ingest into the filter field.)

4. Confirm Graylog shipping (look for):
   [INFO] gelf_shipper: Shipped to Graylog: RULE|HOST (Navigate to the Journal
   tab in the interface and type 'gelf-shipper' into the filter field.)

5. Confirm LLM triage (look for):
   [INFO] __main__: Triage: RULE|HOST | [LEVEL] VERDICT / CONFIDENCE

   NOTE: Depending on your config.json settings and network activity,
   it may take a while before the first LLM triage appears. Alerts
   must meet min_rule_level (default 6) OR trigger an escalation
   override (abuse score, baseline spike, always_include network,
   first-seen rule, or rules.json force_escalate_if) before reaching
   the LLM. On a quiet network or with a high threshold, this could
   take minutes or hours. This is normal — check that alerts are being
   seen and shipped (steps 3 and 4) to confirm the pipeline is healthy.

6. Understanding the stats output (logged every 10 cycles):
   Stats | seen=525 deduped=354 shipped=171 triaged=1 emailed=0 | llm_rpm=0.4

   seen    = total raw alerts read from Wazuh alerts.json
   deduped = alerts suppressed by dedup silence window — not lost,
             just collapsed because the same rule+host was seen
             recently. Normal to be high on noisy environments.
   shipped = unique alerts enriched and sent to Graylog
   triaged = alerts reviewed by the LLM
   emailed = NOTIFY verdicts sent to the analyst
   llm_rpm = LLM requests per minute, rolling rate since the last
             stats line. Useful for spotting LLM saturation: a
             healthy pipeline produces a steady rate; a value
             stuck near zero while triaged keeps climbing means
             alerts are queueing.

7. Confirm ntopng is connected (look for at startup):
   [INFO] ntopng_fetch: ntopng login successful

8. Check Graylog for incoming messages:
   Search: gl2_triage_complete:"true"
   Should show recent triaged alerts with verdict fields populated.

9. Watch for FAILED verdicts:
   Search Graylog for: gl2_llm_verdict:"FAILED"

   FAILED is the synthetic verdict the pipeline emits when ALL
   configured LLM endpoints fail for a single alert. It is not
   a parsing failure or a content filter — it means the pipeline
   tried every endpoint in your strategy and got no usable
   response from any of them.

   RECOMMENDATION: configure a cloud API endpoint as failover.
   This is not an optional nicety — the economics are decisive.
   This lab's production numbers: local-primary (gemma4:26b on an
   RTX 3090) averages ~10 alerts/day that the local model cannot
   close within the request timeout. With failover configured,
   each of those goes to gpt-4o-mini at roughly $0.0007 per call
   — well under a penny a day for a clean verdict on exactly the
   alerts that most benefit from a smarter model. Without
   failover, those same alerts ship as FAILED and an analyst has
   to triage them by hand, which costs more than a penny.

   Expected rates from production observation:
     - With cloud fallback configured (this lab: gemma4:26b →
       gpt-4o-mini): FAILED is vanishingly rare. The failures
       concentrate on genuinely-hard alerts where the local model
       cannot reach a confident verdict within the request timeout
       — by definition the alerts that most benefit from a smarter
       cloud model. The cloud model handles those in 4-6 seconds
       with a clean verdict.
     - Single-endpoint local only: those ~10/day (at this lab's
       volume; scale with yours) ship as FAILED instead.

   Finding alerts that escalated to cloud fallover:
   With fallover configured, alerts where the local model failed
   but the cloud model succeeded ship to Graylog with the cloud
   model as the answering endpoint, not as FAILED. To find these
   events, search by the cloud model name rather than by verdict:

     gl2_llm_model:<cloud_model_name>

   For example, if your fallover is configured as GPT-4.1 mini:

     gl2_llm_model:gpt\-4.1\-mini

   (backslash-escape hyphens in the model name for Lucene query
   syntax). This returns every alert where local hit timeout and
   cloud took over with a clean verdict. Pair with a search for
   gl2_llm_verdict:"FAILED" over the same window to get the full
   picture: cloud-handled escalations plus any alerts where cloud
   also failed (rare).

   Set a Graylog alert on the FAILED rate climbing significantly
   above your established baseline. A sudden spike usually
   indicates an LLM endpoint health issue (Ollama down, network
   blip to Fedora host, cloud API outage, expired API key) rather
   than a pipeline problem. The pipeline keeps running and
   producing FAILED records during the issue, so the alert tells
   you what to investigate without losing alert audit trail.

10. Confirm the threat-intelligence sources (v1.1) are producing:
    Each enabled source ships a flat Graylog field on qualifying
    alerts. Coverage checks:

      _exists_:gl2_kev_listed          (any CVE-bearing alert)
      _exists_:gl2_epss_max            (any CVE-bearing alert)
      _exists_:gl2_vt_malicious        (any hash-bearing or
                                        external-IP alert)
      _exists_:gl2_otx_pulses          (same)
      _exists_:gl2_greynoise_class AND NOT gl2_greynoise_class:"N\/A"
                                       (any external-IP alert)

    Hash-bearing FIM alerts (file or registry-value changes) fire
    the VT and OTX hash paths on ordinary daily traffic. The
    IP-path fields only populate on alerts with an external party
    IP, which a well-tuned network may produce rarely — absence of
    IP-path results on a quiet network is not a fault. See
    graylog_searches.txt, THREAT INTELLIGENCE SEARCHES, for the
    full field semantics (including which fields are
    numeric-when-present vs always-present strings).


============================================================
THREAT-INTELLIGENCE ENRICHMENT — OPERATIONAL NOTES (v1.1)
============================================================

The five v1.1 sources (GreyNoise, CISA KEV, EPSS, VirusTotal,
AlienVault OTX) share one operational contract. Knowing it makes
the journal and the shipped records self-explanatory.

DEGRADED LOOKUPS FAIL LOUD, AND NEVER CACHE:

A rate-limited lookup (HTTP 429) annotates the record RATE_LIMITED
— a first-class, searchable state, never a silent N/A. The prompt
tells the LLM the lookup could not be made and to treat the status
as unknown. Degraded records are NEVER cached, so the very next
alert re-asks the moment quota recovers; recovery is automatic and
needs no operator action. The design rationale: hitting a
reputation-source rate limit during normal operation often means
elevated external-IP volume, which can itself be the attack signal
— silently skipping reputation lookups at that exact moment is the
wrong behavior.

Per-source rate-limit WARNINGs in the journal are controlled by
each source's rate_limit_warnings toggle. Records are annotated
either way; the toggle only governs journal noise. Leave warnings
ON for keyed production sources; turn OFF where quota bounces are
routine (keyless OTX, free-key VirusTotal).

AUTH FAILURES (where they are observable):

  - VirusTotal: a rejected key (HTTP 401) logs a WARNING once per
    process and annotates affected lookups AUTH_FAILED — a
    configuration problem, distinguishable from both "clean" and
    "rate limited." Fix the key; no restart-loop of errors.
  - OTX: key validity is NOT observable — the API returns the same
    public data with a valid key, a bad key, or no key. A mistyped
    key silently behaves as keyless. Verify at otx.alienvault.com.
  - GreyNoise/KEV/EPSS: KEV and EPSS have no keys. GreyNoise
    keyless is a supported mode, not an error.

CISA KEV CATALOG LIFECYCLE:

The KEV catalog is fetched in the background and cached locally
with a 24h refresh. No alert ever blocks on the download. If the
CISA feed is unreachable, the cached catalog keeps serving with a
visible STALE annotation once it ages past 48h, and UNAVAILABLE
when no catalog has ever been fetched. Both states ship in
gl2_kev_listed, so verdicts made without fresh KEV context are
searchable: gl2_kev_listed:STALE OR gl2_kev_listed:UNAVAILABLE.

QUOTA PROFILES (free tiers):

  Source      Free-tier budget          Notes
  ----------- ------------------------- ---------------------------
  GreyNoise   ~10 lookups/day keyless   Keyed (commercial/trial)
                                        for real deployments
  CISA KEV    unlimited                 one catalog download / 24h
  EPSS        effectively unlimited     batched per alert
  VirusTotal  4/min, 500/day            NON-COMMERCIAL license on
                                        free keys; per_alert_cap
                                        protects the daily budget
  OTX         generous; keyless runs    key raises the ceiling
              at a lower ceiling

  VirusTotal is the tightest budget in the set and the only one
  where a busy FIM day can visibly exhaust quota — expect
  RATE_LIMITED annotations on VT lines during registry/file change
  storms on a free key. This is the cap and the fail-loud contract
  working as designed, not a fault.

PARTY vs MENTIONED IP FIELDS:

gl2_greynoise_class and gl2_abuse_score describe IPs that are
PARTIES to the alert (structured source/destination fields).
External IPs that merely appear in the alert's raw log text ship
separate worst-of fields — gl2_mentioned_greynoise_worst and
gl2_mentioned_abuse_max — so a domain controller alert whose log
contains a resolved malicious IP never pollutes the party fields.
On hash-bearing alerts (and only those), VT and OTX also run on
mentioned external IPs — IP reputation corroborating hash intel on
the same record — rendering on the mentioned line and shipping
gl2_mentioned_vt_malicious / gl2_mentioned_otx_pulses. Hash-less
alerts spend no VT/OTX quota on mentioned IPs.
See graylog_searches.txt for the search patterns.

ENRICHMENT LOOKUPS IN YOUR OWN NETWORK EVIDENCE:

The pipeline's own enrichment traffic is visible to your sensors:
DNS queries to www.virustotal.com, otx.alienvault.com, and the
other source endpoints originate from the jrSOCtriage host and
will appear in Zeek/DNS context windows that overlap enrichment
activity. If your host inventory notes explain ip-api.com DNS
noise, extend the same note to the v1.1 source domains so the LLM
never reads the pipeline's own lookups as a finding.

============================================================
STORM MANAGEMENT
============================================================

A "storm" is a high-volume burst of alerts that causes the pipeline to
fall behind real-time processing. Storms are expected behavior in real 
environments, not system failure. Common causes:
  - apt upgrade / package installs on monitored hosts
  - Wazuh vulnerability scanner running (fires one alert per CVE per package)
  - New software installed — FIM baseline has no history for new files
  - System configuration changes — many files modified at once
  - First time a host is seen — no baseline, everything is "new"

Signs you have a storm:
  - Pipeline stats show deduped >> shipped (many unique rule+host combos)
  - Email flood — dozens of NOTIFYs in minutes
  - journalctl shows pipeline cycling faster than LLM can respond
  - Graylog shows large volume of alerts from one host in a short window

------------------------------------------------------------
OPTION 1: MAINTENANCE MODE (planned changes)
------------------------------------------------------------

If you know you are about to make system changes that will generate
storms (apt upgrade, new software install, major config change), put
the host in maintenance mode BEFORE making changes.

Maintenance mode suppresses LLM escalation for non-external alerts
on that host for a set duration. All alerts still ship to Graylog.
External IP alerts still escalate normally — security is maintained.

The pipeline picks up the maintenance mode flag on the next poll cycle
(within 30 seconds, controlled by the poll_interval config setting).
No restart required.

There are two ways to manage maintenance mode: the web interface (the
recommended path for interactive use) and the command-line tool (for
automation, scripts, and remote shells).

GUI workflow (web interface, recommended):

  1. Open the interface and click the Maintenance tab.
  2. Pick a host from the dropdown (populated from hosts.json).
  3. Pick a duration (15 / 30 / 60 / 120 / 240 / 480 minutes).
  4. Click Set Maintenance.

  The "Currently In Maintenance" card above shows all active entries
  with a countdown and a Clear button per row. The list refreshes
  every 30 seconds while the tab is open. Clicking Clear prompts
  for confirmation before removing the entry.

  Maintenance set/clear actions are logged with the authenticated
  username (set_by="user:<username>") for audit trail.

CLI workflow (terminal, for automation):

  Set maintenance mode:
    cd /path/to/jrsoctriage
    sudo python3 maintenance.py --host HOSTNAME --minutes 60

  Check maintenance mode status:
    sudo python3 maintenance.py --status

  Clear maintenance mode early:
    sudo python3 maintenance.py --host HOSTNAME --clear

  Example — putting HOSTNAME in maintenance mode for 2 hours:
    sudo python3 maintenance.py --host HOSTNAME --minutes 120

  CLI invocations record set_by="manual" in the database. This is
  useful for cron jobs, Ansible playbooks, or pre-upgrade hooks
  where no interactive user is available.

Both paths write to the same SQLite table; the GUI is a thin shell
over the same logic the CLI runs. You can set via GUI and clear via
CLI (or vice-versa) without issues.

------------------------------------------------------------
OPTION 2: RULES.JSON TUNING (recurring noisy rules)
------------------------------------------------------------

For rules that repeatedly generate storms, add them to rules.json
with rate limiting and extended dedup silence. See rules_instructions.txt
for full documentation.

Common storm rules to tune:
  2902  — dpkg package installed
  2904  — dpkg package upgraded/removed
  23504 — Wazuh vulnerability detector
  23505 — Wazuh vulnerability detector
  23506 — Wazuh vulnerability detector
  550   — FIM integrity check changed
  533   — netstat port change
  752   — file/registry entry added

A starter set of rule overrides covering the rule IDs above ships
with the repo as rules.json.kevin. To use it as a starting point:

  cp rules.json.kevin rules.json

Then restart. Review each entry against your own Wazuh ruleset
before relying on it — see rules_instructions.txt for the full
discussion. The pipeline does not reference rules.json.kevin
directly; the active file is always rules.json.

NOTE: Never add never_escalate or external_ip-only conditions to
vulnerability, authentication, or privilege escalation rules.
See rules_instructions.txt SECURITY GUIDANCE section.

------------------------------------------------------------
OPTION 3: GLOBAL DEDUP SLEDGEHAMMER
------------------------------------------------------------

Raise dedup_silence_seconds in config.json temporarily during a storm.
Default is 240 seconds (4 minutes). Raising to 3600 (1 hour) means
each rule+host combination can only fire once per hour regardless of
how many times Wazuh generates the alert.

This is a blunt instrument — use rules.json for surgical control.

------------------------------------------------------------
THROUGHPUT AND CONCURRENCY MODEL
------------------------------------------------------------

The pipeline scales horizontally for LLM calls (configurable workers)
and is non-blocking for most enrichment work (DNS via dnspython,
external-IP enrichment via per-IP cache). The remaining bottleneck
is LLM call latency, not Python concurrency: each gemma4:26b call on
local hardware takes 7-20 seconds depending on prompt size, and each
worker handles one call at a time.

A practical sizing example, anchored on local Ollama:
  - Single worker: ~3-9 LLM-triaged alerts/minute sustained
  - 4 workers:     ~12-36 LLM-triaged alerts/minute sustained
  - Storm bursts above the worker rate queue up; the pipeline catches
    up once the rate drops below capacity. Most home labs see steady-
    state under 1/min once dedup, rate limits, and never_escalate
    rules are tuned, so a single worker is typically enough.

Tuning levers:
  - llm.endpoints[].max_concurrent — workers per endpoint
  - filtering.min_rule_level — coarse gate before any LLM work
  - rules.json never_escalate / max_escalations_per_hour — surgical
    suppression of known-noisy rules. Without this, a major system
    change on any monitored host can cause significant lag and email
    flooding. Plan to tune rules.json for your environment before
    deploying to production.

What's intentionally still single-threaded:
  - SQLite database access — single writer per process is correct for
    SQLite; multi-writer would need WAL + BEGIN IMMEDIATE coordination
    and isn't worth the complexity at v1.0 scale. This is a design
    choice, not a capacity constraint: v1.0's per-alert write profile
    is already minimal (one fsync per dedup-passing alert), and the
    measured disk-bound ceiling on reference hardware is orders of
    magnitude above any realistic deployment. v2.0 swaps cold
    history (verdicts, audit log) to PostgreSQL.
  - alerts.json position tracking — single ingester reads the Wazuh
    file sequentially. No fan-out at this layer, by design.

Neither is a runtime bottleneck at typical scales. The numbers above
are LLM-bound, not database-bound or ingest-bound.

------------------------------------------------------------
SCALING — WHAT'S MEASURED AND WHAT'S ESTIMATED
------------------------------------------------------------

These tests establish what the pipeline handles comfortably, NOT
where it breaks. The breaking point has not been measured — the
goal was to confirm the pipeline works at high volume, not to push
it to failure. The actual ceiling is above the tested volumes, but
how far above is unknown.

For context: the original design target was roughly 900k alerts/day.
Sustained load testing has since cleared that by more than 30×. The
pipeline ran clean at input rates up to ~29.8M alerts/day in
level-6-plus-filters mode and to ~6.8–7.2M alerts/day in
escalate-everything mode, with zero ingest loss and zero abandoned
work at every rate. The level-6 figure is not a measured ceiling: the
test load is this project's real, organic alert volume multiplied
across synthetic hosts, so ~29.8M/day is simply the most that
multiplication ever produced in a test hour — the pipeline processed
all of it cleanly, so the true limit is higher. Testing stopped where
it did because the volumes had so thoroughly exceeded the 900k design
target that pushing higher at real API cost wasn't worth it.

Every load test this project runs uses the same 24-worker
configuration, so 24 workers is the standard, repeatedly-validated
test point and every measured anchor below rests on it. The detailed
per-worker measurements that the throughput model is built on come
from the May 2026 runs against GPT-4.1-mini:

  - Escalate-everything mode: ~2.7M raw alerts/day pace for ~62 min,
    24 workers, sustained ~154 LLM verdicts/min through the full
    pipeline, zero ingest loss, zero abandoned work.
  - Level-6-plus-filters mode: ~6.3M raw alerts/day pace for ~63 min,
    24 workers, 29,027 Stream 2 enrichments shipped, 773 LLM calls,
    zero ingest loss, zero abandoned work.

The recent level-6 capacity testing — also 24 workers — ran at far
higher ingest rates (up to ~29.8M raw alerts/day), and in the peak
hour completed ~11,100 LLM calls, about ~185 LLM calls/min sustained,
comfortably above the ~154 verdicts/min the earlier 24-worker run
demonstrated. So the 24-worker anchor is confirmed across multiple
independent runs, not a single test. (Note: ~185 calls/min was the
offered LLM load that hour — the volume that passed dedup and
filtering — not the workers running flat-out, so it reflects
sustained achieved throughput, not a per-worker capacity ceiling.)

Both test generations ran on on-disk SQLite on a single consumer NVMe
with no lock contention — the current pipeline has no database-lock
behavior to engineer around, so on-disk is the default (see the
RAM-disk section for the optional in-memory setup and when it helps).
See the WORKED EXAMPLE earlier in this document for full test results
and costs.

From those anchors, per-worker LLM throughput against GPT-4.1-mini
(~2-3 sec/call on this project's ~13-21k character prompts) works
out to roughly ~20 LLM calls/min/worker raw sustained. The
extrapolation across worker counts (these are LLM-CALLS-per-minute,
NOT alerts-per-minute — most alerts are dropped by dedup and
filtering before the LLM stage):

  - 1 worker:    ~20 LLM calls/min raw sustained (measured-backed)
  - 10 workers:  ~200 LLM calls/min raw sustained (measured-backed;
                 the 24-worker tests sustained ~154-185/min)
  - 50 workers:  ~1000 LLM calls/min raw (linear extrapolation,
                 not directly measured)

Beyond 50 workers is untested — no numbers are quoted for it.
The worker-sizing math applies a 40% headroom multiplier on top
of these raw rates; provision to the headroom figure, not the raw
rate.

For a slower endpoint such as Claude Haiku (~4-6 sec/call, roughly
2× GPT-4.1-mini's response time), halve the per-worker numbers
above — see the conservative-floor figures in the cost-ceiling
section earlier in this document. Slower premium tiers (Opus,
GPT-5) at 6-10 sec/call go lower still. Size to your actual
endpoint's measured response time, not to a generic assumption.

Where the architecture would actually break under sustained load:
  1. AlertJSON ingest serialization — single ingester reads the Wazuh
     file sequentially. No fan-out at this layer, by design. At very
     high inbound rates this is the binding constraint.
  2. Per-worker enrichment overhead — Python GIL and per-alert
     enrichment cost (Graylog, Zeek, ntopng, AbuseIPDB) limit
     per-worker throughput. Adding workers helps until upstream
     APIs become the bottleneck.
  3. Upstream API stress — Graylog and ntopng fetches scale linearly
     with worker count. Both are configurable but have their own
     limits.
  4. AbuseIPDB rate limits — free tier is 1000 lookups/day. The
     enrich.py per-IP cache mitigates repeat lookups, but burst
     traffic from many distinct IPs can exhaust quota fast.
  5. Cloud LLM API rate limits — provider-specific, but typically
     in the hundreds of RPM for paid tiers. Haiku is generous; Opus
     and other premium tiers are tighter.

SQLite is deliberately not on this list as a throughput limit, and
on-disk storage is not a contention limit either. The disk-bound
theoretical ceiling on reference hardware (consumer NVMe at ~5,200
fdatasync ops/sec, ~1 fsync per dedup-passing alert) is on the
order of hundreds of millions of alerts per day. The practical
question was settled by direct test: on-disk single-disk SQLite
ran clean to ~30.9M raw alerts/day with zero lock contention on a
fully shared disk (high-end Gen4 consumer NVMe), no ceiling
reached. The RAM-disk option (see "HOSTING jrsoc.db IN RAM"
earlier) is now an optional wear/offload convenience, not a
contention fix.

For most home-lab and small-team deployments, none of these
ceilings is the binding constraint. Cost is — see "DON'T REACH FOR
FRONTIER MODELS" above for the cost/quality tradeoff.

The worker-count extrapolations above (the 50-worker row in
particular) will be replaced with directly measured numbers as
future load-harness work across more worker counts continues. The
local-LLM numbers earlier in this section and the load-test results
above are measured; the per-worker extrapolations to untested worker
counts are estimates.

============================================================
DEDUP_SILENCE_SECONDS — THE DOMINANT TUNING VARIABLE
============================================================

What deduplication does: when the same Wazuh rule fires repeatedly
on the same host in a short span — the same login failure, the same
file-integrity event, the same scanner hit, over and over —
jrSOCtriage does not treat each repeat as a new thing to enrich and
send to the LLM. Instead, the first occurrence opens a dedup
"window," and every subsequent matching alert (same rule ID + same
host) that arrives while the window is open is folded into that one
record rather than processed separately. When the window closes,
that single deduplicated record carries a count of how many raw
alerts it represented. The effect is that a burst of 500 identical
alerts becomes one LLM call and one operator-visible record, while
still preserving the fact that it fired 500 times. This is what
keeps repeat-fire noise (which is most of the raw volume in a
typical environment) from flooding the LLM stage and the operator.

dedup_silence_seconds is the length of that window — how long
jrSOCtriage keeps folding new same-rule+same-host alerts into an
existing record before it considers the burst "over."

After Wazuh-side filtering, dedup_silence_seconds is the single
most powerful knob for controlling jrSOCtriage's LLM call volume.
A larger value collapses more high-frequency repeat-fire patterns
into single records; a smaller value lets repeats through.

This section explains the principle, the measured impact in this
lab, how to choose your value, and the tradeoffs.

------------------------------------------------------------
WHY DEDUP IS THE DOMINANT LEVER
------------------------------------------------------------

Three layers reduce raw Wazuh alert volume before the LLM sees
anything:

  1. Wazuh-side filtering (min_rule_level, agent groups, rule
     exclusions). This happens outside jrSOCtriage, in your Wazuh
     configuration. For most SMB deployments this is the biggest
     cut — your Wazuh manager is already configured not to ship
     informational chatter to jrSOCtriage in the first place.

  2. dedup_silence_seconds (this knob). Same-rule+same-host events
     within the window collapse into one record. This is the
     dominant lever for repeat-fire patterns: domain controller
     login storms, AppArmor Docker audit events, port-scan-induced
     Suricata cascades, vulnerability-scanner CVE-per-package
     bursts. These patterns generate dozens to hundreds of raw
     alerts per minute; dedup collapses them into one record per
     window.

  3. rules.json pre-LLM filtering (min_rule_level on the
     jrSOCtriage side, plus per-rule suppression). This trims the
     long tail of low-value rules pre-LLM. It's surgical: per-rule,
     per-host, with explicit operator intent.

Among these three, dedup is the dominant lever for *operational*
volume control because it operates on the largest-fan-out patterns.
A single domain-controller bug can generate 10,000 raw alerts in
an hour. Wazuh-side filtering rarely catches this (the rule is
legitimately important, just over-firing). rules.json suppression
is too coarse to apply (you don't want to silence the rule
entirely). Dedup is exactly the right tool: collapse the burst
without losing the signal that the burst happened.

------------------------------------------------------------
WHAT DEDUP LOOKS LIKE IN THE JOURNAL
------------------------------------------------------------

Every ingest cycle emits a [DEDUP] line summarizing what happened
during the just-closed cycle:

  [DEDUP] period_s=5.0 ingest=250 dedup=240 stream2=10 rate_pct=96.0
  [DEDUP_NOTE] Dedup is retrospective of last ingest cycle.

Reading the line:
  period_s     The cycle's duration in seconds.
  ingest       How many alerts were read from Wazuh during the cycle.
  dedup        How many of those were dropped as duplicates.
  stream2      How many proceeded to full pipeline processing
               (Stream 2: enrichment, rule evaluation, LLM if enabled).
  rate_pct     dedup / ingest, expressed as a percentage.

The [DEDUP_NOTE] line that follows every [DEDUP] line says
explicitly: this is RETROSPECTIVE. The alerts counted on the
[DEDUP] line have already been handled (either shipped or dropped)
by the time the line is emitted. The line is not a snapshot of
pending work. It is a report on the cycle that just finished.

This semantics matters because misreading it as forward-looking
("there are 240 deduped alerts pending") drove a multi-day
investigation in this project's history. The [DEDUP_NOTE] reminder
exists to prevent the next operator from repeating that misread.

------------------------------------------------------------
QUIET PERIODS THAT LOOK LIKE STALLS
------------------------------------------------------------

Operators new to jrSOCtriage often see what looks like the pipeline
freezing — long stretches with no Triage lines, no Stream 2
shipping, no LLM activity. This is almost never a stall. Two
patterns produce it:

  1. Repeat-fire dedup periods line up. Each repeating alert
     source (a busy domain controller, an AppArmor Docker audit
     cascade, a Suricata port-scan response, a vulnerability
     scanner) has its own dedup window — once jrSOCtriage sees
     the first alert of a given rule+host, it suppresses the
     rest until that window expires (dedup_silence_seconds,
     default 240s). When the active windows from multiple
     repeating sources line up, every incoming alert maps to
     one of those already-active windows, and the dedup rate
     sits at 100% until at least one window expires.

     The alignment isn't deliberate; it just happens. Some
     burst types tend to fire on similar cadences (scheduled
     tasks, periodic scans, hourly health checks), which makes
     their windows likely to overlap. Once enough sources are
     overlapping, the pipeline can sit at 100% dedup for the
     full 240s. Nothing reaches Stream 2 during that stretch.
     No Triage lines, no LLM activity. Looks like a stall.

     This is most visible on smaller or more homogeneous networks
     where a handful of busy hosts dominate the traffic mix. This
     project's home-lab test network exhibits it constantly.
     Larger networks with more diverse traffic sources still see
     it but the periods tend to be shorter because new rule+host
     combinations break through more often.

  2. Genuinely quiet periods. Less commonly, the network simply
     has nothing happening worth alerting on. More common overnight,
     on weekends, or during holidays — scheduled tasks pause, user
     activity drops, the alert stream genuinely thins out.

How to tell which one you're looking at: check the [DEDUP] lines
during the quiet stretch.

  - ingest > 0 with rate_pct close to 100 over consecutive cycles →
    Pattern 1. Alerts are arriving continuously and being
    deduplicated. The pipeline is working correctly; the noise
    just happens to be highly repetitive in this window.

  - ingest close to 0 across consecutive cycles → Pattern 2. The
    network is actually quiet. The pipeline is working correctly;
    there is simply nothing to process.

  - rate_pct=-1.0 → cycle had zero ingest, no rate to compute.
    Same as Pattern 2.

What is NOT happening: the pipeline is not stuck. The watchdog's
abandoned_total counter (visible in [WATCHDOG_HEARTBEAT] lines) is
the trustworthy "is the pipeline actually stalling?" signal. If
abandoned_total=0, no work has been abandoned regardless of how
quiet the surrounding logs look. A pipeline that emits
[WATCHDOG_HEARTBEAT] consistently and completes clean shutdowns is
a working pipeline, not a stalled one.

------------------------------------------------------------
MEASURED IMPACT IN THIS LAB
------------------------------------------------------------

This project's home lab measured the dedup window's impact
directly during development:

  - dedup_silence_seconds=30 (early development):
    ~30,000 raw alerts/day → ~4,000 LLM calls/day in
    escalate-everything mode.
    Dedup collapse ratio: ~7.5x

  - dedup_silence_seconds=240 (current production):
    ~30,000 raw alerts/day → ~2,400 LLM calls/day in
    escalate-everything mode.
    Dedup collapse ratio: ~12.5x

Widening from 30s to 240s reduced LLM call volume by ~40% with
no loss of operationally-meaningful coverage. The additional
collapses happened on high-frequency repeat-fire patterns that
were not surfacing new information on each fire — DC logon storms
when an account-policy change rippled through, AppArmor Docker
audit events at the 10-second container audit interval, port
scanner cascades hitting many ports on a single host within
seconds. None of those were producing operationally distinct
alerts that the longer window suppressed; they were all variations
of "this thing is happening, you saw it the first time."

Cost implications are proportional. At measured GPT-4.1-mini
pricing ($0.0023/call against this project's full-fat prompts):
  - 30s dedup, escalate-everything (~4,000 LLM/day): ~$3,400/year cloud-only
  - 240s dedup, escalate-everything (~2,400 LLM/day): ~$2,000/year cloud-only

Same coverage, 40% lower cloud bill. dedup_silence_seconds is one
of the only tuning variables in the pipeline where you genuinely
get something for nothing within the validated range.

------------------------------------------------------------
HOW TO CHOOSE YOUR VALUE
------------------------------------------------------------

Defaults:
  - 240 seconds (4 minutes) is this project's current production
    value. Recommended starting point.

Lower (60-180 seconds):
  - Choose if your environment has genuinely time-sensitive
    repeat-fire signals you want to see twice in a 5-minute
    window. Rare in practice — most repeat-fires are noise.
  - Consider if you have very low alert volume and want to err
    toward seeing more.

Higher (300-900 seconds):
  - Choose if your home lab consistently sees alert storms that
    blow past the 240s window with the same rule+host combination.
  - SMB deployments at higher raw alert volumes often benefit
    from 600-900 seconds for the same coverage with less cost.
  - Above 900 seconds, you start losing operationally meaningful
    second-fire visibility. Stop here unless you have a specific
    reason.

Above 3600 seconds (1 hour):
  - Effectively per-rule, per-host hourly cap. Use only as a
    storm-response sledgehammer (see STORM MANAGEMENT above), not
    as a steady-state value. Real second-fires within an hour
    will be lost.

------------------------------------------------------------
DIAGNOSTIC: IS YOUR DEDUP WINDOW RIGHT?
------------------------------------------------------------

Pull your measured dedup ratio (raw alerts vs LLM calls) after the
pipeline has been running for at least 24 hours:

  RAW=$(sudo journalctl -u jrsoctriage --since "24 hours ago" | \
        grep "ingest: Read" | \
        awk '{sum+=$8} END {print sum}')
  ESC=$(sudo journalctl -u jrsoctriage --since "24 hours ago" | \
        grep "Triage:" | wc -l)
  echo "Raw alerts ingested: $RAW"
  echo "LLM-triaged:          $ESC"
  [ "$ESC" -gt 0 ] && echo "Dedup ratio: $(echo "scale=2; $RAW / $ESC" | bc)"

Interpretation:

  - Ratio under 5: dedup window is too narrow OR your environment
    genuinely has few repeat-fire patterns. Sample what's reaching
    the LLM. If you see the same rule+host firing 10+ times per
    hour, widen the window.

  - Ratio 8-15: typical for an environment with moderate repeat-
    fire activity (this project's lab lands here at 12.5).

  - Ratio above 25: dedup may be collapsing legitimately distinct
    events. Sample what's getting dropped — pull pre-dedup alerts
    for a 5-minute window from Wazuh and confirm none represent
    operationally distinct conditions. If they do, narrow the
    window.

  - Ratio sensitivity: the ratio changes much more with dedup
    window changes than with rules.json changes. dedup is the
    primary axis; rules.json is for surgical follow-up.

------------------------------------------------------------
TRADEOFFS AND WHEN TO NARROW
------------------------------------------------------------

A wider dedup window is not always better. Specific signals that
the window may be too wide:

  - You notice in Graylog that a host went down and came back up
    multiple times, but only the first event triggered an alert.
    Window may be too wide for hosts where state-change visibility
    matters per occurrence.

  - A security-relevant rule (lateral movement attempt, privilege
    escalation, repeated authentication failure) is firing multiple
    times in production but jrSOCtriage's verdict count shows one
    record. Verify the dedup window isn't hiding sequential attack
    attempts that should each be evaluated.

  - You're seeing rule_id collisions in dedup state (rare but
    possible if rule IDs are reused across event types). Verify
    the dedup key is (rule_id, canonical_hostname) and that's
    actually disambiguating in your environment.

Most operators tune dedup once, validate the ratio, and don't
revisit. The default 240 has held up across approximately one
month of continuous production operation in this lab with no
observed coverage gaps. SMB operators at higher raw alert volumes
should consider 600 as their starting point and measure from there.

============================================================
MAX_BATCH_SIZE — THE INGEST CEILING NOBODY THINKS ABOUT
============================================================

dedup_silence_seconds controls how much LLM work the pipeline
produces. max_batch_size controls how much alert work the pipeline
can absorb in the first place. They tune different stages, and at
SMB scale max_batch_size becomes a real constraint that's easy to
miss because it fails silently.

If R raw alerts arrive per second and the pipeline polls alerts.json
every poll_interval_seconds with max_batch_size alerts per read,
the pipeline keeps up as long as:

  max_batch_size ≥ R × poll_interval_seconds

If that inequality breaks, the pipeline reads the first
max_batch_size alerts per poll cycle and leaves the rest. The
position pointer advances by max_batch_size. The unread alerts
wait until the next poll. If raw arrival rate continues to exceed
batch-per-interval capacity, backlog grows linearly forever.

There is no error log saying "we fell behind." There is no exception.
The pipeline keeps running and produces verdicts on the alerts it
sees — those verdicts are simply increasingly stale relative to
real-time Wazuh state. End-to-end latency drifts upward, and the
gap between "raw alerts ingested" (from journald) and "alerts in
Wazuh's alerts.json" (from the file size or Wazuh dashboard) widens
over time.

This is the silent failure mode you have to think about explicitly,
because the symptoms look like generic slowness rather than a
specific bottleneck.

------------------------------------------------------------
SIZING max_batch_size FOR YOUR DEPLOYMENT
------------------------------------------------------------

Step 1 — measure your steady-state raw alert rate:

  Raw alerts per day ÷ 86,400 = raw alerts per second (R)
  (Get the daily figure from Wazuh's dashboard or ingest logs)

  steady_state_floor = R × poll_interval_seconds

This floor is the bare minimum to keep up with a constant arrival
rate. Do NOT size to it — real arrival is bursty, and the bursts,
not the average, are what cause drops.

Step 2 — size well above the floor. This lab's measured experience
is the guide here: at ~0.35 alerts/sec sustained (steady-state
floor of ~10 alerts for a 30-second poll), a batch size of 100
(≈10× the floor) STILL dropped 8% of alerts to bursts. Only at
250 (≈25× the floor) did the drop rate reach 0%. The typical rate
badly under-predicts the burst peaks: ~9 alerts in a typical
30-second window, but the peaks forced a batch size of 250 to
avoid any loss.

So the operational guidance is **size to roughly 25× the
steady-state floor as a starting point**, not some smaller
"headroom factor." A 4× or even 10× multiplier is not enough —
the lab measured drops at 10×. 25× is what produced zero drops
here; bursty environments may need more.

Step 3 — verify with the drop-rate diagnostic (below). The formula
gets you a starting value; the journal tells you the truth. Raise
max_batch_size (or shorten poll_interval_seconds) until the
"Batch limit reached" lines stop appearing under your real
traffic. That measured zero-drop point is your correct value —
there is no constant-rate formula that reliably predicts it,
because it is driven by your environment's burst behavior, not
its average.

------------------------------------------------------------
WORKED EXAMPLES
------------------------------------------------------------

Home lab (this project's actual configuration):
  R = 30,000 raw alerts/day = 0.35 alerts/sec
  poll_interval_seconds = 30
  steady-state floor = 0.35 × 30 = ~10 alerts
  CURRENT DEFAULT (250) is 25x over steady-state — measured 0% drop
  rate in production. This is the validated operational config.
  For reference: max_batch_size=100 (10x steady-state) was tested
  and showed 8% drop rate at home-lab burst patterns — measurement
  that established the 25x guidance.

SMB (~10× home lab raw volume, 300,000 raw alerts/day):
  R = 300,000 / 86,400 = ~3.5 alerts/sec
  poll_interval_seconds = 30
  steady-state floor = 3.5 × 30 = ~105 alerts
  25x headroom target = ~2,625 alerts
  CURRENT DEFAULT (250) is only ~2.4x the steady-state floor — far
  below the 10x ratio that already produced 8% drops at home-lab
  scale. Expect significantly worse drop rates at SMB volume with
  the default — likely 30-50%+ depending on burst patterns. Raise
  max_batch_size to 2,000-3,000 to hit the same 25x headroom that
  achieves 0% drops in this lab, OR shorten poll_interval_seconds
  to 5-10 seconds. Both work; the choice depends on how bursty
  your environment is (see below).

Large SMB (~30× home lab raw volume, ~900,000 raw alerts/day):
  R = 900,000 / 86,400 = ~10 alerts/sec
  poll_interval_seconds = 30
  steady-state floor = 10 × 30 = ~300 alerts
  25x headroom target = ~7,500 alerts
  CURRENT DEFAULT (250) is BELOW the steady-state floor — pipeline
  will fall behind continuously, not just on bursts. Required:
  raise max_batch_size to 5,000-10,000 AND/OR shorten
  poll_interval_seconds to 5 seconds. This is roughly the original
  design-target volume, and sustained load testing has since run the
  full pipeline clean far above it — up to ~29.8M/day in level-6 mode
  and ~6.8–7.2M/day in escalate-everything mode — so the pipeline
  itself handles this scale comfortably; batch sizing and poll
  interval are the knobs that have to be tuned correctly to reach it.

Very high volume (100× home lab, ~3,000,000 raw alerts/day):
  R = ~35 alerts/sec
  steady-state floor = 35 × 5 = ~175 alerts at a 5-second poll
  In the May 2026 testing at roughly this pace, max_batch_size was
  set to 25,000 and zero alerts were dropped. That is the measured
  value, not an estimate — and it is far above what the steady-state
  math would suggest, which is exactly the point this whole section
  makes: burst behavior, not average rate, sets the required batch
  size. Later load testing ran the full pipeline clean well past this
  volume (up to ~29.8M/day in level-6 mode), so the pipeline handles
  this scale with room to spare. At this scale: set max_batch_size in
  the tens of thousands (25,000 was the tested zero-drop value here),
  shorten poll_interval_seconds to ~5 seconds, and provision LLM
  workers and endpoint rate limits to match. On-disk SQLite is fine at
  this volume (the database showed no contention even in a dedicated
  ~30.9M/day stress test); the optional in-memory setup in the RAM-disk
  section can lower disk wear further if you want it. The v2.0
  PostgreSQL roadmap makes very large multi-tenant installs lighter to
  operate, but v1.0 is not ruled out at this volume — it was tested
  there.

------------------------------------------------------------
BATCH SIZE vs POLL INTERVAL: WHICH KNOB TO TURN
------------------------------------------------------------

Both control the same underlying quantity: how many alerts the
pipeline can absorb per unit time. The choice between them is a
tradeoff:

Wider max_batch_size (e.g., 1000 alerts every 30 seconds):
  + Fewer poll cycles overall (less file I/O overhead)
  + Larger work units mean better amortization of per-batch setup
  - Each poll cycle does a larger memory burst
  - Worst-case latency is longer (alerts arriving just after a
    poll wait the full poll interval)
  - Variance in per-cycle work is higher

Shorter poll_interval_seconds (e.g., 250 alerts every 5 seconds):
  + Smoother work distribution over time
  + Lower worst-case ingest latency
  + Better fit for environments with steady high-rate arrival
  - More frequent file I/O (modest cost)
  - More frequent timestamp-based dedup work
  - Slightly more overhead overall, but usually negligible

Default guidance: for bursty environments, prefer wider
max_batch_size. For steady-rate environments, prefer shorter
poll_interval_seconds. If unsure, scale max_batch_size first
because it's a simpler change and most SMB environments are
moderately bursty.

------------------------------------------------------------
DIAGNOSTIC: MEASURE YOUR DROP RATE
------------------------------------------------------------

The pipeline logs every batch-limit hit at INFO level. This makes
drop measurement straightforward — no instrumentation required.

When ingest reads alerts and hits the cap, it logs:

  ingest: Batch limit 250 reached, stopping read

When it reads alerts without hitting the cap, it logs:

  ingest: Read N alerts at or above level 6

If you see "Batch limit reached" in your journal, the poll cycle
hit the cap and left more alerts unread. They wait until the next
cycle. If the next cycle also hits the cap, you have a sustained
backlog growing every poll. If "Batch limit reached" appears
multiple times in a row, you are dropping.

The single most useful metric is the percentage of poll cycles
that hit the batch cap during a representative time window.
Each poll cycle emits exactly one "Read N alerts" line, and
capped cycles additionally emit a "Batch limit reached" line
just before it. So total polls = count of "Read" lines, and
capped polls = count of "Batch limit" lines:

  # Count polls that hit the batch cap in the last hour
  CAPPED=$(sudo journalctl -u jrsoctriage --since "1 hour ago" | \
           grep -c "Batch limit .* reached")

  # Count total polls in the same window
  TOTAL=$(sudo journalctl -u jrsoctriage --since "1 hour ago" | \
          grep -c "ingest: Read .* alerts at or above level")

  # Calculate percentage
  if [ "$TOTAL" -gt 0 ]; then
    echo "Capped polls: $CAPPED of $TOTAL ($(echo "scale=1; $CAPPED * 100 / $TOTAL" | bc)%)"
  fi

Interpretation:

  0%      No drops. Current max_batch_size has sufficient
          headroom for your environment's burst behavior.
          This is the home-lab default-of-250 state.

  1-5%    Occasional drops on outlier bursts. Acceptable for
          most home labs. SMB deployments should investigate
          whether the dropped bursts contained security-
          relevant alerts.

  5-15%   Material drops. This lab measured 8% at
          max_batch_size=100 in home-lab conditions and
          considered it unacceptable for production — that
          observation drove the 250 default. Raise
          max_batch_size or shorten poll_interval_seconds.

  15%+    Severe drops. Pipeline is losing significant alert
          volume to caps. Address immediately: either raise
          max_batch_size by 4x, shorten poll_interval_seconds
          to 5-10 seconds, or both.

NORMAL EXCEPTION — cold startup after a pipeline pause:

After a long pipeline pause (planned maintenance, host reboot,
crash recovery, post-upgrade restart), the file position is
behind real-time by however long the pause was. The first
several polls after startup are catching up on accumulated
backlog and will almost certainly hit the batch cap. This is
expected behavior, not a sizing problem.

To distinguish cold-startup catch-up from real undersizing,
either:

  - Wait until the pipeline has been running for at least an
    hour after restart before measuring drop rate, OR

  - Run the measurement window starting after the catch-up
    has completed (visible as the first poll cycle that does
    NOT log "Batch limit reached" — at that point real-time
    parity is restored).

For example, to exclude the first 30 minutes after startup
from your measurement window:

  sudo journalctl -u jrsoctriage --since "30 minutes ago" | \
    awk '/ingest: Read .* alerts at or above level/ { total++ }
         /Batch limit .* reached/ { capped++ }
         END { if (total>0) printf "%.1f%% (%d/%d polls capped)\n",
                                   capped*100/total, capped, total }'

If you're measuring after a recent pause and your cap-hit rate
is high but is *decreasing* poll over poll, that's catch-up
behavior — let it complete and re-measure. If cap-hit rate is
high and *stable* hours after startup, that's an undersizing
problem requiring action.

For a longer-window measurement (24 hours catches daily-cycle
burst patterns like scheduled scans):

  sudo journalctl -u jrsoctriage --since "24 hours ago" | \
    awk '/ingest: Read .* alerts at or above level/ { total++ }
         /Batch limit .* reached/ { capped++ }
         END { if (total>0) printf "%.1f%% (%d/%d polls capped)\n",
                                   capped*100/total, capped, total }'

For SMB deployments with high baseline volume, also compare
ingest rate to expected raw rate:

  # What the pipeline ingested in the last hour
  INGESTED=$(sudo journalctl -u jrsoctriage --since "1 hour ago" | \
             grep "ingest: Read" | \
             awk '{sum+=$8} END {print sum}')

  # Your expected hourly rate (daily ÷ 24)
  # For example: 30,000/day = 1,250/hour
  echo "Ingested in last hour: $INGESTED"
  echo "Compare against your Wazuh dashboard's hourly rate"

If INGESTED is consistently smaller than your expected hourly
rate AND you see "Batch limit reached" frequently, you are
falling behind on intake.

------------------------------------------------------------
TUNING WORKFLOW
------------------------------------------------------------

1. Deploy with defaults (max_batch_size=250,
   poll_interval_seconds=30).

2. After 24 hours of operation, measure your cap-hit rate using
   the diagnostic above.

3. If cap-hit rate is 0%, no action needed.

4. If cap-hit rate is 1-15%, raise max_batch_size by 4x and
   re-measure after another 24 hours. Iterate until cap-hit
   rate is at or near 0%.

5. If cap-hit rate is 15%+, raise max_batch_size by 10x AND
   shorten poll_interval_seconds to 5-10 seconds in the same
   change, then re-measure.

6. If you cannot reach 0% cap-hit rate even with very large
   batch sizes (10,000+) and short poll intervals (5 seconds),
   your raw alert volume has outgrown what v1.0 single-process
   ingest can absorb. Filter more aggressively at the Wazuh layer
   to reduce what reaches jrSOCtriage, and watch the v2.0
   (multi-host worker pool) roadmap, which addresses this
   directly.

Secondary signals that batch size may be undersized even at
low cap-hit rates:
  - End-to-end latency from alert generation to verdict is
    growing over the day (alert timestamp vs verdict timestamp)
  - Graylog shows verdicts arriving in clusters at poll-interval
    multiples rather than smoothly
  - Position pointer in .ingest_position grows slower than
    alerts.json file size

------------------------------------------------------------
WHEN TO STOP TUNING BATCH AND START SCALING WORKERS
------------------------------------------------------------

max_batch_size and poll_interval_seconds let the pipeline ABSORB
high arrival rates. They do not change how fast the pipeline can
PROCESS those absorbed alerts.

If batch capacity is sufficient (you're not falling behind on
ingest) but verdicts are still queueing because the LLM stage is
saturated, the bottleneck has moved. Solutions at that point are
different: more workers (max_workers), faster cloud endpoints
(cloud-primary mode), or aggressive dedup widening to reduce LLM
call volume in the first place.

The rough sequencing for SMB-scale deployment sizing:

  1. Right-size max_batch_size and poll_interval_seconds to
     absorb your raw rate with 4x headroom.
  2. Set dedup_silence_seconds appropriately (240 for home-lab-
     style, 600 for SMB-scale).
  3. Tune rules.json to filter chatty rules pre-LLM.
  4. Scale max_workers and/or LLM endpoint capacity to handle
     the resulting LLM call rate.

Steps 1-3 are operator-tunable for free. Step 4 has cost
implications (more GPU, more cloud spend, or both).

============================================================
COMMON ISSUES
============================================================

Pipeline starts but reads 0 alerts:
  - Check Wazuh is running: docker ps | grep wazuh
  - Check alerts file exists: ls -la /path/to/wazuh/logs/alerts/alerts.json
  - Check position file: cat /path/to/jrsoctriage/.ingest_position
    If position > file size, delete it: rm /path/to/jrsoctriage/.ingest_position

No Zeek flows in alerts:
  - Check Zeek is running: sudo systemctl status zeek
  - Check current_log_dir contains conn.log:
      ls "$(jq -r .sources.zeek.current_log_dir config.json)/conn.log"
  - Check archive_log_dir contains dated subdirs (for older alerts):
      ls "$(jq -r .sources.zeek.archive_log_dir config.json)" | head
  - See ZEEK LOG PATH MODEL section for layout examples

No LLM triage:
  - Check Ollama is running on the LLM host
  - Check llm.enabled is true in config.json
  - Check LLM endpoint is reachable: curl http://YOUR_OLLAMA_HOST:11434
  - Check threshold criteria — alerts only reach the LLM if rule level
    >= min_rule_level, OR if an escalation override fires (abuse score,
    baseline spike, or always_include network). If all alerts are below
    level 6 and no escalation conditions are met, triaged will be 0.

No emails being sent:
  - Check email.enabled is true in config.json
  - Check SMTP credentials in config.json
  - Emails only send for NOTIFY verdicts at or above min_confidence_to_email

ntopng not enriching:
  - Check ntopng is running: ps aux | grep ntopng
  - Check ntopng.enabled is true in config.json
  - Check credentials in config.json
  - Verify API works: curl -s "http://YOUR_NTOPNG_HOST:3001/lua/rest/v2/get/ntopng/interfaces.lua" -u admin:PASSWORD
  - If ntopng fails to start at boot, or is started AFTER jrsoctriage,
    ntopng enrichment will be silently absent for that entire session.
    jrsoctriage must be restarted after ntopng is confirmed running.
    Check boot order if this happens repeatedly — ntopng should start
    before jrsoctriage. This applies to all optional services (Graylog,
    Zeek, ntopng) — connections are initialized once at startup and not
    retried automatically during the session.

SOURCES
------------------------------------------------------------

sources.wazuh.enabled
  → true/false. Enable Wazuh alert ingestion. Required for pipeline.

sources.wazuh.alerts_file
  → Path to Wazuh alerts.json. Usually inside the Wazuh Docker volume.

sources.zeek.enabled
  → true/false. Enable Zeek flow enrichment. Recommended.

sources.zeek.current_log_dir
  → Path to Zeek's current log directory containing conn.log, dns.log,
    ntlm.log (live, unrotated files).
    e.g. /opt/zeek/logs/current

sources.zeek.archive_log_dir
  → Path to the parent directory of Zeek's dated rollover subdirs
    (named YYYY-MM-DD, containing compressed .log.gz archives).
    If unset, defaults to the parent of current_log_dir.
    See the ZEEK LOG PATH MODEL section above for layout examples.

sources.zeek.log_dir (DEPRECATED)
  → Older single-path form. Still accepted for backward compatibility
    but will log a warning. Use current_log_dir / archive_log_dir instead.

sources.ntopng.enabled
  → true/false. Enable ntopng active flow enrichment. Optional.

sources.ntopng.endpoint
  → URL of ntopng web interface. e.g. http://192.168.30.4:3001

sources.ntopng.ifid
  → ntopng interface ID for your SPAN interface. Check ntopng UI.

sources.ntopng.auth.username / .password
  → ntopng login credentials.

sources.ntopng.skip_networks
  → List of CIDR ranges (as strings) that should NOT be queried
    against ntopng. Useful for host-private addresses that ntopng
    will never have data for, such as Docker bridge gateways.
  → Loopback (127.0.0.0/8, ::1) and link-local (169.254.0.0/16,
    fe80::/10) are skipped automatically and don't need to be
    listed here.
  → Default: [] (no extra skips)
  → Common use case for hosts running Docker:
      "skip_networks": ["172.17.0.0/16", "172.18.0.0/16",
                        "172.19.0.0/16", "172.20.0.0/16"]
    Adjust the list to match the bridge networks Docker actually
    uses on your monitored hosts (check with `docker network ls`
    and `ip addr show`).
  → If your alerts include other host-private addresses (VPN
    gateway IPs, K8s pod ranges, hypervisor virbr0/virbr1 IPs),
    add those CIDRs as well. Each entry that matches an alerted
    IP saves one ntopng API call and one log warning.

sources.graylog.enabled
  → true/false. Enable Graylog context log fetching. Recommended.

sources.graylog.endpoint
  → URL of Graylog API. e.g. http://192.168.30.4:9000

sources.graylog.context_window_minutes
  → How many minutes of Graylog logs to pull around each alert.
  → 0.5 = 30 seconds before and after (default).
  → 1.0 = 1 minute before and after.
  →
  → TRADEOFF: This is the single largest knob affecting prompt size.
  →
  → IMPORTANT: prompt size scales heavily with the number of users and
  → endpoints on the network because Zeek flows and Graylog logs scale
  → with overall traffic volume, not alert count. A small home lab with
  → 2 users and 19 hosts will see prompts much smaller than a real
  → office with 12 users on the same number of hosts.
  →
  → As a calibration anchor: a 2-user home lab running at 1.0 minute
  → produces typical prompts of 14-17k characters with peaks around 22k
  → on a busy domain controller. A 12-user office with similar host
  → count would likely produce 30-50k+ prompts at the same setting. A
  → real enterprise deployment may need to LOWER context_window_minutes
  → below 0.5 to keep prompts within LLM context budgets.
  →
  → Larger prompts mean:
  →   - More tokens consumed per LLM call (cost on cloud endpoints)
  →   - Higher chance of hitting LLM context limits (especially llama.cpp
  →     where -c must be sized accordingly; for 1.0-minute windows on a
  →     small lab use -c 32768; for larger deployments size -c to your
  →     observed peak prompt + 2k response headroom, rounded up to next
  →     power of 2)
  →   - Slightly slower LLM inference per call
  →
  → BENEFIT: more surrounding events means the LLM sees the full pattern
  → leading up to and following the alert. For low-volume hosts or
  → environments where alerts are sparse, 1.0 minute gives better context
  → without much cost. Observation from this lab: moving from the default
  → 0.5-minute window to 1.0-minute window produced a slight reduction
  → in NOTIFY-and-NOTE volume — the LLM had more context to recognize
  → benign patterns and downgrade marginal alerts that would otherwise
  → have escalated. The reduction was small in this environment and
  → may be larger in busier deployments where surrounding-context
  → patterns are more informative. The tradeoff is real: better verdict
  → quality and slightly lower analyst load against higher per-call
  → token cost and longer inference times.
  →
  → Recommended: start at the default (0.5) and watch journalctl for
  → prompt size logs (search for "Sending prompt"). Raise to 1.0 if
  → verdicts frequently flag "MISSING INFO" about events just before
  → or after the alert window AND your prompts stay comfortably under
  → your LLM's context budget. Lower below 0.5 if prompts approach
  → your LLM context limit or you see truncation warnings.

sources.zeek.context_window_minutes
  → Same tradeoff as graylog. Zeek flow volume scales with window size.
  → For a host doing heavy DNS traffic, 1.0 vs 0.5 can be the difference
  → between ~20 flows and ~50 flows in the prompt.

sources.graylog.auth.username / .password
  → Graylog API credentials.

------------------------------------------------------------
PROCESSING
------------------------------------------------------------

processing.poll_interval_seconds
  → How often (seconds) to check for new Wazuh alerts. Default: 30.

processing.max_batch_size
  → Max alerts to read per poll cycle. Default: 250.
  → Sized for home-lab and small-team deployments. SMB-scale
  → deployments need to scale this up — see MAX_BATCH_SIZE section
  → for the formula and worked examples. If your raw alert rate
  → × poll_interval exceeds max_batch_size, the pipeline falls
  → behind silently.
  → Raise during storms to process faster.

processing.dedup_silence_seconds
  → Suppress duplicate rule+host alerts for this many seconds.
  → Default: 240 (4 minutes). Raise to reduce LLM call volume.
  → This is the primary volume control knob.

processing.baseline_multiplier
  → Hourly/daily count must exceed average by this factor to flag
  → as above baseline. Default: 2.0 (2x average = flag it).

processing.min_baseline_days
  → Minimum days of history required before baseline comparison.
  → Default: 3. Alerts on new hosts skip baseline comparison.

  Note on Graylog baseline fields during establishing period:
  When baseline data is insufficient (rule too new, host too new, or
  fewer than min_baseline_days of history), the numeric GELF fields
  (_gl2_count_last_hour, _gl2_count_last_24h, _gl2_count_last_7d,
  _gl2_hourly_avg, _gl2_daily_avg) default to 0, and the boolean
  fields (_gl2_above_hourly_baseline, _gl2_above_daily_baseline)
  default to false. This produces a visual collision: a real "rule
  fired zero times" looks identical to a "rule too new for baseline."
  The disambiguation lives in _gl2_baseline_note, which describes
  the establishing status in human-readable form. When building
  dashboards or queries that depend on baseline values, filter on
  _gl2_baseline_note first to confirm the baseline is established
  before trusting the numeric counts.

processing.escalation_multiplier
  → Count must exceed average by this factor to trigger an
  → escalation override (forces LLM review regardless of rule level).
  → Default: 4.0.

processing.max_workers
  → Global concurrent pipeline workers — each carries one alert
  → through all stages (enrichment, DB writes, LLM call, GELF ship),
  → not just the LLM. Default: 1 (serial).
  → Raise to 4-8 for faster processing during alert storms.
  → Each worker needs its own DB connection — handled automatically.
  → You can run more workers than LLM endpoints/instances, but do
  → not configure more concurrent LLM instances than workers.

------------------------------------------------------------
FILTERING
------------------------------------------------------------

filtering.min_rule_level
  → Minimum Wazuh rule level to process. Default: 6.
  → Level 3 = nearly everything. Level 6 = medium+. Level 9 = high+.
  → Alerts below this level are skipped UNLESS an escalation override
  → fires (abuse score, baseline spike, always_include network, etc.)

filtering.abuse_escalation_threshold
  → AbuseIPDB score threshold to force LLM escalation regardless
  → of rule level. Default: 50. Requires abuseipdb.enabled = true.

filtering.frequency_escalation_enabled
  → (In the interface this is the toggle "Escalate frequency anomalies
  →  (count > multiplier x daily baseline)" on the Config screen, under
  →  Filtering — that is the name you look for to turn it on or off.)
  → Whether a rate spike can force an alert to the LLM regardless
  → of rule level. Default: true.
  → When on, an alert whose 24h count exceeds escalation_multiplier
  → x its daily average (with at least min_baseline_days of history)
  → is escalated even if it is below min_rule_level. This is the
  → "something that is normally quiet just got loud" signal —
  → a low-severity rule that suddenly fires far above its own
  → baseline is often worth a look precisely because the volume
  → changed, not the severity.
  →
  → Why you might turn it OFF — the reason it exists:
  →   Devices that are only sometimes on the network. The baseline
  →   is a daily average over ALL time, including the hours or days a
  →   device is absent — so a laptop that's on-site during the
  →   workday and gone overnight, a machine powered off on weekends,
  →   anything that connects and disconnects on a schedule, ends up
  →   with a misleadingly LOW daily average. Then an ordinary day of
  →   completely normal on-site activity easily exceeds
  →   escalation_multiplier x that depressed average and trips the
  →   override — auto-escalating traffic that is normal for that
  →   device, for no real reason. THIS is the case the toggle was
  →   built for. In a fleet with many come-and-go devices, frequency
  →   escalation can generate more noise than signal, and turning it
  →   off is the cleaner choice (until those devices are tuned out
  →   via rules.json).
  →
  →   (Two lesser reasons: you may simply not want rate-based
  →   escalation at all — preferring to decide what reaches the LLM
  →   by rule level, abuse score, and explicit rules.json entries;
  →   and synthetic load testing produces artificial rate spikes
  →   that trip the override on nearly every rule, so it's turned off
  →   during such tests.)
  →
  → Leaving it on is the right default for production: it is one of
  → the mechanisms that lets you run a higher min_rule_level (fewer
  → routine alerts) without going blind to a quiet rule suddenly
  → going loud. Turning it off trades that safety net for fewer,
  → more predictable escalations.

filtering.always_include.networks
  → List of CIDR ranges — any alert involving these IPs is always
  → escalated to the LLM regardless of rule level.
  → Use for DMZ, internet-facing, or high-value segments.
  → e.g. ["192.168.40.0/24"]

filtering.always_include.hosts
  → List of hostnames — same as networks but per-host.

filtering.escalate_first_seen_rule
  → true/false. Escalate to LLM the first time a rule fires on a
  → host, even if below min_rule_level. Default: true.

filtering.first_seen_lookback_days
  → How many days back to check "first seen" history. Default: 14.

ENRICHMENT
------------------------------------------------------------

enrichment.geo_ip.enabled
  → true/false. GeoIP lookup for external IPs via ip-api.com.

  jrSOCtriage uses the free tier of ip-api.com for GeoIP enrichment.
  The free tier has two characteristics worth knowing:

  - HTTP-only (no HTTPS). Free-tier requests use plain HTTP. A
    network-level attacker between the jrSOCtriage host and ip-api.com
    can observe which IPs are being looked up and could in theory
    modify responses (e.g., flag a malicious IP as benign). For a
    home lab over a residential ISP this is mostly theoretical. For
    SOC use cases where GeoIP feeds verdict decisions, MITM-modified
    responses are a legitimate concern.

  - Rate limit: 45 requests per minute per source IP. The pipeline's
    in-memory IP enrichment cache mitigates this for repeat IPs, but
    a burst of unique external IPs (port scan, DDoS analysis) could
    briefly hit the limit. Rate-limited responses return as empty
    enrichment without an error.

  For deployments that need higher rate limits or HTTPS, ip-api.com
  offers a paid Pro tier (~$13-15/month for 30,000 requests/day with
  HTTPS). v1.3 adds a provider-agnostic GeoIP backend supporting
  several online providers and a local MaxMind database, so
  high-volume operators can avoid rate limits entirely.

  For a home lab the free tier is fine. For a production SOC, plan
  on the v1.3 GeoIP backend for higher limits or HTTPS.

============================================================
MODEL SELECTION (read first)
============================================================

The LLM model you configure as your primary endpoint is the single
most important decision for this project's value. Smaller models, or
heavily quantized larger models, do not produce triage worth shipping
to an analyst. This section explains what we have tested and what we
recommend.

------------------------------------------------------------
TESTED AND RECOMMENDED
------------------------------------------------------------

PRIMARY (LOCAL):
  gemma4:26b (Q4_K_M) on RTX 3090 / 4090 / 5090 (24GB+ VRAM).
  Two local engines have been validated against this model with
  identical quality — pick either:

  Option A: via ollama
    The original validated configuration. Approximately one month
    of continuous operation against a real home lab network at
    approximately 2,400 LLM-triaged alerts per day, with 1-2
    VERDICT: FAILED records per day on genuinely-hard alerts.
    Mean response time ~15s (range 9-24s on the production
    sample). Verdicts consistently synthesize across host inventory
    notes, alert content, network flow evidence, and frequency
    baselines.

  Option B: via llama.cpp
    Same model, same quantization, same verdict quality — faster on
    identical workload. Production sample (n=45+ over a 30-minute
    window): mean ~11s, median 10s, p95 14s, range 8-16s, with
    one 33s outlier. That's roughly 25-30% faster than ollama for
    the same prompts. Verdicts validated against the same 58
    consecutive alerts that ollama produced; all 58 parsed cleanly
    at HIGH confidence with correctly-formatted output, including
    site-specific-note citation from hosts.json and rules.json
    where applicable.

    llama.cpp's scheduling is tighter than ollama's — no idle
    recovery windows between calls — producing noticeably higher
    sustained GPU memory bandwidth and thermal load under
    continuous operation. On hardware with limited thermal margin
    or under warm ambient conditions, this may produce more
    FAILED-to-converge events than ollama. This project's
    reference deployment on consumer-grade RTX 3090 has observed
    1-10 FAILED per day on llama.cpp depending on conditions,
    versus 1-2 per day on ollama. Operators with newer GPUs,
    better cooling, or cooler ambient should expect different
    rates. See "Single-endpoint reliability under sustained load"
    earlier in this file for the full discussion.

    llama.cpp requires more setup work than ollama and has two
    response-handling quirks that the shipped llm_caller.py
    already accounts for:

      1. Use the `/v1/chat/completions` endpoint, NOT
         `/v1/completions`. The raw completions endpoint returns
         channel markers (e.g. `<channel|>thought`) inline with
         the model output, which corrupts parsing. The chat
         endpoint applies the harmony parser and separates
         thinking from final answer.

      2. Read `choices[0].message.content` first, then fall back
         to `choices[0].message.reasoning_content` if content is
         empty. Some prompts cause gemma to produce a structured
         VERDICT/CONFIDENCE block inside reasoning_content rather
         than content; the shipped code handles both paths.

    llama-server launch command (the exact invocation used in
    production validation):

      cd /opt/llama.cpp
      ./build/bin/llama-server \
        -m ~/models/google_gemma-4-26B-A4B-it-Q4_K_M.gguf \
        -c 65536 \
        -ngl 999 \
        --parallel 1 \
        --host 0.0.0.0 \
        --port 8080 \
        --temp 0.2

    Notes on the flags:
      -c 65536    Context window size. Real triage prompts
                  reach ~21,000 characters (~5,000-6,000 tokens).
                  Per-slot context = c / parallel. At --parallel 1
                  (single-worker pipeline) the full 65536 is
                  available per call, comfortably absorbing input
                  plus thinking-capable reasoning traces. At
                  --parallel N for multi-worker pipelines, each
                  slot gets c/N tokens — recompute to confirm a
                  real prompt plus reasoning trace fits in one
                  slot. Going below 32768 total is not recommended
                  at any parallel setting because hard alerts can
                  require substantial reasoning headroom.
      -ngl 999    Offload all layers to GPU. Required for the
                  RTX 3090 performance numbers above.
      --parallel  Match this to your jrSOCtriage worker count.
                  llama-server with --parallel N provisions N
                  inference slots and partitions the context window
                  across them — each slot gets c/N tokens of
                  context. The pipeline serializes LLM calls per
                  worker (one inference in-flight per worker
                  against a given endpoint), so the right
                  --parallel value is the number of jrSOCtriage
                  workers pointed at this endpoint.

                  For single-worker deployments (this project's
                  reference configuration), use --parallel 1.
                  Observation from extended production: --parallel 4
                  against a single-worker pipeline produced a
                  higher wall-clock timeout rate than --parallel 1
                  on the same prompts and model. The internal
                  mechanism is something we haven't characterized;
                  the bottom line is more failures. Match parallel
                  to worker count.

                  Multi-worker jrSOCtriage against a single
                  local llama-server endpoint is not a recommended
                  configuration. Each in-flight inference competes
                  for the same GPU compute and memory bandwidth,
                  so N concurrent calls take roughly N times longer
                  wall-clock per call than a single call does on
                  the same hardware. If you attempt this anyway
                  (e.g. testing scenarios, transient bursts during
                  v1.2+ surge handling), raise the per-endpoint
                  timeout_seconds proportionally — at --parallel 4
                  with 4 concurrent workers, expect per-call wall-
                  clock latency to grow by roughly 4x and size the
                  timeout accordingly. For production multi-worker
                  scaling with local endpoints, run one llama-server
                  endpoint per worker on separate GPUs.

                  Cloud endpoints work differently: the provider
                  scales inference capacity under the hood, so
                  many jrSOCtriage workers can point at a single
                  cloud endpoint without per-call latency growing
                  the way it does on a local GPU. The constraint
                  on cloud endpoints is the rate-limit ceiling
                  (RPM, TPM), not concurrent-inference-per-GPU
                  contention. See the rate-limit headroom section
                  for sizing cloud-primary deployments at scale.
      --temp 0.2  Matches the jrSOCtriage default for ollama.

    Other launch configurations may also work; this is the
    one we ran in production. Build notes (Fedora 44 in our
    test environment): gcc 15 was used (gcc 16 failed),
    fresh CUDA toolkit from NVIDIA's site rather than the
    distro package. The Q4_K_M GGUF was sourced from Bartowski's
    HuggingFace repository (a widely-used community quantization
    source) rather than an official Google release.

    Build llama.cpp from the latest git HEAD, not a tagged
    release or a distro package. gemma4 support is recent
    enough that older builds — including builds from a few
    weeks before this was written — will fail to load the
    model. The error mode is a load-time tensor failure, not
    a runtime issue, so you'll know immediately if your
    build is too old. If `git pull && rebuild` doesn't help,
    your build environment is the next thing to look at
    (the gcc 15 / fresh CUDA notes above).

    Important: the ollama-managed model blob for this same
    model does NOT work with llama.cpp. Even though it's the
    same model and same quantization, pointing llama-server
    at the ollama blob path produces tensor errors and fails
    to load. Download a standalone GGUF file (we used
    Bartowski's HuggingFace repository) and pass that path to -m.
    Treat the ollama and llama.cpp installations as fully
    separate model storage.

  Both options:
    Quantization Q4_K_M is what the bulk of the production data was
    generated against; Q4_K_L (from the same Bartowski source) has
    also been validated in production and performs effectively
    identically — roughly 0.2s/call slower, negligible in practice.
    Either Q4_K_L or Q4_K_M is a good choice. Aggressive quantization
    below this (Q4_0, Q3_K_M, or lower) breaks reasoning quality and
    is not recommended.

  Developer note (for operators forking jrSOCtriage): see
  "THINKING-CAPABLE MODELS — THE TOKEN-BUDGET TRAP" below for the
  detailed principle. Short version: do NOT add a `num_predict`
  parameter to the ollama options block when calling thinking-
  capable models; ollama needs to run unbounded. llama.cpp uses
  max_tokens=16384 (high enough to let thinking-capable models
  finish naturally); cloud engines use max_tokens=4096 as
  defensive headroom because cloud APIs do not have the channel-
  leak issue.

    floor for this task; use GPT-4o mini (the cheapest validated
    OpenAI option) instead of dropping to nano.

    where the API silently truncates prompts (e.g., gemma4:31b
    via Gemini free tier). When jrSOCtriage's 13-21k character
    prompts exceed an undocumented hidden context window, the
    API truncates silently and the model generates output based
    on partial input. The output contains hallucinated context-
    shaped values (e.g. invented IPs that look plausible) and/or
    broken response format (missing VERDICT/CONFIDENCE/SUMMARY
    structure). The failure is the API's silent truncation, not
    the model's reasoning. Operators seeing this should reduce
    sources.graylog.context_window_minutes to shrink prompts,
    move to a paid tier with documented larger context windows,
    or use a local model.

------------------------------------------------------------
RATE LIMIT HEADROOM AND DEPLOYMENT SIZING
------------------------------------------------------------

The cloud endpoints documented above have substantially different
rate-limit ceilings at the lowest paid tier. For most home-lab
deployments the differences are theoretical — alert volume is low
enough that all three providers handle it comfortably. For SMB
scale the differences become real planning constraints.

Tier 1 ceilings (verified via API response headers where exposed):

  Provider               RPM   TPM
  ---------------------- ----- --------------------------
  OpenAI gpt-4.1-mini    500   200,000
  Gemini 3 Flash Preview 150   not header-exposed
  Anthropic Haiku 4.5     50    60,000 (50k in / 10k out)

Translated to maximum LLM escalations per day (RPM × 60 × 24,
assuming sustained throughput at the documented ceiling):

  Provider               Theoretical max LLM escalations/day
  ---------------------- -----------------------------------
  OpenAI gpt-4.1-mini    720,000
  Gemini 3 Flash Preview 216,000
  Anthropic Haiku 4.5     72,000

These extreme volumes are cloud-provider rate-limit ceilings, not
benchmarked sustained rates. A future load-harness campaign will
replace these documented maxes with measured numbers as that work
progresses.

WHY 60% IS THE PLANNING TARGET, NOT 100%:

Alert traffic is bursty. A single triggering event — login storm,
scanner sweep across a /24, service flap, ransomware sweep, AD
replication failure, internal pentest — can spike multiple alerts
into the LLM queue within seconds. Sustained operation at 100%
of the rate-limit ceiling leaves zero headroom for these bursts
and will produce rate-limit failures during exactly the events
that most need timely triage.

Planning for 60% of the documented ceiling gives roughly 1.67×
burst capacity:

  Provider               Recommended sustained max/day (60%)
  ---------------------- -----------------------------------
  OpenAI gpt-4.1-mini    432,000
  Gemini 3 Flash Preview 129,600
  Anthropic Haiku 4.5     43,200

For reference, this project's home-lab baseline runs about 2,400
LLM escalations/day in escalate-everything mode (or ~240/day in
level-6-plus-filters mode), far below all three recommended
sustained maxes. The rate-limit ceiling becomes a planning
constraint at SMB scale where alert volume and host inventory
grow with deployment size.

The same comfortable margin exists on the local-inference path,
not just against cloud rate limits. This lab's local endpoint
(gemma on llama.cpp) averages ~11 seconds per call, so a single
worker tops out around 5.5 LLM calls/min. Actual escalation
arrival averages ~1.6 calls/min. That is roughly 3× headroom on
a single worker — escalation volume could about triple before the
worker saturates. This margin is not a deliberate tuning choice;
it falls out of running one worker against this deployment's real
escalation rate, and it is why a normal doubling of alert volume
(or an absorbed storm) is barely noticeable. Your own margin
depends on your endpoint's per-call time, worker count, and
escalation rate — measure it rather than assuming the same ratio.

THE TPM ANGLE:

TPM is the secondary constraint and is more cost-shaped than
performance-shaped at the volumes documented above. It governs
how much spend you can burn through before being rate-limited at
Tier 1 budget caps. For typical jrSOCtriage prompts (13-21k
characters, roughly 4-6k tokens):

  - OpenAI at 200,000 TPM: ~33-40 prompts/min before TPM ceiling
  - Anthropic at 50,000 ITPM: ~8-10 prompts/min before ITPM ceiling
  - Gemini: undocumented at API level; observed comfortable headroom

OpenAI's TPM ceiling is high enough that RPM (500/min) hits first.
Anthropic's TPM ceiling is low enough that ITPM (50k/min) is the
real constraint at Tier 1 rather than RPM. This is worth knowing
when sizing: at Anthropic Tier 1, even though the table above says
72,000 escalations/day ceiling based on RPM, the TPM ceiling will
constrain to roughly 14,400 escalations/day in practice on typical
jrSOCtriage prompts. Operators planning Anthropic-primary
deployments above that range should request a Tier 2 upgrade.

WHEN TO RAISE TIER:

OpenAI auto-tiers upward based on sustained usage and reliable
payment history — operators don't typically need to take action.
Anthropic and Gemini require explicit tier-upgrade requests
through their respective consoles. For Anthropic this is
straightforward ($40 cumulative spend qualifies for Tier 2, which
raises Haiku 4.5 to 1,000 RPM and substantially higher TPM). For
Gemini the tier qualification thresholds are documented in the
AI Studio console.

------------------------------------------------------------
HOW TO SIZE WORKERS FOR YOUR DEPLOYMENT
------------------------------------------------------------

Worker count is the lever that controls whether jrSOCtriage can
keep up with your alert volume. Too few workers and the queue
backs up — alerts wait, verdicts arrive late, the pipeline lags
real time. Too many workers and you waste API quota on idle
concurrency. The formula below gives you a deterministic answer
based on your actual workload and your actual model latency,
rather than guessing.

THE FORMULA:

Given:
  R = raw alerts per day (from Wazuh dashboard or estimate)
  D = dedup ratio (this project's lab measures 12.5 with
      dedup_silence_seconds=240; use yours if you've tuned
      dedup/min_rule_level/escalation differently)
  F = filter pass-through factor (what fraction of deduplicated
      records actually reach the LLM):
        F = 1.0 → escalate-everything mode (no rules.json filtering)
        F = 0.1 → default level-6-plus-filters mode (this project's
                  rules.json suppresses ~90% of records pre-LLM)
        Your F depends on how aggressively your rules.json suppresses
        chatty rules. Measure it once your deployment is running by
        comparing records-produced to LLM-calls-actually-made.
  T = average LLM response time in seconds (measured from your
      journald logs, or use the table below as a starting estimate)

Compute:
  1. Records_per_day          = R / D
  2. LLM_calls_per_day        = Records_per_day * F
  3. Sustained_calls_per_min  = LLM_calls_per_day / 1440
  4. Calls_per_min_per_worker = 60 / T
  5. Sustained_workers        = Sustained_calls_per_min / Calls_per_min_per_worker
  6. Workers_with_headroom    = Sustained_workers * 1.40 (round up to next integer)

The 1.40 multiplier sizes total worker capacity to 1.40x the
projected per-minute volume. This isn't arbitrary: it's chosen
so that the 40% reserve absorbs the gap between mean response
time and p95 response time, which is what determines whether
the pipeline keeps up during sustained-rate alert surges.

Concretely from this project's earlier parallel-4 production data:
mean response time was 11.4s, p95 was 16.6s. p95 is ~34% slower
than mean, which means per-worker capacity drops by roughly that
much when sustained traffic happens to land in the upper tail.
Sizing workers at 1.40x mean-capacity (40% reserve) covers the 34%
p95 drop with 6 percentage points of extra cushion. The 1.40×
multiplier is appropriate guidance regardless of parallel setting;
pull your own mean and p95 from journald to refine for your specific
deployment. If your own mean-to-p95 gap is wider than 34% (say,
50%+), bump the multiplier accordingly. Pull p95 from journald with
the same grep pattern that pulls mean, but use the sort + awk
approach from the validation section of these docs to get the 95th
percentile sample.

Round up to the next whole worker — a worker that's only 30%
utilized on average is fine; what you cannot tolerate is being
one worker short during a storm.

WHY OVER-PROVISIONING WORKERS IS DELIBERATE, NOT WASTE:

It is worth understanding what adding workers actually does,
because the instinct to size workers to your *average* load is
exactly wrong. Extra workers do NOT increase your average LLM
calls per minute. Your average throughput is capped by your alert
*arrival* rate — the pipeline cannot process escalations faster
than they arrive, no matter how many workers are idle waiting.
What extra workers increase is your *peak* capacity: the maximum
calls per minute the pipeline can sustain when a burst of
escalations lands at once.

This is the whole point. In steady state, a single worker running
at 30% utilization looks over-provisioned. But escalations do not
arrive at a constant rate — they arrive in bursts (a storm, a
scan, a host event fanning out across an inventory). When a burst
hits, the alerts that exceed a single worker's per-minute capacity
queue up and wait. With more workers, that same burst drains in
parallel instead of serializing behind one worker. The extra
workers sit idle most of the time and earn their keep in the
minutes that matter — they are what keep the pipeline from falling
behind during the exact events that most need timely triage.

So running more workers than the average-load formula calls for
is a deliberate design choice, not accidental over-provisioning.
This project's own deployment does exactly this. The formula above
gives you the floor for keeping up with average load; size above
it for burst headroom, and treat the idle capacity as insurance
rather than waste.

RESPONSE TIME STARTING ESTIMATES:

  sudo journalctl -u jrsoctriage --since "1 week ago" | \

    awk '{for(i=1;i<=NF;i++) if($i=="in") print $(i+1)}' | \

    awk '{sum+=$1; n++} END {print "mean:", sum/n, "n:", n}'

  caught an 11.6s outlier while its median stayed ~2.2s; another

  tail-range differences are sample noise at n=15-19, not a real
  speed gap. (An earlier ~3.1s 4.1-mini figure was from an OpenAI
  service-degradation period — superseded by the ~2.1s clean median.)

  GPT-4o mini is the recommended default cloud endpoint: same
  median speed as GPT-4.1-mini, same triage quality, at under half
  the per-token cost. No latency penalty for the cheaper model.
  Cloud latency varies with provider conditions, so measure your
  own endpoint over a representative window rather than trusting
  any single quoted figure.

The ollama and llama.cpp numbers above are apples-to-apples on the
same model, same quantization, same prompts, same GPU. llama.cpp
runs the same gemma4:26b Q4_K_M weights faster across every
percentile observed. Both produce identical-quality triage.

Full stats from production data:
  ollama   (n=4895):           mean 15.0s, median 14.0s, p95 22.8s, max 59.6s
  llamacpp (n=93,   1 hour):   mean 11.4s, median 11.2s, p95 16.6s, max 21.1s

The response-time numbers above are conservative for default-config

and sources.zeek.context_window_minutes=1.0 — wider than the
shipped defaults of 0.5 — which produces larger Zeek flow blocks,
larger host log windows, and larger prompts overall. Larger prompts
mean longer inference times and a heavier latency tail. An operator
running default-config context windows will see meaningfully faster
mean and p95 than the numbers above; the sizing math built from
these numbers therefore has headroom built in.

SETTING THE PER-ENDPOINT CALL TIMEOUT (TIMEOUT (SECONDS)):

Each LLM endpoint in the interface has its own TIMEOUT (SECONDS)
field (next to PRIORITY and MAX CONCURRENT). It caps how long the
pipeline waits for a call before giving up on it. The shipped
default is a loose placeholder, not a tuned value — at a very loose
timeout a genuinely stuck call drags the pipeline a long way behind
before it is reaped; too tight and you cut loose calls that would
have finished fine. The right value is a TRADEOFF you tune to your
own situation. There is no single correct number.

Set it PER ENDPOINT. A local endpoint (gemma4 on your GPU) and a
cloud endpoint (GPT-4o mini) have very different latency
distributions — the local one runs ~10-15s typical, the cloud one
~2s typical — so each endpoint's timeout should be set against THAT
endpoint's own measured latency, not a shared number.

WHAT THE DIAL DOES (when you have a fallback endpoint configured):
With a fallback, a timed-out call does not lose the verdict — it
fails over to the next endpoint. So the timeout becomes a
cost-vs-smoothness throttle, and where you set it is a business
decision keyed to two things: (a) do you actually have a lag
problem, and (b) how cheap is your fallback.

  - Looser timeout: fewer calls trip it, so fewer failover events,
    so less money spent on the paid fallback — but when a call does
    hang, the pipeline falls further behind before the call is cut
    loose. Cheap, but laggy on the occasional stuck call.

  - Tighter timeout: stuck calls get reaped fast so the pipeline
    stays smooth — but more calls trip the timeout and fail over to
    the paid fallback, so more money spent.

MEASURE YOUR DISTRIBUTION FIRST. Pull your own per-call latencies
from journald (the RESPONSE TIME grep above gives the mean; this
one gives the full distribution so you can read the percentiles the
timeout is set from):

  sudo journalctl -u jrsoctriage --since "12 hours ago" \
    | grep -oP 'responded in \K[0-9.]+(?=s)' \
    | sort -n \
    | awk '{a[NR]=$1; sum+=$1}
      END{if(NR==0){print "no matches";exit}
        p50=a[int(NR*0.50)]; p90=a[int(NR*0.90)];
        p95=a[int(NR*0.95)]; p99=a[int(NR*0.99)];
        printf "count=%d min=%.1f mean=%.1f p50=%.1f p90=%.1f p95=%.1f p99=%.1f max=%.1f\n",
               NR,a[1],sum/NR,p50,p90,p95,p99,a[NR]}'

Use a window with enough calls to be meaningful (a few hundred is
good; the percentiles are noisy on a handful of calls). If a config
change or restart altered the endpoint mid-window, set --since to
just after that change so old data does not pollute the sample.

WHERE TO SET IT (three points on the dial; it is YOUR call, keyed
to your situation):

  - p99 + 10s — DEFAULT. The "cheap but not sloppy" setting: only
    the worst ~1% of calls even approach the timeout, and the +10s
    cushion means most of even those still complete locally, so you
    almost never pay fallback — but a genuinely stuck call still
    gets reaped promptly rather than dragging the pipeline. Use this
    if your pipeline catches up fine and you have no felt lag. (Note:
    a cheap fallback ALONE is not a reason to go tighter — if the
    pipeline recovers fine at this setting, the smoothness a tighter
    timeout buys is worth nothing to you regardless of how cheap the
    fallback is. Tighten only when you actually have a problem to
    solve.)

  - p99 + 5s — tighter. For a cheap fallback plus some felt slow-tail
    lag: reaps the tail a few seconds sooner, at the cost of a few
    more failover calls.

  - p95 — tightest sensible, a legitimate choice for someone ON THE
    EDGE of falling behind with a cheap fallback. At p95 roughly 5%
    of calls fail over to the paid fallback — a real, ongoing cost —
    but in exchange the pipeline stays ahead instead of backing up.
    If you are near saturation and your fallback is inexpensive,
    paying to stay smooth is a reasonable trade. This is your call.

  Below p95 is not advised: you would be failing over normal calls
  the local model would have finished fine in a couple more seconds,
  spending fallback money for little benefit.

NO FALLBACK CONFIGURED is a different situation, not just another
point on this dial. With no secondary endpoint, a timed-out call is
a LOST verdict (it synthesizes FAILED with nothing to catch it) —
there is no cost dimension, only verdict loss. So do not tune for
cost; tune to avoid losing verdicts: set the timeout to p99 + 10s
(the default) or looser. Never tighter than p99 + 5s with no
fallback, and prefer p99 + 10s — with nothing to catch a timeout,
erring loose only costs a little pipeline wait, while erring tight
throws away real verdicts.

WORKED EXAMPLE (this project's local gemma4 endpoint): a 12-hour
run measured p99 ≈ 20.5s, so p99 + 10s ≈ 30s — which is where this
deployment runs its local endpoint's timeout. The fallback (GPT-4o
mini) is cheap (~$0.0007/call, so tightening would cost about a
dollar a month), BUT the pipeline catches up fine from the
occasional 30s call, so there is no benefit to tightening — the
setting correctly stays at p99 + 10s. Cheap fallback did not change
the decision, because there was no lag problem to solve.

COST REALITY CHECK — WHAT A DEPLOYMENT ACTUALLY COSTS
------------------------------------------------------

Cost is driven by one thing: how many alerts reach the LLM. Worker
count does not affect it (workers change how fast the queue drains,
not how many calls get billed — see PROVISIONING ABOVE THE MINIMUM).
Hardware generation does not affect it. The only levers are your raw
alert volume and how aggressively you filter before the LLM stage.

Two real measurements ground everything below:

  - The May 2026 GPT-4.1-mini runs: a level-6 run at a ~6.3M/day pace
    billed ~$2 for the hour (770 LLM calls); an escalate-everything
    run at a ~2.7M/day pace billed ~$18 for the hour (9,529 LLM calls).
    Measured on GPT-4.1-mini's ~$0.0023/call rate.
  - The recent gpt-4o-mini capacity run: level-6 mode at a ~29.8M/day
    pace billed about $8 for the hour (~11,100 LLM calls), on
    gpt-4o-mini's ~$0.0007/call rate.

gpt-4o-mini delivers comparable triage quality at roughly a third of
GPT-4.1-mini's per-call cost, so it is the cost-sensible default and
the basis for the table below.


CLOUD COST — level-6-plus-filters on gpt-4o-mini
-------------------------------------------------

How many raw alerts survive filtering and dedup to reach the LLM is
not a fixed number: it moves with how aggressively rules and the
level-6 floor filter, and with the dedup collapse rate (often around
90%, sometimes less). Better-tuned deployments push a smaller fraction
through. The table shows a range per volume — the low (cheaper) end
assumes a well-tuned ~1:150 raw-to-LLM survival ratio, the high end a
looser ~1:100.

  | Raw alerts/day | LLM calls/day  | Cost/year (gpt-4o-mini) |
  |----------------|----------------|-------------------------|
  | 300,000        | 2,000-3,000    | ~$510-$770              |
  | 600,000        | 4,000-6,000    | ~$1,020-$1,530          |
  | 900,000        | 6,000-9,000    | ~$1,530-$2,300          |
  | 3,000,000      | 20,000-30,000  | ~$5,110-$7,670          |

A deployment at ~900k raw/day — roughly the original design-target
volume — lands around $1,530-$2,300 per year depending on filter and
dedup tuning. On the pricier GPT-4.1-mini the same volumes run roughly
3x these figures.

To estimate escalate-everything cost, multiply these figures by
roughly 10x — escalate-everything sends about ten times as many alerts
to the LLM (no level-6 or rules filtering), so a 900k-raw deployment
that costs ~$2,300/year filtered on 4o-mini would cost roughly
~$23,000/year run as escalate-everything on the same model. On a local
model the ~10x lands as hardware rather than a per-call bill, and the
cost comes out roughly even either way (see the local-vs-cloud
discussion below).

A caveat on all cloud figures: they assume current gpt-4o-mini pricing
(~$0.0007/call as measured). Cloud pricing and model availability are
outside your control — providers deprecate older low-cost models over
time, and their replacements typically cost more. Treat this table as
a current-pricing snapshot, not a fixed rate you can bank on for years.


LOCAL vs CLOUD — a cost wash, decided on other grounds
-------------------------------------------------------

At realistic level-6-plus-filters volumes on gpt-4o-mini, the pure
cost comparison between cloud and local is roughly a wash. Worked at
900k raw/day (~9,000 LLM calls/day at the looser 1:100 ratio), cloud
runs about $2,300/year. A modest local GPU rig serving that same
level-6 volume costs a few thousand dollars up front plus a few
hundred a year in electricity — so the payback against ~$2,300/year of
avoided cloud cost lands on the order of several years, close to how
long you can expect a consumer GPU to last under sustained inference
load. (This project's RTX 3090 is five years old and still in
continuous production, but that is not something to count on — many
cards would not be.) The break-even roughly races the hardware's own
lifespan, which means at level-6 volumes you should not buy local
hardware to save money: on cost alone it is close to even either way.

The cost wash holds at escalate-everything volume too. Escalate-
everything is ~10x the LLM volume, but that 10x lands on both sides —
~10x the cloud bill (on the order of ~$23,000/year at 900k raw instead
of ~$2,300) and ~10x the local inference hardware to serve it — so the
two still come out roughly even on cost, the same as level-6. (Scaling
both the cloud bill and the required hardware by the same factor
preserves the payback ratio; volume cancels. What lengthened the
payback versus older estimates in this project is the move to
gpt-4o-mini's cheaper rate, which shrinks the annual cloud cost the
local hardware is measured against — not the escalation mode. Exact
worker and GPU counts for a given volume belong in PROVISIONING ABOVE
THE MINIMUM, not here; this section sizes dollars, not workers.)

Because cost is close to even either way, the local-vs-cloud decision
comes down to two non-cost factors — and both favor local:

  1. Information security and data residency. A local model keeps your
     alert content, host details, network evidence, and enrichment
     context entirely inside your infrastructure — none of it is sent
     to a third-party API. For defense contractors, critical
     infrastructure, regulated data, or anyone on an attacker target
     list, keeping security telemetry in-house is often the deciding
     factor on its own, independent of dollars.

  2. Pricing and availability independence. A local model you own
     cannot be deprecated out from under you or repriced on a vendor's
     schedule. The cloud table above is a snapshot of today's
     gpt-4o-mini rate; a local endpoint insulates you from that rate
     moving, and from the cheap model you depend on being retired in
     favor of a pricier one.

Level-6-vs-escalate-everything is a separate decision, made on coverage
and threat model rather than cost (it costs ~10x either way, as a
recurring cloud bill or as upfront local hardware). Escalate-everything
is worthwhile in high-threat-model environments — defense contractors,
critical infrastructure, organizations facing patient
below-threshold adversaries — and hard to justify for typical SMB IT
threat models (commodity malware, opportunistic ransomware,
phishing-driven compromise), where filtered mode catches what matters.

The clean rule of thumb: cost is roughly even between local and cloud
at any volume and in either mode, so decide local-vs-cloud on data
security and pricing independence, not on a payback calculation; and
decide level-6-vs-escalate-everything on coverage and threat model,
since that choice costs ~10x either way. If you find yourself sizing
for escalate-everything without a threat-model reason, the better
question is usually "should I filter more aggressively at the Wazuh
layer?" — often yes.


WHEN YOUR DEDUP RATIO ISN'T 12.5:

The 12.5 ratio is what this project's home lab measures with its
current config (dedup_silence_seconds=240, context_window_minutes=1,
escalate-everything mode). At a shorter dedup_silence_seconds of
30s the same lab measured a ratio of about 7.5; widening the window
to 240s collapsed more repeat-fire patterns and shifted the ratio
upward. Your ratio will differ if you've tuned any of:

  - min_rule_level (raise = fewer escalations = higher D)
  - dedup_silence_seconds (raise = more grouping = higher D)
  - escalation override thresholds (looser = lower D)
  - rules.json pre-LLM mechanisms — never_escalate, severity
    downgrades, escalate_if conditional filters, per-rule rate
    limits. These reduce LLM call volume by suppressing or
    deferring records before they reach the LLM stage.

Note: rules.json also has post-LLM functions (rule notes and
per-host rule notes that inform the LLM's verdict) which do
NOT affect D. Only the pre-LLM mechanisms above affect the
records-to-LLM ratio. Similarly, hosts.json does not affect D
at all — host inventory notes run downstream of dedup and the
rules engine. They inform the LLM's verdict and suppress NOTIFY
emails for known-benign host activity, but every record that
passes pre-LLM filtering still consumes an LLM call regardless
of hosts.json content or rules.json post-LLM notes. Tightening
post-LLM context (hosts.json notes, rules.json rule notes)
improves verdict quality and reduces operator email volume
without reducing LLM cost.

To measure your own ratio after the pipeline has been running
for at least 24 hours:

  RAW=$(sudo journalctl -u jrsoctriage --since "24 hours ago" | \
        grep "ingest: Read" | \
        awk '{sum+=$8} END {print sum}')
  ESC=$(sudo journalctl -u jrsoctriage --since "24 hours ago" | \
        grep "Triage:" | wc -l)
  echo "Raw alerts ingested: $RAW"
  echo "LLM-triaged:          $ESC"
  [ "$ESC" -gt 0 ] && echo "Dedup ratio: $(echo "scale=2; $RAW / $ESC" | bc)"

Plug your measured ratio into the formula above. If your ratio
is much lower than 12 (say, under 6), your dedup or escalation
config is likely permissive and you should consider tightening.
If much higher (above 20), your filters may be dropping signal
you'd actually want to see — verify by sampling what gets dropped.

PROVISIONING ABOVE THE MINIMUM: COST, RAM, AND SMOOTHNESS
---------------------------------------------------------

The formula above gives you the minimum stable worker count plus
a 40% reserve. That is a floor for correctness, not a target you
must hold exactly. Whether it is worth provisioning above that
floor depends entirely on whether your LLM endpoint is local or
cloud, because the two have opposite cost structures.

The key fact that makes this decision simple: worker count does
not change how many LLM calls get made. The number of LLM calls
is fixed by your alert stream and your dedup/filter config (the
R/D/F inputs above), not by how many workers drain the queue in
parallel. More workers means more alerts enriched and triaged
concurrently — the queue drains faster and cycle times stay even
— but the same number of LLM calls are billed either way. What
scales with worker count is RAM: each in-flight alert holds its
enrichment state (Zeek rows, ntopng, Graylog context, the built
prompt), and max in-flight is bounded at workers × 4, so peak
memory grows roughly linearly with worker count.

LOCAL LLM ENDPOINT (SINGLE ENDPOINT) — a few workers help; do not
over-provision.
This guidance is for a single local LLM endpoint — one GPU serving
roughly one inference at a time (e.g. gemma4:26b on one RTX 3090).
With a single endpoint, the GPU — not the worker pool — is the
throughput bottleneck, and adding workers does not raise the
sustained LLM-call ceiling: only one call runs at a time regardless
of how many workers you have.

Even so, on a single-endpoint config a few workers beat exactly one.
While one worker is parked on its LLM call, the others can finish
hot-path work (dedup, enrich, first-ship) instead of everything
serializing behind that call. That makes the pipeline more efficient
in general and noticeably smoother on drain — as each worker frees
from its LLM call it can immediately clear queued hot-path work
rather than waiting for the single worker to cycle. So even against
one local GPU, provision a few workers (start around 3-4) rather than
one. Past that, extra workers mostly just block waiting on the single
GPU while consuming RAM, so there is little reason to provision far
above the formula's minimum plus its built-in reserve.

CLOUD LLM ENDPOINT — the minimum is a floor, not a target.
A cloud endpoint (GPT-4.1-mini, Claude Haiku, Gemini Flash) has
effectively unbounded concurrency at any volume this pipeline
realistically produces. The endpoint is no longer the bottleneck;
the worker pool and queue-drain rate are. Because the LLM-call
count — and therefore the cloud bill — is invariant to worker
count, a cloud operator is free to provision well above the
formula's minimum. The only thing that grows is RAM, never the
API spend. If log flow is uneven — bursty Triage: clusters,
variable cycle_duration, an alert-processing-age that runs hot
during sustained load — that is the signature of running near
the queue-drain knee, and adding workers smooths it at no dollar
cost. Provision upward until log flow is smooth.

THE UPPER BOUND ON CLOUD OVER-PROVISIONING.
"Provision until smooth" is bounded by the database write path,
not by cost or by available RAM. Every triaged alert performs a
serialized SQLite write under a single writer lock. Past some
worker count the serialized write path saturates: workers finish
their LLM calls and then queue on the DB lock, so additional
workers stop improving smoothness and only add RAM and lock
contention. This is the same single-writer fsync ceiling
described in the THROUGHPUT AND CONCURRENCY MODEL and SCALING
CEILING sections above. From consumer-NVMe fsync benchmarks
(the same ~5,200 fdatasync ops/sec figure those sections cite),
that write path corresponds to a worker ceiling on the order of
~100 workers on consumer NVMe — substantially lower on slower
storage (SATA SSD, networked or overlay filesystems). This
figure is derived from disk benchmarks and the v1.0 per-alert
write profile; it has NOT been verified by pipeline load testing,
and it will move — almost certainly downward — if the v1.3+
durability roadmap changes per-alert fsync behavior. Treat it,
like the throughput ladders above, as benchmark-derived guidance
rather than a measured guarantee. In practice the useful cloud
operating band is: at or above the formula minimum, provisioned
upward until log flow is smooth, and well below the DB-write
ceiling — a range most cloud deployments sit comfortably inside.

DEDUP EFFECTIVENESS DEPENDS ON SHALLOW QUEUE DEPTH.
There is a second reason to provision above the minimum, separate
from smoothness, and it affects correctness rather than just even
log flow: the deduplication silence window is eroded by queue
wait time, and the resulting failure can cascade.

Dedup suppresses a repeated (rule_id, host) alert if it recurs
within dedup_silence_seconds (default 240s). That window is
measured from when the alert's dedup check actually executes —
not from when the alert arrived. When workers are saturated and
alerts wait in the queue before processing, that wait time is
effectively subtracted from the silence window: an alert delayed
30s in the queue experiences a ~210s effective window instead of
240s. The deeper the queue, the shorter the effective window, and
the fewer repeats dedup collapses.

The relationship is not linear. Whether a shorter window matters
depends on how tightly a given alert's repeats are spaced: a
noisy host re-tripping a rule every few seconds is collapsed
regardless of whether the window is 240s or 120s, while repeats
spaced near the window edge are lost as soon as the window
shrinks past them. Two deployments with the same queue depth can
see very different dedup impact depending on their traffic's
repeat-timing structure.

This can cascade, and the cascade is the dangerous part. Dedup
is this pipeline's primary load-shedding mechanism — it is what
reduces raw alert volume by an order of magnitude before the LLM
call. The feedback loop is: a deep queue compresses the effective
dedup window; a compressed window collapses fewer repeats; fewer
collapsed repeats means more alerts survive into the processing
path; more surviving alerts deepen the queue further; the deeper
queue compresses the window further. Each turn of the loop makes
the next turn worse. Because the mechanism that sheds load is the
one being degraded, the pipeline loses its protection against its
own input volume at exactly the moment it is already overloaded.
The practical consequence is that behavior past the saturation
threshold is not a gentle slope of slightly-worse dedup — it can
run away, and degradation accelerates rather than levels off.
This is why the provisioning guidance below is a floor for
stability, not a comfort margin: the regime past it does not fail
gracefully.

This effect is real but observed only under queue pressure. In
production at normal alert rates the queue stays at or near zero
(dedup collapses the bulk of volume before the LLM call, so the
post-dedup stream the workers drain is small), the effective
window is the full configured value, dedup behaves exactly as
designed, and the cascade never initiates. The effect was visible
only in synthetic overload testing, where deep queues coincided
with markedly lower collapse than the same pipeline achieves on
real production traffic at zero queue depth — including one
overload run whose collapse fell far below what window
compression alone would predict, consistent with the cascade
having engaged. That comparison is confounded by other
differences between synthetic and real traffic and should be read
as directional evidence of the mechanism, not a calibrated
figure.

LOCAL vs CLOUD — the mitigation differs, and for local there
isn't a clean one. The operational fix is the same as for
smoothness: keep the queue shallow so wait time stays near zero
and the loop never initiates. For a CLOUD endpoint that is
achievable — provision workers upward (within the DB-write
ceiling above) until the queue does not build under sustained
load. For a LOCAL single-GPU endpoint there is no such lever:
queue depth under sustained load is bounded by GPU inference
throughput, and adding workers cannot drain faster than the GPU
serves. A local deployment driven near or beyond its sustainable
LLM rate will build a queue, the queue will compress the dedup
window, the compression can engage the cascade above, and no
worker count arrests it — once the GPU is saturated there is no
in-config lever that recovers the window or breaks the loop short
of reducing input or stopping the pipeline. For local deployments
the headroom is therefore not a comfort margin but the actual
boundary between a stable system and a runaway one: size so the
post-dedup, post-filter LLM-call rate stays comfortably below
sustained GPU throughput, and treat that as a hard requirement,
not a recommendation.

Even with generous provisioning, a sufficiently large traffic
burst can transiently build a queue regardless of worker count or
endpoint type, and a large enough burst can itself be the trigger
that initiates the cascade rather than merely causing brief
compression. This is inherent to a wall-clock silence window
evaluated at processing time; it is a known v1.0 characteristic,
not a defect, and is documented here so operators size for
headroom rather than discover it under load. (A design that
anchors the dedup check on alert arrival time rather than
processing time would remove the effect entirely; this is noted
on the forward roadmap and is deliberately not changed in v1.0 to
avoid altering a validated processing path.)

-----------------------------------------------------------------------
DON'T REACH FOR FRONTIER MODELS (read this before reaching for Opus/GPT-5)
-----------------------------------------------------------------------

The validated models above are the ceiling of useful capability for
this pipeline, not the floor of acceptable. Frontier-tier models
(Claude Opus, GPT-5, Gemini 3 Pro, etc.) cost 10-30x more per call
and produce verdicts that are not meaningfully different from what
gemma4:26b or Claude Haiku produce on the same prompts.

The job this pipeline asks the LLM to do is bounded: read the
structured enrichment block, read the host inventory notes, read the
alert payload, follow the response format, output a verdict with
brief reasoning. That's a competent-mid-tier task, not a frontier-
reasoning task. The hard work was done upstream by the enrichment
pipeline — the LLM is reading and synthesizing well-prepared context,
not solving novel problems.

Why operators reach for bigger models anyway:
  - "Bigger is better" intuition transferred from chat use cases
  - A small number of MEDIUM-confidence verdicts that an analyst
    second-guesses, and an instinct to fix that with horsepower
  - Marketing pressure (provider docs naturally promote top tiers)

What actually moves verdict quality:
  - Better host inventory notes (hosts.json) — the LLM cites these
    constantly and they directly shape verdicts
  - Better rule notes (rules.json) — site-specific context for noisy
    or ambiguous rules
  - Better triage_guidance and sensor_context in config — these
    catch systemic false-positive patterns
  - More signal in the prompt (Zeek flows, baseline data, ntopng)

A frontier-tier LLM with a sparse hosts.json will produce worse
verdicts than gemma4:26b with a well-curated hosts.json. The leverage
is in the context, not the model size.

Cost example for a home lab (anchored to measured GPT-4.1-mini
pricing at $0.0023/call against this project's full-fat prompts,
current lab volumes):
  - Filtered mode (~240 LLM calls/day): ~$200/year cloud-only on
    GPT-4.1-mini. Lower with local primary plus cloud fallback
    (only ~$1/year for the rare fallback calls; local handles
    the rest).
  - Escalate-everything mode (~2,400 LLM calls/day at this lab's
    raw volume): ~$2,000/year on GPT-4.1-mini — about 10× more
    for 10× more LLM calls.
  - Same volumes on Claude Opus 4.5: 10-30× the GPT-4.1-mini
    figures (different provider tier; not measured at this prompt
    size by this project).
  - The Opus output is not 10-30× better. It is, in this project's
    testing, indistinguishable from Haiku on typical alerts.

Claude Haiku 4.5 and Gemini 3 Flash Preview are also validated
cloud endpoints but have different per-token pricing and have not
been measured against jrSOCtriage's full-fat prompts. Verify your
specific provider's pricing for your expected call volume.

If you're considering a frontier model because verdicts feel weak,
audit hosts.json and rules.json first. The fix is almost always
there.

------------------------------------------------------------
WHERE FRONTIER LLMs DO HELP: WRITING THE CONTEXT
------------------------------------------------------------

The previous section argues against frontier-tier LLMs for the
runtime triage path. That argument is specific to the per-alert
triage workload — bounded reading-and-synthesizing of well-prepared
context, where Haiku-tier and gemma4:26b perform indistinguishably
from Opus-tier.

There is a different jrSOCtriage workload where frontier-tier
models are genuinely useful: writing the host inventory notes,
rule notes, sensor context, and triage guidance that shape every
prompt. Unlike per-alert triage, this is a one-time-per-host or
one-time-per-rule effort that pays off thousands of times. Spending
more compute per piece of context to get sharper context is good
leverage.

What frontier-tier models help with here:
  - Drafting host inventory notes that capture the operational
    nuance of a host (what it does, what's normal for it, what
    would be anomalous, how to distinguish background services
    from user activity)
  - Refining rule notes that explain when a noisy rule is actually
    informative versus when it's expected background
  - Articulating sensor context (SPAN artifacts, ntopng L7 caveats,
    deployment-specific behaviors) in the precise language the
    runtime LLM will use to reason
  - Reviewing existing notes against actual production alerts to
    identify gaps the runtime LLM kept flagging in MISSING INFO

A reasonable workflow: paste a host's recent alerts, the current
hosts.json entry, and any operator commentary into a frontier-tier
chat assistant. Ask it to propose a tightened note that addresses
the false-positive patterns visible in the alerts. Iterate. Paste
the result into the Hosts tab in jrSOCtriage. The cost is a few
cents per host, the result is context that improves verdicts on
every future alert for that host.

This is "design-time work" rather than "runtime work." Frontier
models are good at design-time work. Haiku-tier and local 24B+
models are good at runtime work. Picking the right tier for each
keeps cost low without leaving verdict quality on the table.

------------------------------------------------------------
TESTED AND NOT RECOMMENDED
------------------------------------------------------------

The following configurations have been observed in real production
triage to produce misleading verdicts:

  - phi-4 14B (any quantization, including q8)
    Failure mode: ignores host inventory notes; reasoning often
    addresses the wrong question entirely (e.g., describing network
    traffic when the alert is about authentication); falls back to
    generic "looks normal" reasoning regardless of alert content;
    output format degraded with meta-commentary outside the structured
    response.

  - gemma4:31b (Gemini free API tier)
    Failure mode: heavy hallucination, escalates routine activity as
    suspicious. Larger parameter count did not compensate.

  - gemma4:26b below roughly Q4-class quantization (Q3 and smaller)
    Failure mode: heavy hallucination, escalates routine activity as
    suspicious. Quantizing this model aggressively enough to fit a
    smaller GPU degrades reasoning quality past the point of being
    useful. Note: Q4-class quantization itself is fine — see below.

The common pattern: smaller models, and even larger models heavily
quantized to fit on consumer hardware, produce confident-sounding
wrong answers. Confident wrong answers are worse than no triage at
all — they consume analyst attention without earning it, and they
erode trust in the pipeline's verdicts as a whole.

DO NOT run jrSOCtriage with a model that fits comfortably on a 16GB
GPU. The validated path is gemma4:26b at Q4-class quantization
(Q4_K_M and Q4_K_L both run in this project's production with
effectively identical results — Q4_K_L is roughly 0.2s/call slower,
negligible) on 24GB+ VRAM, or a capable cloud LLM (Claude Haiku
validated, others likely fine but unvalidated). The quantization
warning above is about going *below* Q4-class to squeeze onto
smaller hardware, not about Q4 itself — Q4_K_M/Q4_K_L on a 24GB
card is the reference configuration.

UNTESTED

Models not in either of the lists above. Some other models may work

RECOMMENDED CONFIGURATIONS
------------------------------------------------------------

PRODUCTION (RECOMMENDED): Local primary + cloud fallback.

  Strategy: fallback
  Endpoint 1: gemma4:26b on local 24GB+ GPU, priority 1
  Endpoint 2: any validated cloud model (claude-haiku-4-5-20251001,
              gpt-4.1-mini, gemini-3-flash-preview, etc.), priority 2,
              anonymize: true

  Local engine choice is a real tradeoff. Pick based on operational
  priorities:

  Option A — Ollama (recommended for most operators):
    Easier setup and operation. Gentler GPU stress profile —
    Ollama's scheduler provides idle recovery windows between
    calls, keeping sustained thermal and memory bandwidth load
    moderate. Approximately one month of continuous "escalate
    everything" operation on this project's RTX 3090 reference
    deployment produced 1-2 VERDICT: FAILED records per day
    against ~2,400 LLM-triaged alerts/day. Expected cloud cost
    at this rate: approximately $1/year on GPT-4.1-mini (measured
    rate; other validated cloud models priced separately).

  Option B — llama.cpp (recommended for operators prioritizing latency):
    Same model, same quantization, same verdict quality, ~25-30%
    faster per call (mean ~11s vs ~15s in this project's reference
    deployment). More setup work — build from source, manage the
    server process, account for the response-handling quirks
    documented earlier in this file. Tighter scheduling — no idle
    recovery windows between calls — produces higher sustained GPU
    thermal and memory bandwidth load than Ollama. On hardware with
    limited thermal margin, this may produce more FAILED-to-converge
    events. This project's reference deployment on consumer-grade
    RTX 3090 has observed 1-10 FAILED per day on llama.cpp across
    different conditions. Expected cloud cost at the upper end:
    approximately $8/year on GPT-4.1-mini (measured rate; other
    validated cloud models priced separately). Still trivial against
    the per-call latency improvement.

  Either engine paired with cloud fallback produces effectively
  100% reliable verdict production at near-zero cloud cost. The
  cloud endpoint isn't a redundancy backstop — it's a specialist
  for the genuinely-hard alerts where the local model's reasoning
  budget was insufficient. Local-plus-cloud operates as
  generalist-plus-specialist rather than primary-plus-backup.
  Anonymization aliases sensitive identifiers before they leave
  the network. Pick whichever cloud provider your team already
  has API access to.

QUICK START / NO LOCAL GPU: Cloud as primary, no local LLM.

  Strategy: fallback (or single-endpoint)
  Endpoint 1: any validated cloud model, anonymize: true
  Expected cost at typical home lab volumes (rules engine active,
  dedup at 240 seconds, min_rule_level at 6, ~240 LLM calls/day):
  approximately $200/year on GPT-4.1-mini (measured at $0.0023/call
  against this project's full-fat prompts, 13-21k characters with
  full enrichment). Claude Haiku 4.5 and Gemini 3 Flash Preview
  are also validated and produce high-quality verdicts at this
  prompt size, but have different per-token pricing and have not
  been measured against jrSOCtriage's specific prompt profile;
  check your provider's pricing for your expected call volume.

  Cloud-as-primary at "escalate everything" mode (all alerts above
  level 0 reaching LLM, ~2,400 LLM calls/day at this lab's volume)
  costs approximately $2,000/year sustained on GPT-4.1-mini —
  roughly 10× more than filtered mode. This configuration assumes
  disciplined filtering for cost control; if you intend to run
  escalate-everything, plan on local GPU.

PRIVACY-SENSITIVE: Local primary only, no cloud fallback.

  Strategy: single-endpoint or fallback (with no cloud endpoint)
  Endpoint 1: gemma4:26b on local 24GB+ GPU

  Expected behavior depends on local engine:

    Ollama: 1-2 VERDICT: FAILED records per day at this lab's
    ~2,400 LLM calls/day rate in escalate-everything mode, or
    proportionally fewer in level-6-plus-filters mode.

    llama.cpp: 1-10 VERDICT: FAILED records per day observed in
    this project's reference deployment on consumer-grade RTX 3090,
    with substantial variation by ambient temperature and
    operational load. Operators considering llama.cpp for
    privacy-sensitive single-endpoint deployment should baseline
    FAILED rates on their own hardware before relying on the
    configuration. See "Single-endpoint reliability under
    sustained load" earlier in this file for the full discussion.

  The pipeline emits FAILED rather than silently dropping.
  Operators who don't want any data leaving the network even with
  anonymization should accept this as the cost of strict privacy
  and monitor the FAILED rate to catch LLM stack issues early.

LOCAL-ONLY MULTI-MODEL (NOT RECOMMENDED): Two local endpoints, both
local models.

  Don't do this with a small or over-quantized model on the second
  endpoint. Running phi-4 14B or a heavily-quantized sub-Q4 model as
  a secondary endpoint on the expectation that its verdicts will be
  useful is misleading yourself. (Note this is not a knock on
  quantization per se — gemma4:26b at Q4_K_M/Q4_K_L is the validated
  primary; the caution is about weak or aggressively-shrunk models,
  not Q4-class quants of a capable model.) If you want coverage for
  the hard-alert tail your primary can't handle, use cloud fallback
  as the specialist endpoint (~$1/year for the rare escalations on
  GPT-4.1-mini at measured rate). If you want load balancing, use
  round-robin
  between two GPUs running the same validated model — but each
  GPU has to actually be capable of running gemma4:26b at Q4-class
  quantization (i.e. 24GB+ VRAM).

------------------------------------------------------------
WHY THIS MATTERS
------------------------------------------------------------

jrSOCtriage's value proposition is that LLM triage produces verdicts
good enough to direct an analyst's attention to the alerts that
actually matter, while suppressing routine noise. That value
proposition depends entirely on verdict quality. A model that
escalates routine activity creates more analyst work, not less. A
model that suppresses genuine concerns hides risk. Either failure
mode is operationally worse than running without LLM triage at all.

The cost regimes above (filtered ~$200/year cloud, escalate-everything
~$2,000/year cloud, fallback-only ~$1/year cloud) are based on a
measured per-call rate of $0.0023 on GPT-4.1-mini against this
project's full-fat prompts. Your numbers will vary with prompt
size, alert volume, filtering aggressiveness, and cloud provider.
Tune your filters first and check your spend against your provider's
billing before assuming a cost regime.

------------------------------------------------------------
THINKING-CAPABLE MODELS — THE TOKEN-BUDGET TRAP
------------------------------------------------------------

This section documents a non-obvious failure mode that affects
any local engine adapter calling a thinking-capable model. If
you are forking jrSOCtriage to add a new local engine (vLLM, an
updated llama.cpp build, a different backend entirely), read this
section before touching token-budget parameters.

The principle:
  Thinking-capable models — gemma3-thinking, deepseek-R1, the
  Claude extended-thinking variants, OpenAI o1/o3, GPT-5, and
  others — split their output into two channels: a reasoning
  trace and a final answer. The reasoning trace can be thousands
  of tokens; the final answer is typically a few hundred. A
  "max output tokens" parameter (`num_predict` in ollama,
  `max_tokens` in llama.cpp and OpenAI-compatible APIs) caps the
  SUM of both channels. If the cap is reached while the model
  is still in the reasoning channel, the final answer never gets
  written.

This produces two distinct failure modes, both observed in this
project's production data during development:

Failure mode 1 — channel leak:
  The model finishes reasoning naturally and writes a final
  answer, but the cap forces the final answer into the reasoning
  channel instead of the content channel. When the caller reads
  `content`, it gets an empty string and falls back to
  `reasoning_content`, which contains the entire thinking trace
  with the answer embedded somewhere inside it. The verdict
  regex then matches the FIRST instance of "VERDICT:" it finds
  (typically an intermediate verdict the model reconsidered)
  rather than the final answer.

Failure mode 2 — mid-reasoning truncation:
  The model is still in the reasoning channel when its reasoning
  budget runs out — whether that budget is a token cap
  (num_predict, max_tokens) or a wall-clock timeout at the caller
  layer. It is cut off mid-thought. The response contains a
  reasoning trace but no final answer at all. The caller either
  produces a parse failure (no VERDICT found), matches an
  intermediate verdict from inside the truncated reasoning, or in
  the wall-clock case never receives a response at all and logs
  a timeout error. Token-cap and timeout truncation are the same
  underlying mechanism; the FAILED-verdict handling described
  later in this document treats both as expected occasional
  outcomes that the fallback chain absorbs.

  Critically, this failure mode is biased toward the most
  important alerts: easy alerts (routine logon, expected agent
  disconnect, recognizable scanner traffic) finish in a few
  hundred reasoning tokens and don't hit either bound. Hard
  alerts (lateral movement chains, multi-source correlation,
  ambiguous attack patterns, alerts requiring synthesis across
  host inventory + network flows + baseline data) require the
  most reasoning and are exactly the ones that hit the cap or
  timeout and fail. The bug eats your highest-value verdicts
  while leaving routine triage intact. An operator spot-checking
  random verdicts to QA the pipeline would see clean output and
  conclude all is well — the wrong verdicts are concentrated on
  the alerts a human analyst would have spent the most time on.

Both modes produce wrong verdicts that look superficially valid.
The diagnostic signal is response size: clean verdicts in this
pipeline are typically 400-800 characters. Leaked-reasoning
responses are 5,000-40,000 characters. A spike in response sizes
above 1,500 characters is the leading indicator that token-
budget truncation is happening.

A secondary diagnostic signal is latency clustering at the
truncation boundary. If your model's `num_predict` or
`max_tokens` is set such that it takes (say) 33-35 seconds to
fill the budget, you will see a tight latency cluster around
33-35 seconds for the affected calls. Clean responses will be
distributed across the model's natural latency range.

Per-engine handling in jrSOCtriage v1.0:

  ollama:
    The shipped llm_caller.py deliberately omits `num_predict`
    from the options block. Ollama then lets the model run
    unbounded until it produces its end-of-output token
    naturally. This is the correct configuration. Do NOT add
    `num_predict` to the options block under any circumstances
    if calling a thinking-capable model.

  llama.cpp:
    The shipped llm_caller.py sets `max_tokens=16384`. This is
    high enough that thinking-capable models reach their natural
    end-of-output before hitting the cap on jrSOCtriage's prompt
    sizes (13-21k input characters producing ~500-2000 reasoning
    tokens plus ~150 answer tokens). If you reduce this value,
    you reintroduce the bug. If you increase prompt size
    significantly (richer host inventory, longer context windows),
    consider raising max_tokens further.

  Cloud endpoints (Anthropic, OpenAI, Gemini):
    All three providers' APIs treat output channels independently
    at the API level — the `max_tokens` parameter applies to the
    answer channel only, not the reasoning channel. The shipped
    llm_caller.py uses `max_tokens=4096` for cloud endpoints as
    defensive headroom; this does NOT have the channel-leak
    issue because the API surfaces the channels separately. If
    you are calling a thinking-capable cloud model with extended
    thinking explicitly enabled (Claude extended-thinking, etc.),
    refer to that provider's documentation for the reasoning-
    budget parameter and set it independently from max_tokens.

For new engine adapters (future releases):
  Before merging a new local engine adapter, verify the following
  for each thinking-capable model the adapter will support:

    1. Identify the engine's token-budget parameter name and
       semantics. Determine whether it caps total output, answer-
       channel output, or reasoning-channel output specifically.

    2. If the parameter caps total output (most common in local
       engines): either omit it entirely (let the engine run
       unbounded) or set it high enough that no realistic
       reasoning length plus answer length can hit the cap.

    3. Test against a representative production prompt at the
       upper end of expected size. Check response sizes in the
       resulting verdicts — if any are above 1,500 characters,
       the token budget is interacting badly with the model's
       reasoning. Investigate before shipping.

    4. Document the engine-specific choice in llm_caller.py with
       a comment referencing this section, so future maintainers
       understand why the parameter is set the way it is.

The historical context:
  This pipeline encountered failure mode 1 once during ollama
  bring-up (resolved by removing `num_predict`) and failure mode 2
  once during llama.cpp bring-up (resolved by raising max_tokens
  from 4096 to 16384). Both bugs produced verdicts that looked
  superficially correct on cursory inspection but were
  systematically wrong on hard alerts. The diagnostic in both
  cases was the response-size spike — looking at character counts
  on the verdict text revealed the channel leak immediately, but
  no other surface signal was visible. Operators forking the
  codebase should treat this principle as a first-class concern,
  not an engine-specific footnote.

LLM CONFIGURATION

llm.enabled
  → true/false. Master switch for LLM triage.
  →
  → When true (default): every alert passing filtering goes to a
  → configured LLM endpoint, gets a verdict, and may be emailed if
  → confidence thresholds are met.
  →
  → When false: ENRICHMENT-ONLY MODE. The pipeline still ingests,
  → deduplicates, enriches with host inventory + IP reputation +
  → geo + MITRE + baseline + Zeek/ntopng context, and ships all of
  → that to Graylog. No LLM call is made, no email is sent. The
  → _gl2_triage_complete field stays false on every alert.
  →
  → Operationally useful for:
  →   - Evaluation phase before committing to LLM cost
  →   - A separate SIEM/SOAR consuming the enriched alerts
  →   - MSSP feeding an upstream analysis system
  →   - Privacy-sensitive deployments where even local LLM is
  →     too much
  →
  → Graylog stream rules need awareness of this mode. Rules that
  → filter on _gl2_triage_complete:true will not match. Rules
  → that key on enrichment fields (_gl2_abuse_score, wazuh_rule_level,
  → host/IP fields, baseline counts) work normally.
  →
  → Toggle via the "Enable LLM triage" checkbox at the top of the
  → LLM Endpoints card in the web interface, or set directly in
  → config.json. Restart the pipeline service after the change.

llm.strategy
  → "round_robin" — distribute calls across enabled endpoints evenly.
  → "fallback"    — use endpoints in priority order, only fall back
  →                 on failure.

llm.endpoints[].name
  → Display name for this endpoint. Shown in logs and Graylog fields.

llm.endpoints[].type
  → "ollama"    — local Ollama server (default)
  → "gemini"    — Google Gemini API
  → "openai"    — OpenAI API or compatible endpoint
  → "anthropic" — Anthropic Claude API
  → "llamacpp"  — llama.cpp server

llm.endpoints[].model
  → Model string sent to the endpoint. The values below are the strings
  → as the provider's API expects them; copy them exactly.
  →
  → Anthropic (Messages API) accepts both an alias ("claude-haiku-4-5")
  → that auto-updates to the latest minor revision of that family, and
  → a dated/pinned form ("claude-haiku-4-5-20251001") that stays on a
  → specific tested version. For production, prefer the dated form so
  → behavior doesn't shift under you when Anthropic ships an update.
  →
  → Ollama:     "gemma4:26b"
  → Gemini:     "gemini-2.5-flash"
  →             "gemini-3-flash-preview"  (preview tier, less stable)
  → OpenAI:     "gpt-4o"
  →             "gpt-4.1-mini"
  → Anthropic:  "claude-haiku-4-5-20251001"  (recommended pinned)
  →             "claude-haiku-4-5"           (alias, auto-updates)
  →             "claude-sonnet-4-5"          (also dated form available)

llm.endpoints[].url
  → For Ollama/llama.cpp: base URL e.g. http://192.168.30.30:11434
  → Not used for cloud endpoints (gemini/openai/anthropic).

llm.endpoints[].api_key
  → API key for cloud endpoints (gemini/openai/anthropic).
  → Leave empty for local Ollama endpoints.

llm.endpoints[].priority
  → Lower number = higher priority. Used in fallback strategy.
  → In round_robin, all enabled endpoints are used equally.

llm.endpoints[].timeout_seconds
  → How long to wait for a response before failing over. Default: 60.
  → Increase for large local models. Cloud endpoints rarely need >30s.

llm.endpoints[].max_concurrent
  → Max simultaneous calls to this endpoint.
  → Set to 1 for local Ollama (GPU can only run one at a time).
  → Set to 3-4 for cloud endpoints.

llm.endpoints[].anonymize
  → true/false. Anonymize hostnames, usernames, IPs, and domains
  → before sending to this endpoint. Always true for cloud endpoints.
  → Not needed for local Ollama.

llm.endpoints[].keep_alive
  → Ollama only. How long to keep model loaded in VRAM after a call.
  → -1 = keep loaded indefinitely (recommended for triage pipelines).

output.graylog.enabled
  → true/false. Enable shipping triaged alerts to Graylog as GELF.
  → Disabled by default. Most deployments want this on if they have
  → Graylog at all.

output.graylog.host
  → Hostname or IP of the Graylog GELF UDP input. Often the same host
  → as the API endpoint above but on a different port.
  → Example: "127.0.0.1" or "graylog.lab.local"

output.graylog.port
  → UDP port the GELF input listens on.
  → Default: 12201 (Graylog's default GELF UDP port).

The destination must be a "GELF UDP" input in Graylog. Create one in
Graylog's System -> Inputs -> Launch new input -> "GELF UDP". This
input is independent of the API endpoint used for sources.graylog
(which is HTTP, not UDP).

What gets shipped:
  - alert metadata (rule id, host, agent, timestamp, ips)
  - LLM verdict (NOTIFY/NOTE/SUPPRESS) and reasoning
  - enrichment (canonical_hostname, abuse score, mitre tags)
  - prompt (controlled by logging.prompt_log_mode — see below)
  - status fields for filtering (gl2_triage_complete, etc.)

If output.graylog.enabled is true but host is empty, the pipeline
logs an error and the alert is NOT shipped. Verify by tailing logs
or watching for "Shipped to Graylog" / "Failed to ship" lines.

This setting is exposed in the Sources tab under "Graylog Output".
Changes take effect after restart.

------------------------------------------------------------
PROMPT CUSTOMIZATION
------------------------------------------------------------

The prompt sent to the LLM has two layers of guidance:

  1. BASELINE TRIAGE GUIDANCE — three rules permanently included
     in every prompt's "When evaluating alerts:" rubric. They define
     how this pipeline does triage (distinguishing user from service
     activity, citing corroborating evidence, interpreting baseline
     numbers correctly). Not exposed in the config UI. Operators who
     genuinely need to override them can edit the
     _BASELINE_TRIAGE_GUIDANCE constant near the top of
     prompt_builder.py, but doing so is not recommended — these
     rules were arrived at empirically over weeks of production
     observation and changing them tends to make verdicts worse.

  2. SITE-SPECIFIC GUIDANCE — the four config keys below. These
     describe the deployment's sensor shape, network notes, and
     environment-specific triage rules that don't apply universally.
     Editable through the Config tab in the web interface or
     directly in config.json.

The defaults shipped in config_json.sample assume the most common
self-hosted IDS deployment shape: Suricata on a SPAN/mirror port,
ntopng for active flow visibility, Wazuh for endpoint signal.
Operators with different deployment shapes should remove or edit
the items that don't apply.

prompt_customization.sensor_context
  → List of strings injected into the SENSOR CONTEXT block of every
  → LLM prompt. Default content describes a SPAN-deployed Suricata +
  → Wazuh setup. If your Suricata is inline (not SPAN), remove the
  → SPAN-related items. The Wazuh-specific items (rootcheck rule 510
  → false-positive note, host-local event note) are universal to
  → Wazuh deployments and worth keeping. Edit through the Config tab.

prompt_customization.triage_guidance
  → List of strings injected as SITE-SPECIFIC TRIAGE GUIDANCE,
  → rendered as a separate section AFTER the baseline rubric. Default
  → content covers SPAN+abuse-score interaction and ntopng L7 label
  → caveats. Remove the SPAN item if not running SPAN; remove the
  → ntopng item if ntopng isn't enabled. Add your own
  → environment-specific guidance here as needed.

prompt_customization.network_notes
  → List of strings appended to the network context block.
  → Use for notes about specific subnets or routing behavior.

prompt_customization.strip_redundant_fields
  → true / false. Default: true.

  Controls whether the alert JSON in the prompt is "abridged" (default)
  or full. When true, fields that are either always-the-same-value
  (rule.mail) or already represented in the structured ALERT SUMMARY
  block above the JSON (rule.level, rule.description, rule.id,
  rule.firedtimes, rule.mitre, agent.id, agent.name, agent.ip, top-
  level id, decoder) are removed before serialization. Compliance
  metadata fields (rule.pci_dss, rule.gpg13, rule.gdpr, rule.hipaa,
  rule.nist_800_53, rule.tsc) are also stripped — these are useful
  for compliance reports in Graylog dashboards but are not used by
  the LLM for triage decisions.

  When stripping is on, the prompt block is labeled
  "ALERT ABRIDGED JSON:" so operators reading the prompt are not
  misled. When stripping is off, it reverts to "ALERT FULL JSON:".

  Always preserved regardless of this setting:
    - timestamp (LLM cross-references between alert + event time)
    - manager (cheap to retain; future-proofs multi-manager setups)
    - rule.groups (real classification signal — windows, suricata, etc.)
    - data.* (the actual event payload — the most critical block)
    - location (log source: EventChannel / journald / suricata)
    - full_log, predecoder (when present — carry detail not in data.*)

  Only the alert that goes to the LLM is abridged. The full alert
  still ships to Graylog via the GELF shipper unchanged, and is still
  recorded to the database.

  Token savings: roughly 60-170 tokens per alert depending on alert
  type. Windows-event alerts with full compliance metadata save the
  most (~170 tokens); terse Suricata or AppArmor alerts save less
  (~60-80 tokens). At this lab's ~2,400 LLM-triaged alerts/day this
  is roughly 144K-408K tokens/day saved — material for cloud LLM
  costs, modest speedup for local Ollama.

  Set to false if a custom workflow depends on the stripped fields,
  or if you suspect the stripping is degrading verdict quality and
  want to A/B test. Default true is the recommended setting.

============================================================
RULES.JSON - SITE-SPECIFIC RULE NOTES
============================================================

Each rule entry in rules.json supports TWO kinds of site-specific
notes: a global rule note that applies any time the rule fires,
and per-host notes that apply only when the rule fires on a
specific host.

Both note types are edited in the Rules tab of the web interface
on the same rule card. Rules notes are primary; the Hosts tab does
not create or edit rule-related notes. One source of truth for
writes.


GLOBAL RULE NOTE
-----------------

A free-text explanation shown to the LLM every time that rule
fires, regardless of which host triggered it.

Example:
  {
    "rule_id": "60608",
    "comment": "Windows Error Reporting — internal tracking note",
    "note": "Rule 60608 is Windows Error Reporting (WER). Focus on P1 (crashing process) and P4 (error code). Common benign sources: Dell management agents, Edge updater, .NET runtime failures on managed workstations. Recurring crashes on the same process are worth tracking as NOTEs.",
    "dedup_silence_seconds": 3600
  }

The note appears at the top of the LLM prompt as:

  RULE CONTEXT:
  Rule 60608 - Summary event of the report's signatures.
  Site-specific note: "Rule 60608 is Windows Error Reporting..."


PER-HOST NOTES (rule+host combo)
---------------------------------

Sometimes a rule has very different meaning depending on which
host it fires on. Rule 504 (Wazuh agent disconnected) is a
routine event on a laptop that sleeps but a NOTIFY-worthy
incident on a domain controller. A single global rule note
cannot express both realities.

Per-host notes solve this. Each rule can have a host_notes map
keyed by hostname. When the rule fires on a matching host, that
host's note is appended to the prompt alongside the global note.

Example:
  {
    "rule_id": "504",
    "note": "Wazuh agent disconnected. Interpret per host role.",
    "host_notes": {
      "Laptop1":            "Travel laptop. Frequent disconnects expected when asleep or off-VPN.",
      "Workstation1":       "Gaming workstation. Disconnect during shutdown/sleep is normal.",
      "Domain_Controller":  "Primary domain controller. Disconnect is a NOTIFY-worthy incident.",
      "Domain_Controller2": "Secondary domain controller. Disconnect warrants immediate investigation.",
      "Hypervisor1":        "Production hypervisor. Disconnect must be investigated."
    }
  }

When rule 504 fires on Domain_Controller, the prompt includes:

  RULE CONTEXT:
  Rule 504 - Wazuh agent disconnected
  Site-specific note: "Wazuh agent disconnected. Interpret per host role."
  Host-specific note for Domain_Controller: "Primary domain controller. Disconnect is a NOTIFY-worthy incident."

The host match is case-insensitive against the alert's canonical
hostname. Hosts not listed in host_notes get no host-specific line
(the global note still applies).


WHERE NOTES LIVE
----------------

All rule-related notes (global and per-host) are edited in the
Rules tab. The Hosts tab manages host inventory (OS, role, VLAN,
tags, anonymization alias, free-text notes) but NOT rule-specific
behavior. This keeps the edit flow consistent: when you are
thinking "rule 504 means X on host Y," you are thinking about the
rule's behavior, and that is where you edit.

The Hosts tab's "Notes" field is a host-level description of the
system — traffic patterns, ownership, normal behavior in general.
It applies to every alert involving that host regardless of rule.

The Rules tab's per-host notes are rule-specific overrides. They
only appear in the prompt when that specific rule fires on that
specific host.


PHILOSOPHY: CONTEXT OVER FILTERING
-------------------------------------
Both note types embody a core design principle of jrSOCtriage:
explain noise to the LLM rather than filter it out.

A traditional SIEM approach suppresses noisy rules entirely:
  → Rule 60608 always fires on this host → add suppress rule → done.

The problem: that suppress rule is unconditional. If rule 60608 fires
because something genuinely crashed in a suspicious way — an injected
DLL, a security tool being killed, a process crashing after exploitation
— the suppress rule creates a permanent blind spot. You will never see it.

jrSOCtriage's approach:
  → Rule 60608 fires → LLM reads your note → understands the normal
  → pattern → SUPPRESSes when evidence matches → NOTIFYs when it doesn't.

The LLM can distinguish between:
  "Dell.Remediation.Agent.exe crashed with a known .NET error" (SUPPRESS)
and:
  "lsass.exe crashed with an unusual error code during a logon spike" (NOTIFY)

A static filter cannot make that distinction. The note gives the LLM
the context it needs to make the right call without blinding it to
anomalies in the same rule space.

Per-host notes extend this: instead of a one-size-fits-all filter
saying "ignore rule 504 everywhere" or "escalate rule 504 everywhere,"
you describe the correct interpretation per host and let the LLM
reason from there.


WRITING EFFECTIVE NOTES
-------------------------
Good global rule notes answer three questions:
1. What is this rule normally detecting in your environment?
2. What does benign look like for this rule?
3. What would make this rule suspicious despite the normal pattern?

Good per-host notes answer ONE question:
1. On THIS specific host, what does this rule mean and how
   seriously should it be treated?

Global rule note examples:

Rule 86601 (Suricata ET INFO):
  "ET INFO rules are informational Suricata signatures covering expected
  network behavior. Focus on whether the specific domain/IP matches known
  benign patterns for this host. A spike in frequency, an unusual domain,
  or a high AbuseIPDB score on the destination should override the
  informational classification."

Rule 52000/52002 (AppArmor):
  "AppArmor audit and denial events on Ubuntu hosts running Docker are
  expected — container profile transitions fire every few seconds.
  Denials on snap packages (firmware-updater, etc.) are profile
  misconfigurations, not attacks. Flag if the denied process is
  unexpected or if the denied path suggests privilege escalation."

Rule 550 (FIM integrity):
  "FIM checksum changes fire heavily during apt upgrades and config
  changes. Treat as SUPPRESS during documented maintenance windows.
  Flag if the modified path is a sensitive binary (/bin, /sbin, /usr/bin),
  a config file for a privileged service, or if no maintenance is
  scheduled."

Per-host note examples:

Rule 19005 (CIS Windows 11 Benchmark) on MediaPC1:
  "Personal home media PC running Windows 11. Enterprise CIS benchmark
  failures are expected and not actionable. 26% score is normal
  for this host class."

Rule 19005 (CIS Windows 11 Benchmark) on Workstation1:
  "Primary workstation. Moderate benchmark failures acceptable,
  but scores below 40% warrant review of baseline hardening."


ADDING NOTES
--------------
Both note types are edited in the Rules tab of the web interface:

  - Site-Specific Note (textarea above Per-Host Notes) → global,
    applies on every host.
  - Per-Host Notes section → click "+ Add Host Note," pick a host
    from the dropdown, write the host-specific note. Add as many
    rows as needed.

Changes take effect after saving rules and restarting the pipeline.

The note field is separate from the "Comment" field — Comment is
for your internal tracking notes and is never sent to the LLM.
Both Note and Host Notes are what the LLM sees.

EDITING rules.json BY HAND
---------------------------

The web interface is the primary and recommended way to edit
rules.json. The interface enforces structure at save time: rule
conditions must be lists, host_notes must be an object, etc.

If you edit rules.json directly in a text editor, the pipeline
will tolerate common structural errors — individual malformed
rules are logged and skipped rather than crashing the whole
engine — but this means silent behavior changes are possible.

To validate a hand-edited rules.json without guessing: open the
Rules tab in the interface, then save the rules. The interface
will normalize the structure and surface any errors in the
status bar. If it saves cleanly, the file is valid.


============================================================
HOSTS.JSON - HOST TAGS
============================================================

Each host in hosts.json can have a "tags" array. Tags are short
labels that translate into plain-language phrases in the LLM
prompt, giving the model extra context about what kind of
system it is looking at without needing a full notes entry.

  "tags": ["auto_updates", "domain_joined"]

Tags are edited in the Hosts tab of the web interface as a
comma-separated list. No quotes, no brackets — just:

  auto_updates, domain_joined

KNOWN TAGS AND THEIR PROMPT TRANSLATIONS
-----------------------------------------

The pipeline recognizes the following tags and expands them
into descriptive text when building the host inventory block
of the LLM prompt:

  auto_updates    → "runs automatic unattended updates"
                    Use for hosts with unattended-upgrades,
                    Windows Update automatic install, dnf-automatic,
                    or similar. Tells the LLM that package and
                    config churn is expected baseline behavior.

  untrusted       → "UNTRUSTED device"
                    Use for guest devices, IoT gear, or any host
                    that should be treated with extra suspicion.

  critical        → "CRITICAL infrastructure"
                    Use for domain controllers, hypervisors,
                    firewalls, and other hosts where an incident
                    warrants immediate attention.

  transit         → "transit network"
                    Use for routers, switches, and point-to-point
                    links. Tells the LLM the host is a network
                    transit device, not an endpoint.

  domain_joined   → "domain joined"
                    Use for Active Directory domain members.
                    Helps the LLM distinguish between managed
                    and standalone hosts.

  dmz             → "DMZ host"
                    Use for hosts in a DMZ or internet-facing
                    segment. Elevates the significance of
                    inbound external activity.

CUSTOM TAGS
-----------

Any tag not in the known list is passed through to the prompt
as-is. This is useful for environment-specific labels like
"pci_scope" or "lab_only". The LLM will see the raw tag and
interpret it from context.

Tags are stored as an array in hosts.json:

  {
    "name": "Domain_Controller",
    "role": "domain_controller",
    "tags": ["critical", "domain_joined"],
    ...
  }


============================================================
ANONYMIZATION AND DEANONYMIZATION
============================================================

Anonymization is bidirectional. Understanding both directions is
important for interpreting what you see in Graylog.

THE FLOW
--------

1. Pipeline builds the prompt with real hostnames, usernames, IPs,
   and domains.

2. For cloud endpoints (Gemini, OpenAI, Anthropic) with anonymize
   enabled, the prompt is anonymized:
     - Laptop1        -> host-c
     - analyst        -> user-3
     - corp.local     -> corp1.internal
     - 192.168.20.10  -> 10.0.2.147   (subnet + host aliased)
     - 192.168.20.0/24 -> 10.0.2.0/23 (network context block)

   The IP example above shows the two address layers, which are
   controlled by two independent switches:

     - Anonymize subnets replaces the real network ID (the topology)
       with a fake one a single prefix bit wider (192.168.20.0/24 ->
       10.0.2.0/23). This is the primary privacy layer and is usually
       enough on its own — a host number is inert without its real
       subnet. It is cheap.
     - Anonymize IP addresses additionally randomizes the host
       portion within the fake subnet. It costs more CPU and is
       largely redundant once subnets are anonymized. With subnets on
       and IPs off, the host octet is kept real on the fake network
       (192.168.20.10 -> 10.0.2.10).

   IMPORTANT — subnet anonymization requires declared networks. It
   reads the subnet list from the Networks tab (the networks section
   of hosts.json) and can only alias subnets it knows about. Every
   internal subnet you use must be declared there. If an alert
   contains an internal IP that is in none of the declared subnets,
   the cloud call is REFUSED (fail closed) to avoid leaking
   un-aliased topology, and the log shows "internal IP ... match no
   declared subnet". The fix is to add the missing subnet under
   Networks. (Public/external IPs and special addresses such as
   0.0.0.0 are not affected and do not need to be declared.) The
   real-to-fake subnet mapping is stable across restarts and viewable
   read-only in the Subnet Aliases card on the Anonymization tab.

3. The anonymized prompt is sent to the LLM. This is what the LLM
   sees. This is what is logged in the gl2_llm_prompt field in
   Graylog.

4. The LLM responds using the anonymized values (host-c, user-3,
   10.0.2.147, corp1.internal).

5. DEANONYMIZATION: Before the response is written to Graylog
   fields (gl2_llm_summary, gl2_llm_reasoning, gl2_llm_missing,
   gl2_llm_verdict), it is reverse-mapped back to real values.

6. The deanonymized response is what you see in Graylog.

WHY THIS MATTERS
----------------

When reading an alert in Graylog, the response fields are always the
DEANONYMIZED (real) view:
  gl2_llm_reasoning:   LinuxLite, 192.168.20.10    (deanonymized)
  gl2_llm_summary:     LinuxLite, 192.168.20.10    (deanonymized)
  gl2_llm_missing:     LinuxLite, 192.168.20.10    (deanonymized)

The gl2_llm_prompt field depends on logging.prompt_log_mode:
  gl2_llm_prompt:      host-c, 10.0.2.147          (if mode=anonymized,
                                                    the default — shows
                                                    what was sent)
                       LinuxLite, 192.168.20.10    (if mode=deanonymized
                                                    — reverse-mapped for
                                                    readability)

What the cloud API actually received was always the ANONYMIZED
version (host-c, 10.0.2.147, the fake CIDR 10.0.2.0/23), regardless
of how it is logged.

The deanonymized fields CAN look like the LLM somehow "knew" the
real values. It did not. The LLM only saw anonymized values. The
reverse mapping happens inside jrSOCtriage after the API returns. To
see the actual anonymized payload independent of prompt_log_mode,
use debug_llm_payload (below).

VERIFYING ANONYMIZATION IS WORKING
-----------------------------------

CAUTION about the gl2_llm_prompt field in Graylog: what it shows
depends on the logging.prompt_log_mode setting.
  - "anonymized" (the default): gl2_llm_prompt is the ANONYMIZED
    prompt — i.e. exactly what was sent to the cloud API. In this
    mode you CAN use it to verify (look for fake values).
  - "deanonymized": gl2_llm_prompt is reverse-mapped back to REAL
    values for readability. In this mode it shows real hostnames,
    IPs, and CIDRs even when anonymization is working perfectly, so
    it CANNOT be used to verify anonymization.
  - "none": the prompt is not logged at all.
The other gl2_llm_* fields (summary, reasoning, missing, verdict)
are ALWAYS deanonymized (real) regardless of this setting, and email
is always deanonymized — those are the operator-facing view.

So the mode-independent way to confirm what is ACTUALLY sent to the
cloud API is the debug payload audit tool:

1. Set logging.debug_llm_payload to true in config.json and restart.
   (This logs the exact prompt sent to cloud endpoints to the system
   journal at WARNING level. It is cloud-only — it never logs local-
   endpoint prompts, which contain raw data.)
2. Let one alert escalate to the cloud endpoint, then look in the
   journal:
     journalctl -u jrsoctriage --no-pager | grep -A30 \
       "Exact prompt being sent"
3. In that dump, confirm:
   - Hostnames appear as host-X / your configured aliases.
   - The NETWORK CONTEXT block shows FAKE CIDRs (e.g. 10.0.2.0/23),
     with the real role names kept.
   - Internal IPs appear as fake addresses (e.g. 10.0.x.x), not your
     real 192.168.x.x / 10.x ranges.
   - Usernames are user-X / your aliases; domains match domain.json.
4. Set logging.debug_llm_payload back to false and restart. (Do not
   leave it on — it persists full prompts to the journal.)

You can also cross-check the Subnet Aliases card on the
Anonymization tab to see exactly which fake subnet each real subnet
maps to.

If the debug payload still contains real values, anonymization is
NOT working. Check:
  - Endpoint has anonymize: true in config.json
  - Master switches are enabled in the Anonymization tab
  - For subnet anonymization: the subnet is declared under Networks
    (an undeclared internal subnet causes a fail-closed refusal, not
    a leak — look for "match no declared subnet" in the log)
  - Endpoint type is cloud (gemini, openai, anthropic)
    - Local endpoints (ollama, llamacpp) never anonymize

WHY NOT JUST LOG THE ANONYMIZED RESPONSE?
------------------------------------------

The goal of jrSOCtriage is actionable SOC triage. If the LLM
reports "host-c needs investigation", you cannot act on that
without knowing which host is host-c. Deanonymizing the response
before writing to Graylog means analysts can search, filter,
and click through to real systems directly.

Anonymization protects the data in transit to the cloud API.
Deanonymization restores usability once the response is back
inside your network.

CONTROLLING WHAT GETS LOGGED TO GRAYLOG
----------------------------------------

The gl2_llm_prompt field (the prompt log) has three possible
modes controlled by logging.prompt_log_mode in config.json:

  "anonymized"   (default) - gl2_llm_prompt matches exactly what
                 was sent to the cloud API. Aliases like host-c,
                 user-3, 10.6.0.240 appear instead of real values.
                 Best for auditing that anonymization worked.

  "deanonymized" - gl2_llm_prompt shows the prompt with real
                 values instead of aliases. Easier to read during
                 triage because you do not need to cross-reference
                 aliases against the lookup tab. Real data is
                 already in Graylog in other fields regardless.

  "none"         - gl2_llm_prompt is not shipped to Graylog at all.
                 Smallest GELF messages. The LLM response fields
                 (gl2_llm_summary, gl2_llm_reasoning, gl2_llm_missing,
                 gl2_llm_verdict) still ship normally.

IMPORTANT: This setting does NOT affect what is sent to the LLM.
If per-endpoint anonymization is enabled, the LLM always receives
the anonymized prompt. prompt_log_mode only controls what appears
in Graylog for your convenience.

This setting is exposed on the ANONYMIZATION tab, in the "Graylog
Prompt Logging" card (it moved there from the Config tab because it
is an anonymization-visibility setting). Changes take effect after
restart.


============================================================
DEBUG LLM PAYLOAD MODE
============================================================

To verify exactly what is sent to cloud APIs and what is received
back (before deanonymization), enable debug payload logging.

HOW TO ENABLE
-------------

Option 1 (preferred): Use the web interface. On the ANONYMIZATION
tab, the "Graylog Prompt Logging" card has a "Debug LLM Payload"
toggle. Enable it, click that card's Save button, restart
jrsoctriage. (The same card also holds Prompt Log Mode.)

Option 2: Edit config.json directly and set debug_llm_payload
in the logging section:

  "logging": {
      "level": "info",
      "log_file": "jrsoc.log",
      "prompt_log_mode": "anonymized",
      "debug_llm_payload": true
  }

Restart jrsoctriage for the change to take effect.

WHAT IT LOGS
------------

When enabled, each LLM call produces two extra WARNING-level
log entries in the journal:

  [endpoint-name] [DEBUG_PAYLOAD] Exact prompt being sent to API
  ----- BEGIN PROMPT -----
  <full anonymized prompt exactly as sent to the API>
  ----- END PROMPT -----

  [endpoint-name] [DEBUG_PAYLOAD] Raw response from API BEFORE
  deanonymization:
  ----- BEGIN RESPONSE -----
  <exact bytes returned by the API, before reverse-mapping>
  ----- END RESPONSE -----

The prompt log shows the post-anonymization payload, letting you
confirm that hostnames, usernames, IPs, and domains were swapped
to aliases before leaving your network.

The response log shows what the LLM actually produced, before
jrSOCtriage maps aliases back to real values for storage in
Graylog. This is useful if you want to confirm that any real
values appearing in gl2_llm_* fields came from deanonymization
rather than from the LLM somehow seeing real data.

VIEWING DEBUG LOGS
------------------

  sudo journalctl -u jrsoctriage -f | grep DEBUG_PAYLOAD

Or for a specific alert:

  sudo journalctl -u jrsoctriage --since "10 minutes ago" | \
    grep -A 200 DEBUG_PAYLOAD

TURN OFF WHEN NOT DEBUGGING
----------------------------

Debug payload mode logs the full prompt AND response for every
LLM call. At typical volumes this produces a large amount of log
data. Leave it off in normal operation and only enable when
investigating anonymization or prompt content issues.

Set debug_llm_payload: false (or remove the key entirely) and
restart jrsoctriage to disable.
